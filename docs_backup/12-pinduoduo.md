## 一、Java 基础高频题

### Q1: HashMap 实现原理？

**核心概念**：

HashMap 是 Java 最常用的 Map 实现，基于哈希表。

**JDK 1.8 结构**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    HashMap 结构 (JDK 1.8)                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Node[] table                                                   │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐            │
│  │Node │Node │Node │Node │Node │Node │Node │ ... │            │
│  └──┬──┴─────┴──┬──┴─────┴─────┴─────┴──┬──┴─────┘            │
│     │           │                       │                      │
│     ▼           ▼                       ▼                      │
│   链表         红黑树                  链表                     │
│   (长度<8)    (长度≥8)               (长度<8)                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**哈希映射过程**：

```java
// 1. 计算 hash
int hash = (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);

// 2. 定位桶
int index = (n - 1) & hash;  // 等价于 hash % n，但位运算更快

// 为什么用 (n - 1) & hash？
// - n 是 2 的幂，n-1 的二进制全是 1
// - 如 n = 16，n-1 = 15 = 1111（二进制）
// - hash & 1111 = hash 的低 4 位
// - 位运算比取模快
```

**put 流程**：

```
1. 计算 hash
2. 定位桶
3. 桶为空 → 直接插入
4. 桶不为空：
   a. 相同 key → 覆盖 value
   b. 红黑树节点 → 插入树
   c. 链表节点 → 尾插法插入，链表长度 >= 8 转红黑树
5. 扩容检查：size > threshold
```

**面试加分点**：
> TreeMap 实现原理？
> - 基于红黑树
> - key 有序（根据 Comparable 或 Comparator）
> - 查找、插入、删除 O(log n)

### Q2: 为什么 HashMap 要使用红黑树？

**核心概念**：

红黑树解决链表过长导致的查询效率问题。

**对比分析**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    链表 vs 红黑树                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  链表：                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  查找：O(n)                                             │   │
│  │  插入：O(1)（已知位置）                                  │   │
│  │  删除：O(1)（已知位置）                                  │   │
│  │  问题：长度过长时查找效率低                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  红黑树：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  查找：O(log n)                                         │   │
│  │  插入：O(log n)                                         │   │
│  │  删除：O(log n)                                         │   │
│  │  优点：平衡性好，最坏情况也有保证                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  转换条件：                                                      │
│  - 链表长度 >= 8 且数组长度 >= 64 → 红黑树                     │
│  - 红黑树节点数 <= 6 → 链表                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么不用 AVL 树？**
- AVL 树更严格平衡，查找略快
- 红黑树插入删除更快，旋转次数少
- HashMap 插入删除操作多，红黑树更适合

### Q3: Java 反射是什么？

**核心概念**：

反射允许程序在运行时动态获取类的信息，创建对象、调用方法、访问属性。

**核心 API**：

```java
// 1. 获取 Class 对象
Class<?> clazz = Class.forName("com.example.User");
Class<?> clazz = User.class;
Class<?> clazz = user.getClass();

// 2. 创建对象
Object obj = clazz.newInstance();  // 已废弃
Object obj = clazz.getDeclaredConstructor().newInstance();

// 3. 获取方法
Method method = clazz.getMethod("setName", String.class);
method.invoke(obj, "张三");

// 4. 获取字段
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);  // 突破 private 限制
field.set(obj, "李四");

// 5. 获取注解
Annotation[] annotations = clazz.getAnnotations();
```

**应用场景**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      反射应用场景                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 框架开发                                                     │
│     └─ Spring IoC：动态创建 Bean                                │
│     └─ Spring AOP：动态代理                                     │
│     └─ MyBatis：结果集映射                                      │
│                                                                 │
│  2. 工具类                                                       │
│     └─ BeanUtils：属性复制                                      │
│     └─ JSON 序列化/反序列化                                     │
│                                                                 │
│  3. 测试框架                                                     │
│     └─ JUnit：运行测试方法                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**性能开销**：

| 操作 | 直接调用 | 反射调用 |
|------|---------|---------|
| 方法调用 | 快 | 慢（2-100 倍） |
| 字段访问 | 快 | 慢 |

**性能优化**：
- 缓存 Method/Field 对象
- 使用 setAccessible(true) 关闭安全检查
- 使用 MethodHandle（JDK 7+）

## 二、多线程与并发高频题

### Q4: 什么时候会发生死锁？

**核心概念**：

死锁是两个或多个线程互相等待对方释放资源，导致都无法继续执行。

**四个必要条件**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      死锁四个必要条件                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 互斥条件：资源只能被一个线程使用                            │
│                                                                 │
│  2. 请求保持条件：持有资源同时请求其他资源                      │
│                                                                 │
│  3. 不剥夺条件：资源不能被强制抢占                              │
│                                                                 │
│  4. 循环等待条件：线程间形成循环等待关系                        │
│                                                                 │
│  四个条件同时满足才会死锁                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**示例代码**：

```java
public class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();
    
    public static void main(String[] args) {
        new Thread(() -> {
            synchronized (lockA) {
                System.out.println("Thread 1: 持有 lockA");
                sleep(100);
                synchronized (lockB) {  // 等待 lockB
                    System.out.println("Thread 1: 持有 lockA 和 lockB");
                }
            }
        }).start();
        
        new Thread(() -> {
            synchronized (lockB) {
                System.out.println("Thread 2: 持有 lockB");
                sleep(100);
                synchronized (lockA) {  // 等待 lockA → 死锁
                    System.out.println("Thread 2: 持有 lockA 和 lockB");
                }
            }
        }).start();
    }
}
```

**如何预防死锁？**
1. 破坏互斥：不现实
2. 破坏请求保持：一次性申请所有资源
3. 破坏不剥夺：申请不到时释放已持有资源
4. 破坏循环等待：按顺序申请资源

### Q5: CAS 和 synchronized 的区别？

**核心概念**：

CAS 是乐观锁，synchronized 是悲观锁。

**对比**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    CAS vs synchronized                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  CAS（乐观锁）                                                   │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  原理：比较并交换                                        │   │
│  │  实现：Unsafe.compareAndSwapInt()                       │   │
│  │  优点：无锁，无阻塞                                      │   │
│  │  缺点：ABA 问题、自旋开销                                │   │
│  │  适用：竞争不激烈                                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  synchronized（悲观锁）                                          │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  原理：获取 monitor                                      │   │
│  │  实现：JVM 字节码指令                                    │   │
│  │  优点：简单、自动释放                                    │   │
│  │  缺点：阻塞开销                                          │   │
│  │  适用：竞争激烈                                          │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**CAS 代码示例**：

```java
// AtomicInteger
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // CAS 实现

// 源码
public final int incrementAndGet() {
    return unsafe.getAndAddInt(this, valueOffset, 1) + 1;
}

public final int getAndAddInt(Object o, long offset, int delta) {
    int v;
    do {
        v = getIntVolatile(o, offset);
    } while (!compareAndSwapInt(o, offset, v, v + delta));  // CAS
    return v;
}
```

### Q6: ConcurrentHashMap 实现？

**核心概念**：

ConcurrentHashMap 是线程安全的 HashMap。

**JDK 1.7：分段锁**：
- 16 个 Segment
- 每个 Segment 是 ReentrantLock + HashEntry[]

**JDK 1.8：CAS + synchronized**：

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    int hash = spread(key.hashCode());
    
    for (Node<K,V>[] tab = table;;) {
        // 1. 桶为空 → CAS 插入
        if ((f = tabAt(tab, i)) == null) {
            if (casTabAt(tab, i, null, new Node<>()))
                break;
        }
        // 2. 正在扩容 → 协助扩容
        else if ((fh = f.hash) == MOVED) {
            tab = helpTransfer(tab, f);
        }
        // 3. 桶不为空 → synchronized 锁头节点
        else {
            synchronized (f) {
                // 插入链表或红黑树
            }
        }
    }
}
```

**为什么 JDK 1.8 放弃分段锁？**
1. 每个 Segment 占用内存
2. 并发度固定（默认 16）
3. synchronized 优化后性能提升
4. 锁粒度更细（桶级别）

## 四、数据库高频题

### Q8: 数据库事务特点？

**核心概念**：

事务有 ACID 四大特性。

**ACID**：

```
┌─────────────────────────────────────────────────────────────────┐
│                       ACID 特性                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  A - 原子性（Atomicity）                                         │
│     └─ 事务是不可分割的最小单位                                 │
│     └─ 要么全部成功，要么全部失败                               │
│     └─ 实现：Undo Log                                          │
│                                                                 │
│  C - 一致性（Consistency）                                       │
│     └─ 事务执行前后，数据保持一致                               │
│     └─ 约束不被破坏                                             │
│     └─ 是最终目标                                               │
│                                                                 │
│  I - 隔离性（Isolation）                                         │
│     └─ 多个事务并发执行互不干扰                                 │
│     └─ 四种隔离级别                                             │
│     └─ 实现：锁 + MVCC                                         │
│                                                                 │
│  D - 持久性（Durability）                                        │
│     └─ 事务提交后，数据永久保存                                 │
│     └─ 实现：Redo Log                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### Q9: 事务隔离级别？

**核心概念**：

隔离级别解决并发事务的问题。

**隔离级别对比**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---------|------|-----------|------|
| READ UNCOMMITTED | ✅ | ✅ | ✅ |
| READ COMMITTED | ❌ | ✅ | ✅ |

| SERIALIZABLE | ❌ | ❌ | ❌ |

### 二面 & 三面：并发编程大通关

#### Q1: 并发三要素是什么？

**参考答案**：

| 要素 | 含义 | 实现方式 |
|------|------|---------|
| 原子性 | 操作不可分割 | synchronized、Lock、Atomic |
| 可见性 | 修改对其他线程可见 | volatile、synchronized |

#### Q8: 死锁如何定位和避免？

**参考答案**：

```bash
# 1. jstack 查看线程栈

# 3. jvisualvm 检测死锁
```

**避免方法**：
1. **破坏循环等待**：按顺序获取锁
2. **破坏请求保持**：一次性获取所有锁
3. **使用 tryLock**：设置超时时间
4. **使用 Lock 接口**：可中断锁
