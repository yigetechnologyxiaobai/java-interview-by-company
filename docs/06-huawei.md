## 一、Java 并发高频题

### Q1: synchronized 底层原理？

**参考答案**：

**实现方式**：
- 同步代码块：monitorenter/monitorexit 字节码指令
- 同步方法：ACC_SYNCHRONIZED 标志

**锁升级过程**：
```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁
```

**对象头 Mark Word**：
- 锁状态（2 bit）
- 偏向线程 ID（23 bit）
- 偏向时间戳（2 bit）
- 对象分代年龄（4 bit）

**锁升级详解**：
1. **偏向锁**：同一线程重入，CAS 修改 Mark Word
2. **轻量级锁**：多线程竞争，CAS 自旋
3. **重量级锁**：竞争激烈，阻塞等待

**追问**：
- 锁升级为什么不可逆？
- 偏向锁延迟启动原因？
- synchronized 和 ReentrantLock 区别？

### Q2: volatile 关键字？

**参考答案**：

**两大特性**：
1. **可见性**：修改后立即同步到主内存
2. **禁止指令重排**：内存屏障

**内存屏障**：
- LoadLoad：禁止读读重排
- StoreStore：禁止写写重排
- LoadStore：禁止读写重排
- StoreLoad：禁止写读重排

**不保证原子性**：
```java
volatile int count = 0;
count++;  // 不是原子操作：读取、修改、写入
```

**应用场景**：
- 状态标志位
- 双重检查锁单例
- volatile Bean

**追问**：
- volatile 和 synchronized 区别？
- 单例模式双重检查锁为什么要用 volatile？

### Q3: 线程池核心参数？

**参考答案**：

```java
ThreadPoolExecutor(
    int corePoolSize,       // 核心线程数
    int maximumPoolSize,    // 最大线程数
    long keepAliveTime,     // 非核心线程空闲存活时间
    TimeUnit unit,
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

**执行流程**：
```
1. 线程数 < corePoolSize → 创建核心线程
2. 线程数 >= corePoolSize → 放入队列
3. 队列满 → 创建非核心线程
4. 线程数 = maximumPoolSize → 执行拒绝策略
```

**拒绝策略**：
| 策略 | 行为 |
|------|------|
| AbortPolicy | 抛异常（默认） |
| CallerRunsPolicy | 调用者执行 |
| DiscardPolicy | 丢弃 |
| DiscardOldestPolicy | 丢弃最老任务 |

**参数设置建议**：
- CPU 密集型：corePoolSize = CPU 核数 + 1
- IO 密集型：corePoolSize = CPU 核数 * 2

**追问**：
- 为什么不用 Executors？
- 线程池如何监控？

### Q4: ThreadLocal 原理？

**参考答案**：

**数据结构**：
```
Thread
  └── ThreadLocalMap
        └── Entry extends WeakReference<ThreadLocal<?>>
              ├── key: ThreadLocal（弱引用）
              └── value: 值（强引用）
```

**内存泄漏原因**：
1. key 是弱引用，GC 后变 null
2. value 是强引用，无法回收
3. 线程不结束，value 无法释放

**解决方案**：
- 使用完毕后手动 remove()
- 使用 try-finally 确保清理

**追问**：
- key 为什么用弱引用？
- ThreadLocal 如何实现跨线程传递？

### Q5: ConcurrentHashMap 实现？

**参考答案**：

**JDK 1.7**：
- Segment 分段锁（默认 16 个）
- 每个 Segment 继承 ReentrantLock
- 并发度 = Segment 数量

**JDK 1.8**：
- CAS + synchronized
- 锁粒度：桶的头节点
- 链表转红黑树（长度 >= 8）

**put 流程**：
1. 计算 hash，定位桶
2. 桶为空 → CAS 插入
3. 桶不为空 → synchronized 锁头节点
4. 插入链表或红黑树

**为什么 JDK 1.8 放弃分段锁**：
1. 每个 Segment 占用内存
2. 并发度固定
3. synchronized 优化后性能提升

**追问**：
- ConcurrentHashMap 和 Hashtable 区别？
- size() 如何实现？

## 二、JVM 高频题

### Q6: JVM 内存结构？

**参考答案**：

**运行时数据区**：
```
┌─────────────────────────────────────┐
│            方法区（元空间）            │
├─────────────────────────────────────┤
│              堆                      │
├──────────┬──────────┬───────────────┤
│ 虚拟机栈  │ 本地方法栈│  程序计数器    │
├──────────┴──────────┴───────────────┤
│            本地内存                   │
└─────────────────────────────────────┘
```

**堆内存划分**：
- 新生代（1/3）：Eden + Survivor0 + Survivor1
- 老年代（2/3）

**方法区变化**：
- JDK 7：永久代（PermGen）
- JDK 8：元空间（Metaspace，本地内存）

**追问**：
- 堆和栈的区别？
- 对象一定在堆上分配吗？

### Q7: 垃圾收集算法？

**参考答案**：

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 标记-清除 | 标记后清除 | 简单 | 碎片 |
| 标记-整理 | 标记后整理 | 无碎片 | 移动成本 |
| 复制算法 | 复制存活对象 | 高效 | 空间浪费 |
| 分代收集 | 按分代选算法 | 综合 | 复杂 |

**分代收集**：
- 新生代：Eden 区存活率低，用复制算法
- 老年代：存活率高，用标记-整理

**追问**：
- 为什么新生代用复制算法？
- 老年代为什么不用复制算法？

### Q8: CMS 收集器？

**参考答案**：

**四个阶段**：
1. **初始标记**（STW）：标记 GC Roots 直接关联的对象
2. **并发标记**：遍历对象图
3. **重新标记**（STW）：修正并发标记期间的变动
4. **并发清除**：清除垃圾对象

**优点**：
- 低延迟
- 并发收集

**缺点**：
- CPU 敏感（占用线程）
- 浮动垃圾
- 空间碎片

**追问**：
- CMS 和 G1 的区别？
- 为什么 CMS 要用标记-清除？

### Q9: G1 收集器？

**参考答案**：

**核心概念**：
- **Region**：堆划分为多个大小相等的 Region
- **Remembered Set**：跨 Region 引用
- **SATB**：原始快照算法

**回收步骤**：
1. 初始标记（STW）
2. 并发标记
3. 最终标记（STW）
4. 筛选回收（STW）

**特点**：
- 可预测停顿
- 无碎片
- 大内存友好

**追问**：
- G1 如何实现可预测停顿？
- G1 和 CMS 如何选择？

### Q10: 类加载机制？

**参考答案**：

**加载过程**：
1. **加载**：读取 Class 文件，创建 Class 对象
2. **验证**：确保 Class 文件合法
3. **准备**：为静态变量分配内存，设初始值
4. **解析**：符号引用转直接引用
5. **初始化**：执行 `<clinit>` 方法

**双亲委派模型**：
```
启动类加载器
    ↑
扩展类加载器
    ↑
应用类加载器
```

**为什么要双亲委派**：
- 安全性：防止核心类被篡改
- 唯一性：避免类重复加载

**打破双亲委派**：
- 自定义类加载器
- SPI 机制（线程上下文类加载器）
- Tomcat 类加载器

**追问**：
- Tomcat 如何打破双亲委派？
- SPI 机制原理？

## 三、MySQL 高频题

### Q11: MySQL 为什么用 B+ 树？

**参考答案**：

**B+ 树特点**：
1. **树高度低**：3-4 层可存千万级数据
2. **范围查询友好**：叶子节点链表
3. **磁盘利用率高**：非叶子节点只存键值

**与 B 树区别**：
| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点 | 只有叶子节点 |
| 范围查询 | 需要中序遍历 | 叶子节点链表 |
| 树高度 | 较高 | 较低 |

**追问**：
- 聚簇索引和非聚簇索引区别？
- 为什么用自增主键？

### Q12: 索引优化？

**参考答案**：

**索引设计原则**：
1. 选择区分度高的列
2. 遵循最左前缀原则
3. 避免索引列做运算
4. 控制索引数量

**索引失效场景**：
- LIKE 以 % 开头
- 对索引列做运算
- 类型隐式转换
- 使用函数
- OR 条件（部分列无索引）

**EXPLAIN 分析**：
- type：ALL（全表扫描）、index、range、ref、const
- key：实际使用的索引
- rows：预估扫描行数

**追问**：
- 覆盖索引的优势？
- 索引下推是什么？

### Q13: MVCC 原理？

**参考答案**：

**核心组件**：
1. **隐藏字段**：
   - DB_TRX_ID：事务 ID
   - DB_ROLL_PTR：回滚指针

2. **Undo Log**：版本链

3. **Read View**：可见性判断
   - m_ids：活跃事务 ID 列表
   - min_trx_id：最小活跃事务 ID
   - max_trx_id：下一个事务 ID
   - creator_trx_id：创建者事务 ID

**可见性判断**：
- trx_id < min_trx_id → 可见
- trx_id >= max_trx_id → 不可见
- trx_id 在 m_ids 中 → 不可见

**隔离级别**：
- RC：每次 select 生成新的 Read View
- RR：事务第一次 select 生成 Read View

**追问**：
- MVCC 能解决幻读吗？
- 当前读和快照读区别？

### Q14: 主从复制？

**参考答案**：

**复制流程**：
1. Master 写入 Binlog
2. Slave IO 线程读取 Binlog，写入 Relay Log
3. Slave SQL 线程重放 Relay Log

**复制模式**：
- 异步复制：性能好，可能丢数据
- 半同步复制：至少一个从库确认
- 全同步复制：所有从库确认

**主从延迟解决**：
1. 并行复制
2. 读写分离优化（关键读走主库）
3. 分库分表

**追问**：
- 如何监控主从延迟？
- GTID 复制是什么？

### Q16: Redis 持久化？

**参考答案**：

| 方式 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| RDB | 快照 | 文件小、恢复快 | 可能丢数据 |

| 混合 | RDB + AOF 增量 | 综合 | 复杂 |

**AOF 重写**：
- 触发条件：文件大小超过阈值
- 重写过程：fork 子进程，生成新文件

### Q17: 缓存穿透、击穿、雪崩？

**参考答案**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 穿透 | 查询不存在的数据 | 布隆过滤器、缓存空值 |
| 击穿 | 热点 key 过期 | 永不过期、互斥锁 |

**布隆过滤器**：
- 位图 + 多个哈希函数
- 判断元素可能存在或一定不存在
- 不支持删除

#### Q2: 什么是内存泄漏？如何检测与避免？

**常见原因**：
```c
// 1. 忘记释放
int* p = malloc(sizeof(int) * 100);
// 忘记 free(p);

// 2. 异常导致未执行 free
int* p = malloc(sizeof(int) * 100);
if (error) return;  // 提前返回，未释放
```

**检测工具**：Valgrind、AddressSanitizer、Dr. Memory

**避免方法**：
```cpp
// 使用智能指针（C++）
std::shared_ptr<int> p = std::make_shared<int>(10);

// RAII 模式
class Resource {
    int* data;
public:
    Resource() : data(new int[100]) {}
    ~Resource() { delete[] data; }  // 自动释放
};
```

---

#### Q3: new 和 delete 的底层实现？

**new 的过程**：
1. 调用 `operator new` 分配内存（底层是 malloc）
2. 调用构造函数初始化对象
3. 返回对象指针

**delete 的过程**：
1. 调用析构函数
2. 调用 `operator delete` 释放内存（底层是 free）

| 特性 | new | malloc |
|------|-----|--------|
| 返回类型 | 类型安全 | void* |
| 失败处理 | 抛异常 | 返回 NULL |
| 构造函数 | 自动调用 | 不调用 |

---

#### Q4: 堆与栈的区别？

| 特性 | 栈（Stack） | 堆（Heap） |
|------|------------|-----------|
| 分配方式 | 编译器自动 | 程序员手动 |
| 释放方式 | 自动释放 | 手动释放 |
| 大小限制 | 较小（MB 级） | 较大（GB 级） |
| 分配速度 | 快 | 慢 |
| 碎片问题 | 无 | 有 |

---

#### Q5: 什么是内存溢出？常见场景？

**堆溢出**：
```java
int[] arr = new int[Integer.MAX_VALUE];  // OOM
```

**栈溢出**：
```java
void recursion() {
    recursion();  // StackOverflowError
}
```

**排查方法**：
- `jmap -histo <pid>` 查看对象统计
- `jvisualvm` 分析堆内存

#### Q1: C 与 C++ 的主要区别是什么？

**参考答案**：

| 特性 | C | C++ |
|------|---|-----|
| 编程范式 | 面向过程 | 多范式（面向对象+泛型） |
| 类与对象 | 无 | 有 |
| 继承/多态 | 无 | 有 |
| 标准库 | 较少 | STL（容器、算法、迭代器） |
| 引用 | 无 | 有 |
| 异常处理 | 无 | try-catch 机制 |
| 函数重载 | 不支持 | 支持 |
| 默认参数 | 不支持 | 支持 |

**代码示例**：
```cpp
// C 风格
struct Point { int x, y; };
void move(struct Point* p, int dx, int dy) {
    p->x += dx;
    p->y += dy;
}

// C++ 风格
class Point {
public:
    int x, y;
    void move(int dx, int dy) {
        x += dx;
        y += dy;
    }
};
```

#### Q2: 什么是内存泄漏？如何检测和避免？

**参考答案**：

**定义**：程序动态分配的内存未被释放，导致内存无法被重新使用。

**常见原因**：
```c
// 1. 忘记释放
int* p = malloc(sizeof(int) * 100);
// 忘记 free(p);

// 2. 异常导致未执行 free
int* p = malloc(sizeof(int) * 100);
if (error) return;  // 提前返回，未释放

// 3. 指针被覆盖
int* p = malloc(sizeof(int) * 100);
p = malloc(sizeof(int) * 200);  // 原内存泄漏
```

**检测工具**：
- **Valgrind**：Linux 下最常用的内存检测工具
- **AddressSanitizer**：编译器内置，性能开销小
- **Dr. Memory**：Windows 平台

**避免方法**：
```cpp
// 使用智能指针（C++）
std::unique_ptr<int> p = std::make_unique<int>(10);
std::shared_ptr<int> sp = std::make_shared<int>(20);

// RAII 模式
class Resource {
    int* data;
public:
    Resource() : data(new int[100]) {}
    ~Resource() { delete[] data; }  // 自动释放
};
```

#### Q3: new 和 delete 的底层实现？

**参考答案**：

**new 的过程**：
1. 调用 `operator new` 分配内存（底层通常是 malloc）
2. 调用构造函数初始化对象
3. 返回对象指针

**delete 的过程**：
1. 调用析构函数
2. 调用 `operator delete` 释放内存（底层通常是 free）

**代码示例**：
```cpp
// new 表达式
MyClass* obj = new MyClass();

// 等价于
void* memory = operator new(sizeof(MyClass));  // 分配内存
MyClass* obj = new(memory) MyClass();          // 定位 new，调用构造函数

// delete 表达式
delete obj;

// 等价于
obj->~MyClass();                    // 调用析构函数
operator delete(obj);               // 释放内存
```

**对比**：
| 特性 | new/delete | malloc/free |
|------|-----------|-------------|
| 返回类型 | 类型安全 | void* |
| 失败处理 | 抛出 bad_alloc | 返回 NULL |
| 构造/析构 | 自动调用 | 不调用 |
| 可重载 | 是 | 否 |

**追问**：
- placement new 是什么？
- 如何重载 operator new？

#### Q4: 堆与栈的区别？

**参考答案**：

| 特性 | 栈（Stack） | 堆（Heap） |
|------|------------|-----------|
| 分配方式 | 编译器自动管理 | 程序员手动申请/释放 |
| 释放方式 | 函数结束自动释放 | delete/free 手动释放 |
| 大小限制 | 较小（MB 级，约 1-8MB） | 较大（GB 级） |
| 分配速度 | 快（移动指针） | 慢（查找空闲块） |
| 碎片问题 | 无 | 有 |
| 生命周期 | 函数作用域 | 手动控制 |
| 存储内容 | 局部变量、函数参数 | 动态分配的对象 |

**代码示例**：
```cpp
void func() {
    int a = 10;          // 栈上
    static int b = 20;   // 静态区
    int* p = new int(30); // 堆上
    delete p;
}
```

- 为什么栈比堆快？

#### Q5: 什么是内存溢出？常见场景？

**参考答案**：

**定义**：程序请求的内存超过了系统能提供的最大内存。

**常见场景**：

1. **堆溢出（OOM）**：
```java
// 创建超大数组
int[] arr = new int[Integer.MAX_VALUE];  // OutOfMemoryError

// 对象创建过快
List<Object> list = new ArrayList<>();
while (true) {
    list.add(new Object());  // 无限添加
}
```

2. **栈溢出（StackOverflowError）**：
```java
void recursion() {
    recursion();  // 无限递归
}
```

3. **内存泄漏导致溢出**：
```java
// 静态集合持有对象引用
static List<Object> cache = new ArrayList<>();
// 持续添加但不清理
```

**排查方法**：
```bash
# JVM 堆内存分析
jmap -histo <pid> | head -20

# 导出堆转储
jmap -dump:format=b,file=heap.hprof <pid>

# 使用 jvisualvm 或 MAT 分析
```

**追问**：
- 如何定位内存泄漏？
- JVM 内存模型？
