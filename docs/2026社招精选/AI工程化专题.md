## 一、RAG 系统架构设计

### 1. 什么是 RAG？核心组件有哪些？

**RAG（Retrieval-Augmented Generation，检索增强生成）** 是一种结合信息检索与大语言模型生成的架构，用于解决 LLM 知识滞后和幻觉问题。

**核心组件**：
- **向量数据库**：Milvus、Pinecone、Chroma，存储文本向量化后的 Embedding
- **Embedding 模型**：文本向量化（text-embedding-ada-002、bge-large-zh 等）
- **检索优化**：混合检索（向量检索 + 关键词 BM25）、MMR（Maximal Marginal Relevance）去重
- **Prompt 工程**：上下文管理、Few-shot 示例设计、系统 Prompt 约束

**实战问题**：
- 如何评估 RAG 系统的效果？→ 使用精确率/召回率 + LLM-as-Judge 评估生成质量
- 文档切片策略如何设计？→ 按语义边界分割（500-1000 token），保留重叠窗口
- 如何处理长上下文超出限制？→ 分块检索 + Map-Reduce 汇总

### 2. RAG 系统的典型技术栈

```
文档预处理 → 文本切片 → Embedding向量化 → 向量存储
                                          ↓
用户查询 → Query Embedding → 向量检索 → 结果排序 → Prompt组装 → LLM生成 → 返回答案
```

**关键技术点**：
- 混合检索：BM25 关键词 + 向量语义双路召回
- 重排序（Rerank）：Cross-Encoder 模型对检索结果精细排序
- 缓存层：缓存高频查询的 Embedding 和检索结果

## 二、Agent 系统设计与实现

### 1. Agent 核心架构

**四大核心组件**：

| 组件 | 职责 | 典型实现 |
|------|------|----------|
| Planning（规划） | 任务分解、反思迭代 | ReAct、Tree of Thoughts |
| Memory（记忆） | 短期记忆（对话历史）、长期记忆（向量库） | 对话上下文 + 向量数据库 |
| Tools（工具） | 函数调用、API 集成、代码执行 | MCP、Function Calling |
| Action（执行） | 执行引擎、结果解析 | LangChain Agent Executor |

### 2. 主流 Agent 框架对比

| 框架 | 特点 | 适用场景 |
|------|------|----------|
| LangChain | 生态丰富、组件化 | 通用 Agent 开发 |
| AutoGen | 多 Agent 协作 | 多角色对话、复杂任务编排 |
| Dify | 低代码、易部署 | 快速原型、非开发者 |
| AgentScope | 阿里出品，复杂编排 | 企业级 Agent 应用 |

### 3. Agent 面试高频追问

- **如何解决 Agent 的幻觉问题？** → 工具调用结果验证 + RAG 知识约束 + 输出格式校验
- **多轮对话中如何维护上下文？** → 对话历史管理（滑动窗口 / 摘要压缩）+ 长期记忆向量库
- **如何评估 Agent 任务完成的质量？** → 定义任务指标 + 人工评估 + 自动化测试集

## 三、AI 应用中的工程挑战

### 1. 延迟优化

**首字延迟（TTFT）优化策略**：
- 流式输出（Streaming）：SSE / WebSocket，提升用户感知
- 模型蒸馏 / 量化：INT8 / INT4 量化，减少推理计算量
- 缓存层：缓存常见 Prompt 的中间结果（KV Cache 复用）
- 异步预处理：并行执行检索和模型预热

### 2. 稳定性保障

**LLM 服务不稳定的降级策略**：
- 多模型冗余：主模型 + 备用模型自动切换
- 重试机制：指数退避 + 最大重试次数
- 兜底回复：预设规则引擎 fallback
- 超时熔断：Sentinel 集成，防止雪崩

### 3. 数据安全

- **敏感信息脱敏**：Prompt 输入前 PII（个人信息）检测和替换
- **私有化部署**：本地部署开源模型（Qwen、Llama），避免数据外泄
- **Prompt 注入防护**：输入校验、系统 Prompt 约束、输出过滤

### 4. Prompt 版本管理与 AB 测试

```
Prompt 版本控制 → 灰度发布 → 效果指标收集 → 自动对比分析
```

- 建立 Prompt 版本库，每次变更留痕
- A/B 测试：相同查询对比不同 Prompt 的回复质量
- 自动化评估：构建测试集，定期回归测试

## 四、AI 工程化实战题目

### Q1: 你们项目中 LLM 调用失败率多少？如何处理的？

**参考答案**：
- 线上 LLM 调用失败率控制在 **2% 以内**
- 处理策略：
  1. **指数退避重试**：首次失败等待 1s，重试等待 2s、4s，最大 3 次
  2. **多模型切换**：主模型（GPT-4/Claude）失败自动切换备用模型（国产大模型）
  3. **降级兜底**：全部失败时返回预设规则答案或缓存结果
  4. **监控告警**：Prometheus + Grafana 监控调用成功率、延迟分布

### Q2: 如何设计 Prompt 版本管理和 AB 测试？

**参考答案**：
- **版本管理**：使用 Git 管理 Prompt 模板，每次变更有版本号
- **AB 测试**：流量按用户 ID hash 分流，50% 走 A 版本、50% 走 B 版本
- **评估指标**：回复准确率、用户满意度评分、Token 消耗成本
- **自动化回归**：构建 500+ 标准测试集，每次 Prompt 变更自动跑回归

### Q3: 如何防止 Prompt 注入攻击？

**参考答案**：
- **输入过滤**：检测用户输入中的指令性内容（如 "忽略之前指令"、"你是 XXX"）
- **系统 Prompt 加固**：在系统消息中明确禁止覆盖行为
- **输出校验**：对模型输出进行安全检查和格式化
- **隔离层**：用户输入和系统指令用不同角色消息传递
