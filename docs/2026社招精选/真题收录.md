## 一、Java 21 新特性（2026 年社招热点）

### 1. 虚拟线程（Virtual Threads）

**问题**：Java 21 的虚拟线程相比传统线程有什么优势？

**参考答案**：
- **传统线程**：每个线程映射一个 OS 线程，内存开销大（约 1MB/线程），切换成本高
- **虚拟线程**：JVM 层面轻量级线程，可创建数百万个，内存开销仅几百字节
- **使用方式**：`Thread.startVirtualThread(() -> { ... })` 或 `Executors.newVirtualThreadPerTaskExecutor()`
- **适用场景**：大量 I/O 阻塞操作（数据库查询、HTTP 请求），不适合 CPU 密集型计算
- **注意**：虚拟线程不适合 synchronized 代码块，会导致 pinning（建议用 ReentrantLock）

### 2. 模式匹配与 Record 类型

**switch 模式匹配**：
```java
String format(Object o) {
    return switch (o) {
        case Integer i -> String.format("int %d", i);
        case Long l    -> String.format("long %d", l);
        case String s  -> String.format("string %s", s);
        case null      -> "null";
        default        -> o.toString();
    };
}
```

**Record 类型**：
- 不可变数据载体，自动生成 equals()、hashCode()、toString()
- 适合 DTO、API 响应等场景
- 示例：`record User(Long id, String name) {}`

### 3. Sequenced Collections（有序集合）

Java 21 新增 `SequencedCollection` 和 `SequencedSet` 接口：
- `addFirst()` / `addLast()`
- `getFirst()` / `getLast()`
- `removeFirst()` / `removeLast()`
- `reversed()` 返回反向视图

## 二、JVM 深度调优（生产实战向）

### 1. G1 GC 调优实战

**GC 日志解读**：
```
[GC pause (G1 Evacuation Pause) (young), 0.0151234 secs]
 [Parallel Time: 14.5 ms, GC Workers: 8]
 [Object Copy: 12.2 ms]  ← 对象复制耗时最长！
 [Eden: 2048.0M->0.0B(2048.0M)]
 [Heap: 2048.0M(4096.0M)->1024.0M(4096.0M)]
```

**优化策略**：
- 对象复制耗时过高 → 检查是否有大对象直接分配到 Old 区
- 调整 `-XX:G1HeapRegionSize`：根据堆大小选择合适的 Region 大小
- `-XX:MaxGCPauseMillis`：设置目标停顿时间（建议 100-200ms）
- `-XX:InitiatingHeapOccupancyPercent`：触发并发标记的堆占用阈值（默认 45%）

### 2. ZGC 低延迟调优

ZGC 特点：停顿时间 < 1ms，适合对延迟敏感的场景

```bash
java -XX:+UseZGC \
     -Xms16g -Xmx16g \
     -XX:ConcGCThreads=4 \
     -XX:ZAllocationSpikeTolerance=5 \
     -jar application.jar
```

### 3. OOM 排查流程

1. `jmap -dump:live,format=b,file=heap.hprof <pid>` 导出堆转储
2. MAT（Memory Analyzer Tool）分析 Dominator Tree
3. 定位内存泄漏对象 → 查看 GC Roots 引用链
4. 检查代码：缓存未设置上限、ThreadLocal 未 remove()、大对象未释放

## 三、并发编程进阶

### 1. ThreadLocal 内存泄漏深度剖析

**泄漏原因**：
- ThreadLocalMap 的 Key 是弱引用（WeakReference），Value 是强引用
- Key 被 GC 回收后，Value 仍被强引用，无法回收
- 线程池场景下线程复用，泄漏会累积

**解决方案**：
```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove(); // 必须 finally 中清理！
}
```

### 2. CompletableFuture 最佳实践

```java
// 并行执行多个独立任务
CompletableFuture<String> f1 = CompletableFuture.supplyAsync(() -> queryUser());
CompletableFuture<List<Order>> f2 = CompletableFuture.supplyAsync(() -> queryOrders());

// 组合结果
CompletableFuture<UserOrderVO> result = f1.thenCombine(f2, (user, orders) -> {
    return new UserOrderVO(user, orders);
}).exceptionally(ex -> {
    log.error("查询失败", ex);
    return UserOrderVO.defaultVO();
});
```

### 3. 分布式锁进阶

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| Redis SETNX + Lua | 简单、高性能 | 时钟漂移可能导致锁失效 | 一般业务场景 |
| Redisson | 看门狗自动续期、可重入 | 依赖 Redis 集群 | 企业级项目 |

## 四、MySQL 进阶

### 1. 十亿级数据分页优化

**问题**：`LIMIT 1000000, 20` 为什么慢？如何优化？

**原因**：MySQL 需要先扫描 1000020 行，再丢弃前 1000000 行

**优化方案**：
```sql
-- 方案1：延迟关联（覆盖索引 + JOIN）
SELECT t.* FROM table t
INNER JOIN (SELECT id FROM table ORDER BY id LIMIT 1000000, 20) AS tmp
ON t.id = tmp.id;

-- 方案2：游标分页（推荐）
SELECT * FROM table WHERE id > #{lastId} ORDER BY id LIMIT 20;

-- 方案3：ES search_after（深度分页）
```

### 2. MVCC（多版本并发控制）原理

**核心机制**：
- **ReadView**：事务启动时创建一致性视图
- **Undo Log**：维护数据的版本链
- **隐藏列**：DB_TRX_ID（事务ID）、DB_ROLL_PTR（回滚指针）

**可见性判断**：
- 如果事务 ID < ReadView 最小 ID → 已提交，可见
- 如果事务 ID > ReadView 最大 ID → 未提交，不可见
- 在区间内 → 检查活跃事务列表

### 3. 索引设计实战

**什么情况下索引会失效**？
- `LIKE '%keyword'`（左模糊）
- 对索引列使用函数：`WHERE YEAR(create_time) = 2026`
- 隐式类型转换：`WHERE varchar_column = 123`
- OR 条件中有一列无索引
- 联合索引不满足最左前缀

## 五、Redis 高阶

### 1. 缓存一致性方案

| 方案 | 原理 | 缺点 |
|------|------|------|
| 延迟双删 | 先删缓存 → 更新DB → 延迟再删 | 延迟时间难确定 |
| Canal 监听 Binlog | 监听 MySQL Binlog 异步更新缓存 | 架构复杂，引入 Canal |
| 读写穿透 | 读缓存→miss读DB写缓存；写DB同时写缓存 | 强一致但性能低 |

**推荐方案**：Canal + MQ 异步更新缓存，最终一致性

### 2. Redis 大 Key 问题

**如何发现和解决大 Key**？
- `redis-cli --bigkeys` 扫描
- 拆分：Hash 按 field 分片、List 按 segment 分片
- 避免：单 Hash 超过 5000 field 时考虑拆分

### 3. Redis 分布式锁 RedLock 争议

**RedLock 问题**：
- 时钟漂移导致锁过期
- GC 停顿导致持有超期
- 业界趋势：改用 ZooKeeper/etcd 实现强一致分布式锁

## 七、系统设计高频场景题

### Q1: 如何设计一个百万级 QPS 的秒杀系统？

**分层削峰架构**：
```
CDN → Nginx限流 → Redis预减库存(Lua) → 消息队列 → 异步下单 → DB
```

1. **CDN 层**：静态资源缓存，减少回源
2. **Nginx 层**：限流（令牌桶/漏桶）、黑名单
3. **Redis 层**：Lua 脚本原子扣减库存，避免超卖
4. **MQ 层**：RabbitMQ/RocketMQ 异步处理下单请求
5. **DB 层**：最终持久化，乐观锁防超卖

### Q2: 如何设计一个高可用消息队列？

**核心指标**：
- 消息不丢失：生产者确认 + Broker 持久化 + 消费者手动 ACK
- 消息不重复：幂等消费（唯一 ID + 去重表）
- 顺序消费：同一 partition/key 的消息有序
- 高吞吐：零拷贝、顺序写磁盘、批量发送

### Q3: 如何设计短链接系统？

**核心方案**：
- **发号器**：Snowflake 或 DB 自增 ID → Base62 编码（a-zA-Z0-9）
- **存储**：短码 → 长 URL 映射，Redis 缓存热点
- **统计**：异步记录点击次数、地理位置、设备信息
- **过期**：TTL 机制，长期未访问的短链自动清理

## 八、MySQL 锁机制与并发控制（2026-04-23 新增）

### 1. 非唯一索引等值查询的加锁机制（腾讯 TME 二面真题）

**问题**：一个表，name 字段是非唯一索引，有多条 name=张三，查询 `where name='张三'`，是什么锁？

**参考答案**：
- **Next-Key Lock（临键锁）** = Record Lock（记录锁）+ Gap Lock（间隙锁）
- 非唯一索引等值查询会锁定所有 name='张三' 的记录 + 它们之间的间隙
- **目的**：防止其他事务插入新的 name='张三'（避免幻读）
- **影响**：其他事务无法在 gap 中插入新记录

### 2. 用户余额扣减的并发安全（腾讯 TME 二面真题）

**问题**：用户余额扣减，只在 MySQL 层面，如何保证并发安全？

**参考答案**：
- **推荐方案（乐观锁式）**：`UPDATE account SET balance = balance - ? WHERE id = ? AND balance >= ?`
  - 利用数据库原子更新 + 条件判断，无需显式加锁
  - 通过 affected rows 判断是否成功
- **悲观锁方案**：`SELECT balance FROM account WHERE id = ? FOR UPDATE`（行级排他锁）

### 3. 哪些 SQL 语句会加锁？（拼多多一面真题）

**参考答案**：
- `SELECT ... FOR UPDATE` — 排他锁（X Lock）
- `SELECT ... LOCK IN SHARE MODE` — 共享锁（S Lock）
- `UPDATE` / `DELETE` — 排他锁
- `INSERT` — 排他锁 + 可能的间隙锁
- 唯一索引等值查询 → Record Lock；非唯一索引等值/范围查询 → Next-Key Lock

### 4. 什么是幻读？怎么解决？（拼多多一面真题）

**参考答案**：
- **幻读**：同一事务中，两次相同条件的查询返回的结果集不同
- **MySQL RR 级别**：通过 MVCC（快照读）+ Next-Key Lock（当前读）解决幻读
- **串行化**：所有读操作都加锁，彻底杜绝幻读（性能最低）

## 九、JVM 字节码与对象模型（2026-04-23 新增）

### 1. new 对象涉及哪些 JVM 结构？（拼多多一面真题）

**参考答案**：
- **堆（Heap）**：对象实例分配在新生代 Eden 区
- **虚拟机栈**：局部变量引用存储在栈帧的局部变量表中

### 2. 字节码的作用是什么？（拼多多一面真题）

**参考答案**：
- Java 源码编译为 `.class` 字节码，是平台无关的中间格式
- JVM 类加载器加载字节码，解释器/JIT 编译器执行
- 跨平台保证：一次编译，到处运行
- 字节码验证确保安全性
- AOP/动态代理通过字节码增强（CGLIB、ByteBuddy）实现

## 十、操作系统与并发场景（2026-04-23 新增）

### 1. 操作系统层面：为什么要有堆和栈？（拼多多一面真题）

**参考答案**：
- **栈**：LIFO 结构，函数调用栈、局部变量，自动分配/释放，速度快，空间有限
- **堆**：动态内存分配（new/malloc），手动管理生命周期，空间大，分配较慢
- **分离原因**：职责分离提高管理效率，函数调用的嵌套特性天然适合栈，对象生命周期不确定需要堆

### 2. Java 中进程和线程的表现形式（拼多多一面真题）

**参考答案**：
- 一个 JVM 实例就是一个进程（独立地址空间）
- Java 线程映射为 OS 原生线程（1:1 模型），`java.lang.Thread` 封装了 OS 线程
- 每个线程有独立的程序计数器、虚拟机栈、本地方法栈；堆和方法区线程共享
- 虚拟线程（Java 21+）：用户态轻量线程，M:N 模型

### 3. 双核 CPU 上的线程池调度（拼多多一面真题）

**问题**：双核 CPU 对一个 100core/200max 的线程池是怎样调度和使用的？

**参考答案**：
- corePoolSize/maxPoolSize 是逻辑上限，不等于实际并发执行数
- 双核 CPU 同一时刻最多只有 2 个线程在真正运行
- 超出 CPU 核心数的线程处于就绪/阻塞状态，等待 OS 调度器分配时间片

### 4. 多线程文件统计场景（拼多多一面真题）

**问题**：多个文件万级数据量，用线程池分散统计，有没有线程安全问题？

**参考答案**：
- 每个线程独立读取不同文件、独立统计 → 无线程安全问题
- 设计：每个文件一个 Callable Task → `submit()` 提交 → `CompletableFuture` 收集结果 → 主线程汇总
- 如果多个线程写同一个统计容器，需要 `ConcurrentHashMap`

## 十一、缓存模式与布隆过滤器（2026-04-23 新增）

### 1. CacheAside 工作模式（致远互联一面真题）

**问题**：CacheAside 模式是怎么样的？会不会出现数据不一致？先写数据库和后写数据库有什么区别？

**参考答案**：
- **模式**：读 → 先读缓存，miss 则读 DB 写缓存；写 → 先更新 DB，再删除缓存（注意是删除！）
- **不一致场景**：更新 DB 成功但删缓存失败；并发读可能回填旧数据
- **先写 DB 再删缓存**（推荐）：不一致窗口短
- **先删缓存再写 DB**：并发读可能回填旧数据，不一致窗口长

### 2. 布隆过滤器（致远互联一面真题）

**参考答案**：
- 使用 k 个哈希函数将元素映射到位数组，对应位置置为 1
- 查询：k 个位置全为 1 则"可能存在"，否则"一定不存在"
- 空间效率极高，存在误判率（False Positive），不支持删除
- 应用：缓存穿透防护、垃圾邮件过滤、URL 去重

### 3. Redis HyperLogLog（文友科技一面真题）

**参考答案**：
- 不准确，约 0.81% 标准误差，内存仅 12KB（可统计 2^64 个元素）
- 基于 LogLog 算法，记录每个桶中前导零的最大值，用调和平均数估算基数
- 适用：UV 统计、独立访客计数

## 十三、架构设计与安全认证（2026-04-23 新增）

### 1. 单例模式多种实现（致远互联一面真题）

- **双重校验锁（DCL）**：volatile + 双重检查
- **静态内部类**：类加载机制，线程安全且懒加载（推荐）
- **枚举单例**：天然防反射、防序列化破坏（最安全）
- **饿汉式**：类加载时初始化
- **volatile 作用**：防止 `new` 的指令重排序（分配内存→初始化→赋值引用）

### 2. MVC vs DDD（致远互联一面真题）

- **MVC**：关注 UI 层分离，适合传统 Web、CRUD，局限：业务逻辑易堆积
- **DDD**：关注领域建模（实体、聚合根、领域服务），适合复杂业务、微服务拆分
- DDD 优势：富血模型，业务逻辑内聚，限界上下文

### 3. 链表环检测（致远互联一面真题）

- **快慢指针**（Floyd 判圈算法）：慢走 1 步快走 2 步，有环则快追上慢，O(n) 时间 O(1) 空间
- **哈希表法**：遍历存入 Set，遇到已存在则有环，O(n) 空间

### 4. SaToken vs Spring Security（文友科技一面真题）

- **SaToken**：轻量级，API 简洁，功能全面（登录认证、权限、Session、SSO、OAuth2）
- **Spring Security**：官方框架，功能强大生态完善，配置复杂学习曲线陡
- 选型：中小项目优先 SaToken，大型复杂项目用 Spring Security

### 5. 单点登录（SSO）实现（文友科技一面真题）

- 一个认证中心统一管理认证，多个应用信任认证中心
- 流程：访问应用 → 跳转 SSO Server → 认证通过生成 Ticket → 重定向回应用 → 验证 Ticket
- 方案：CAS 协议、OAuth2/OIDC、JWT + Redis 共享 Session

### 6. 接口和抽象类的区别（致远互联一面真题）

| 维度 | 接口 | 抽象类 |
|------|------|--------|
| 继承 | 多实现 | 单继承 |
