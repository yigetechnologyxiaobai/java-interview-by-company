## 一、Java 基础与集合进阶

### 1. ConcurrentHashMap Segment 分段锁的弊端（腾讯 TME 二面真题）

**问题**：ConcurrentHashMap 的 Segment 分段锁有什么弊端？

**参考答案**：
- **JDK 1.7 分段锁**：将 Hash 表分成多个 Segment，每个 Segment 独立加锁，并发度等于 Segment 数量（默认16）
- **弊端**：
  1. 并发度受限：最多只能有 16 个线程同时写，超过则阻塞
  2. 内存开销大：每个 Segment 独立继承 ReentrantLock，额外内存占用高
  3. 扩容复杂：需要逐段扩容，扩容期间影响并发性能
- **JDK 1.8 改进**：改用 CAS + synchronized（锁单个桶），并发度等于数组长度，大幅提升性能

### 2. StringBuffer 线程安全原理（腾讯 TME 二面真题）

**问题**：StringBuilder 和 StringBuffer 的区别？StringBuffer 如何保证线程安全？

**参考答案**：
- **区别**：StringBuffer 所有方法都用 `synchronized` 修饰，StringBuilder 无锁
- **线程安全原理**：通过 `synchronized` 关键字对整个方法加锁，同一时刻只有一个线程可以执行 append/insert/delete 等操作
- **弊端**：全局锁导致并发性能差，多线程下竞争激烈

### 3. 包装类缓存陷阱（腾讯 TME 二面真题）

**问题**：`int a=2; Integer b=2; a==b` 和 `Integer a=200; Integer b=200; a==b` 的结果分别是什么？

**参考答案**：
- **第一题**：`true` — int 与 Integer 比较时自动拆箱，比较值
- **第二题**：`false` — Integer 缓存范围为 [-128, 127]，200 超出范围，每次 `new Integer` 创建不同对象，`==` 比较引用地址
- **原理**：`Integer.valueOf()` 内部维护了 `IntegerCache`，缓存 [-128, 127] 的 Integer 实例

### 4. 接口和抽象类的区别（致远互联一面真题）

| 维度 | 接口（Interface） | 抽象类（Abstract Class） |
|------|-------------------|-------------------------|
| 继承方式 | 多实现（implements） | 单继承（extends） |
| 成员变量 | 只能有 `public static final` 常量 | 可以有各种类型成员变量 |
| 构造方法 | 无构造方法 | 有构造方法，供子类调用 |
| 方法修饰 | 默认 `public abstract`（Java 8+ 支持 default/static） | 可以有各种访问修饰符 |
| 设计理念 | "能不能做"（能力契约） | "是不是"（is-a 关系） |
| 使用场景 | 定义行为规范、解耦 | 抽取公共代码、模板方法 |

## 二、并发编程与多线程

### 1. 双核 CPU 上线程池调度机制（拼多多从鲸科技一面真题）

**问题**：双核 CPU 对一个 100core/200max 的线程池是怎样调度和使用的？

**参考答案**：
- **核心原理**：线程池的 corePoolSize/maxPoolSize 是逻辑上限，不等于实际并发执行数
- **双核 CPU 实际情况**：同一时刻最多只有 2 个线程在真正运行（CPU 核心数决定）
- **调度过程**：
  1. 线程池内部管理任务队列和线程生命周期
  2. 操作系统调度器负责将线程分配到 CPU 核心上执行
  3. 超出 CPU 核心数的线程处于就绪/阻塞状态，等待时间片切换
- **设计建议**：CPU 密集型任务建议线程数 = CPU 核心数 + 1；I/O 密集型可适当放大

### 2. 多线程文件统计场景题（拼多多从鲸科技一面真题）

**问题**：多个文件中存有学生信息和成绩，单个文件万级数据量，用线程池分散统计分段功能，有没有线程安全问题？怎么设计和实现？

**参考答案**：
- **线程安全分析**：如果每个线程独立读取不同文件、独立统计，不共享可变状态 → 无线程安全问题
- **设计方案**：
  1. 每个文件分配一个 Task（实现 Callable），返回该文件的统计结果
  2. 使用 `ExecutorService.submit()` 提交所有任务
  3. 用 `CompletableFuture` 或 `invokeAll()` 收集所有结果
  4. 主线程汇总各 Task 结果
- **注意**：如果多个线程写同一个统计容器（如 Map），需要 `ConcurrentHashMap` 或 `Collections.synchronizedMap`

### 3. 操作系统层面：为什么要有堆和栈？（拼多多从鲸科技一面真题）

**参考答案**：
- **栈（Stack）**：
  - 用途：函数调用栈、局部变量、方法参数
  - 特点：LIFO 结构，自动分配/释放，速度快，空间有限
  - 原因：函数调用的嵌套特性天然适合栈结构
- **堆（Heap）**：
  - 用途：动态内存分配（new 对象、malloc）
  - 特点：手动管理生命周期（GC/手动释放），空间大，分配较慢
  - 原因：对象生命周期不确定，需要灵活的内存管理
- **分离原因**：职责分离提高管理效率，栈的自动管理 + 堆的灵活分配

### 4. Java 中进程和线程的表现形式（拼多多从鲸科技一面真题）

**参考答案**：
- **进程**：Java 中一个 JVM 实例就是一个进程（独立地址空间、独立资源）
- **线程**：Java 线程映射为 OS 原生线程（1:1 模型）
  - `java.lang.Thread` 封装了 OS 线程
  - 每个线程有独立的程序计数器、虚拟机栈、本地方法栈
  - 堆和方法区是线程共享的
- **虚拟线程（Java 21+）**：用户态轻量线程，M:N 模型，多个虚拟线程映射到少量 OS 线程

## 三、JVM 与字节码

### 1. 字节码的作用是什么？（拼多多从鲸科技一面真题）

**参考答案**：
- **中间表示**：Java 源代码编译为 `.class` 字节码，平台无关的中间格式
- **JVM 执行基础**：JVM 的类加载器加载字节码，解释器/JIT 编译器执行
- **跨平台保证**：一次编译，到处运行（Write Once, Run Anywhere）
- **安全性**：字节码验证（Bytecode Verifier）确保代码安全性
- **AOP/动态代理**：通过字节码增强（CGLIB、ByteBuddy）实现动态代理

### 2. new 对象涉及哪些 JVM 结构？（拼多多从鲸科技一面真题）

**问题**：方法中 new 一个对象并赋值给局部变量，涉及到哪些 JVM 结构？

**参考答案**：
- **堆（Heap）**：对象实例分配在堆上（新生代 Eden 区）
- **虚拟机栈（VM Stack）**：局部变量引用存储在栈帧的局部变量表中
- **方法区/元空间（Metaspace）**：对象的 Class 元数据（类结构、方法信息）存储在方法区
- **执行流程**：`new` 指令 → 检查类是否加载 → 分配内存 → 初始化零值 → 执行 `<init>` 构造方法 → 返回引用压入操作数栈

## 四、MySQL 进阶

### 1. 非唯一索引查询加锁类型（腾讯 TME 二面真题）

**问题**：一个表，name 字段是非唯一索引，有多条 name=张三，查询 `where name='张三'`，是什么锁？

**参考答案**：
- **Next-Key Lock（临键锁）** = Record Lock（记录锁）+ Gap Lock（间隙锁）
- 非唯一索引等值查询会锁定：
  1. 所有 `name='张三'` 的记录（Record Lock）
  2. 这些记录之间的间隙（Gap Lock），防止其他事务插入新的 `name='张三'`
- **影响**：其他事务无法在 gap 中插入新记录，也无法修改已锁定的记录

### 2. 用户余额扣减的并发安全（腾讯 TME 二面真题）

**问题**：一个表，用户余额进行扣减，只在 MySQL 层面，如何保证并发安全？

**参考答案**：
- **方案1：乐观锁** — `UPDATE account SET balance = balance - ? WHERE id = ? AND balance >= ?`
  - 利用数据库原子更新，无需显式加锁
- **方案2：悲观锁** — `SELECT balance FROM account WHERE id = ? FOR UPDATE`
  - 行级排他锁，同一行只能被一个事务锁定
- **方案3：CAS 式更新** — 条件更新 `UPDATE account SET balance = balance - ? WHERE id = ? AND balance >= ?`，检查 affected rows > 0

### 3. MySQL 什么情况下加记录锁？（腾讯 TME 二面真题）

**参考答案**：
- 通过**唯一索引**进行等值查询，且记录存在时 → Record Lock
- 通过**主键**等值查询，记录存在时 → Record Lock
- `SELECT ... FOR UPDATE` 或 `SELECT ... LOCK IN SHARE MODE` 显式加锁
- **注意**：唯一索引等值查询但记录不存在时，加 Gap Lock（防止幻读）

### 4. 什么是幻读？怎么解决？（拼多多从鲸科技一面真题）

**参考答案**：
- **幻读**：同一事务中，两次相同条件的查询返回的结果集不同（其他事务插入/删除了记录）
- **MySQL 默认隔离级别（RR）**：通过 MVCC + Next-Key Lock 解决幻读
  - 快照读：通过 MVCC 读取历史快照，避免幻读
  - 当前读：通过 Next-Key Lock 锁定范围，防止插入
- **串行化（Serializable）**：所有读操作都加锁，彻底杜绝幻读（性能最低）

### 5. 哪些 SQL 语句会加锁？（拼多多从鲸科技一面真题）

**参考答案**：
- `SELECT ... FOR UPDATE` — 排他锁（X Lock）
- `SELECT ... LOCK IN SHARE MODE` — 共享锁（S Lock）
- `UPDATE` / `DELETE` — 排他锁
- `INSERT` — 排他锁 + 可能的间隙锁
- 唯一索引等值查询 → Record Lock
- 非唯一索引等值/范围查询 → Next-Key Lock（Record + Gap）

## 五、Redis 高阶

### 1. Redis 持久化：AOF 重复写入同一 Key（腾讯 TME 二面真题）

**问题**：先后执行 `set name 张三` 和 `set name 李四`，在 AOF 持久化过程中都写入 AOF 文件吗？

**参考答案**：
- **是的，都会写入** — AOF 是 append-only 日志，每次写操作都追加到文件末尾
- **原因**：AOF 记录的是操作命令（`SET name 张三`、`SET name 李四`），不关心 Key 是否已存在
- **AOF 重写**：Redis 后台会定期执行 AOF Rewrite，生成精简版 AOF 文件（只保留每个 Key 的最终值），减少文件体积

### 2. Redis Cluster 集群的 Key 分配（腾讯 TME 二面真题）

**问题**：Redis Cluster 集群如何分配 Key 在哪个 Slot？

**参考答案**：
- **16384 个 Slot**：Redis Cluster 将数据划分为 16384 个哈希槽
- **分配算法**：`CRC16(key) % 16384` → 确定 Slot 编号
- **节点分配**：每个 Cluster 节点负责一部分 Slot 区间（如节点 A 负责 0-5460）
- **Hash Tag**：使用 `{user100}` 包裹 Key 的一部分，确保相关 Key 分配到同一 Slot（如 `{user100}.name` 和 `{user100}.age`）

### 3. Redis String 类型大 Value 问题（腾讯 TME 二面真题）

**问题**：String 类型的 value 如果是几十兆，会怎么样？

**参考答案**：
- **阻塞风险**：Redis 单线程模型，大 Value 的序列化/反序列化、网络传输会阻塞其他命令
- **内存碎片**：大对象分配导致内存碎片率升高
- **主从同步延迟**：大 Value 传输增加复制缓冲区压力
- **解决方案**：
  1. 拆分存储：将大 Value 拆成多个小 Key（如 `key:0`, `key:1`）
  2. 压缩存储：使用 Gzip/Snappy 压缩后存入
  3. 迁移到文件系统：大对象存入 OSS/HDFS，Redis 只存 URL 引用

### 4. Redis Set 元素个数过多怎么办？（腾讯 TME 二面真题）

**参考答案**：
- **拆分 Set**：按 Hash 分组，拆成多个小 Set（如 `set:0`, `set:1`）
- **使用 Bitmap**：如果元素是连续整数 ID，用 Bitmap 更省内存
- **使用 HyperLogLog**：如果只需要统计去重数量（如 UV），用 HLL 误差 0.81%，仅占 12KB
- **使用 ZSet 替代**：如果需要排序 + 去重，可考虑 ZSet
- **内存优化**：检查是否开启了 `set-max-intset-entries` 整数集合编码优化

### 5. CacheAside 工作模式与数据不一致（致远互联一面真题）

**问题**：CacheAside 工作模式是怎么样的？会不会出现数据不一致？先写数据库和后写数据库有什么区别？

**参考答案**：
- **CacheAside 模式**：
  1. 读：先读缓存 → miss 则读 DB → 写缓存
  2. 写：先更新 DB → 再删除缓存（注意是删除，不是更新）
- **数据不一致场景**：
  - 写操作：更新 DB 成功，删除缓存失败 → 缓存旧数据
  - 并发场景：线程 A 读缓存 miss → 线程 B 更新 DB 并删缓存 → 线程 A 读到旧 DB 写回缓存
- **先写 DB vs 先写缓存**：
  - **先更新 DB，再删缓存**（推荐）：不一致窗口短，配合重试/消息队列保证最终一致
  - 先删缓存，再更新 DB：并发读可能回填旧数据，不一致窗口长

### 6. 12306 超卖场景题（腾讯 TME 二面真题）

**问题**：类似 12306 买车票，如何保证多个请求不会发生超卖？

**参考答案**：
- **Redis 预扣减**：库存预热到 Redis，用 Lua 脚本原子扣减（`DECR` + 判断）
- **数据库乐观锁**：`UPDATE ticket SET stock = stock - 1 WHERE train_id = ? AND stock > 0`
- **分布式锁**：对同一车次加锁，串行处理（性能差，不推荐）
- **消息队列削峰**：请求入 MQ，消费者串行处理扣减
- **推荐架构**：Redis Lua 预扣减 → 成功则入 MQ → 异步持久化到 DB

## 六、缓存与布隆过滤器

### 1. 布隆过滤器原理（致远互联一面真题）

**参考答案**：
- **原理**：使用 k 个哈希函数将元素映射到长度为 m 的位数组，对应位置置为 1
- **查询**：检查 k 个位置是否都为 1，全为 1 则"可能存在"，否则"一定不存在"
- **特点**：
  - 空间效率极高（比 Hash Set 省数十倍空间）
  - 存在误判率（False Positive），但不会漏判（False Negative）
  - 不支持删除（除非用 Counting Bloom Filter）
- **应用场景**：缓存穿透防护（过滤不存在的 Key）、垃圾邮件过滤、爬虫 URL 去重

### 2. Redis HyperLogLog 准确性（文友科技一面真题）

**问题**：Redis HyperLogLog 是准确的吗？底层实现？

**参考答案**：
- **不准确**：有约 0.81% 的标准误差，但内存占用仅 12KB（可统计 2^64 个元素）
- **底层实现**：基于概率计数的 LogLog 算法改进
  - 将元素哈希后分桶，记录每个桶中前导零的最大值
  - 用调和平均数估算基数
- **适用场景**：UV 统计、独立访客计数、去重计数（不需要精确值的场景）

## 七、Spring 框架进阶

### 1. Spring 扩展机制（致远互联一面真题）

**问题**：Spring 有哪些扩展机制？容器层面或 Bean 层面？怎么动态加载 Bean？

**参考答案**：
- **Bean 层面**：
  - `BeanPostProcessor`：Bean 初始化前后拦截（AOP 代理在此阶段创建）
  - `BeanFactoryPostProcessor`：Bean 定义加载后、实例化前修改 Bean 定义
  - `InitializingBean` / `@PostConstruct`：Bean 初始化回调
- **容器层面**：
  - `ImportBeanDefinitionRegistrar`：自定义 Bean 注册（如 `@MapperScan` 的实现）
  - `BeanDefinitionRegistryPostProcessor`：注册额外 Bean 定义
  - `ApplicationContextInitializer`：容器刷新前初始化
- **动态加载 Bean**：
  1. 使用 `BeanDefinitionRegistry.registerBeanDefinition()`
  2. 使用 `@Import(ImportBeanDefinitionRegistrar.class)` 自定义注册逻辑
  3. 编程式注册：`DefaultListableBeanFactory.registerSingleton()`

### 2. 声明式事务失效的情况（致远互联一面真题）

**参考答案**：
1. **非 public 方法**：`@Transactional` 仅对 public 方法生效
2. **同类内部调用**：A 方法调用 B 方法（都在同一类中），B 的事务注解不生效（自调用绕过代理）
3. **异常被 catch**：方法内部捕获了异常，未抛出，事务不回滚
4. **异常类型不匹配**：默认只回滚 RuntimeException/Error，checked exception 需指定 `rollbackFor`
5. **多线程**：子线程中的数据库操作不在主线程事务范围内
6. **数据库引擎不支持**：MySQL MyISAM 不支持事务，需 InnoDB

## 八、架构设计

### 1. 单例模式多种实现方式（致远互联一面真题）

**问题**：单例模式，懒汉模式下除了双重校验锁还有哪些方式实现？为什么要用 volatile？

**参考答案**：
- **实现方式**：
  1. **双重校验锁（DCL）**：
     ```java
     public class Singleton {
         private static volatile Singleton instance;
         public static Singleton getInstance() {
             if (instance == null) {
                 synchronized (Singleton.class) {
                     if (instance == null) {
                         instance = new Singleton();
                     }
                 }
             }
             return instance;
         }
     }
     ```
  2. **静态内部类（推荐）**：利用类加载机制，线程安全且懒加载
  3. **枚举单例（最安全）**：天然防反射、防序列化破坏
  4. **饿汉式**：类加载时初始化，非懒加载
- **volatile 作用**：防止指令重排序。`new Singleton()` 分三步：分配内存 → 初始化 → 赋值引用。如果没有 volatile，其他线程可能看到初始化未完成的不完整对象。

### 2. 链表环检测（致远互联一面真题）

**参考答案**：
- **快慢指针（Floyd 判圈算法）**：
  - 慢指针每次走 1 步，快指针每次走 2 步
  - 如果有环，快指针一定会追上慢指针
  - 时间复杂度 O(n)，空间复杂度 O(1)
- **哈希表法**：遍历链表，将每个节点存入 Set，遇到已存在的节点则有环（空间 O(n)）

### 3. 架构模式：MVC vs DDD（致远互联一面真题）

**参考答案**：
- **MVC（Model-View-Controller）**：
  - 关注点：UI 层分离（展示、控制、数据模型）
  - 适用场景：传统 Web 应用、CRUD 系统
  - 局限：业务逻辑容易堆积在 Service 层，形成"贫血模型"
- **DDD（领域驱动设计）**：
  - 关注点：业务领域建模（实体、值对象、聚合根、领域服务）
  - 适用场景：复杂业务系统、微服务拆分
  - 优势：富血模型，业务逻辑内聚在领域对象中
  - 核心概念：限界上下文（Bounded Context）、聚合（Aggregate）、领域事件

### 4. HTTP 报文结构（腾讯 TME 二面真题）

**参考答案**：
- **请求报文**：请求行（方法 + URL + 协议版本）+ 请求头 + 空行 + 请求体
- **响应报文**：状态行（协议版本 + 状态码 + 状态描述）+ 响应头 + 空行 + 响应体
- **响应体类型**：
  - 文本：`text/html`, `application/json`, `text/plain`
  - 二进制：`application/octet-stream`, `image/png`
  - 压缩：`gzip`, `br`（Brotli）
- **HTTP 基于 TCP**：TCP 提供可靠传输，HTTP 是应用层协议

### 5. TCP 三次握手和四次挥手（腾讯 TME 二面真题）

**问题**：TCP 三次握手和四次挥手过程，为什么不是五次六次？

**参考答案**：
- **三次握手**：
  1. 客户端 → 服务端：SYN（客户端序列号 x）
  2. 服务端 → 客户端：SYN-ACK（服务端序列号 y，确认号 x+1）
  3. 客户端 → 服务端：ACK（确认号 y+1）
  - 最少 3 次：防止历史连接初始化、同步双方初始序列号
- **四次挥手**：
  1. 客户端 → 服务端：FIN（客户端无数据发送）
  2. 服务端 → 客户端：ACK（确认收到 FIN）
  3. 服务端 → 客户端：FIN（服务端数据发送完毕）
  4. 客户端 → 服务端：ACK（确认收到 FIN）
  - 不能合并为 3 次：因为服务端收到 FIN 后可能还有数据要发送，ACK 和 FIN 不能同时发出

## 九、安全与认证

### 1. SaToken 讲解及与 Spring Security 区别（文友科技一面真题）

**参考答案**：
- **SaToken**：轻量级 Java 权限认证框架
  - 特点：API 简洁、功能全面（登录认证、权限验证、Session 管理、单点登录、OAuth2）
  - 优势：上手简单、文档友好、Spring Boot 集成方便
- **Spring Security**：Spring 官方安全框架
  - 特点：功能强大、生态完善、可定制性高
  - 劣势：配置复杂、学习曲线陡峭
- **选型建议**：中小项目优先 SaToken，大型复杂项目或已有 Spring 生态用 Spring Security

### 2. 单点登录（SSO）实现（文友科技一面真题）

**参考答案**：
- **原理**：一个认证中心（SSO Server）统一管理用户认证，多个应用系统（SSO Client）信任认证中心
- **流程**：
  1. 用户访问系统 A → 未登录 → 跳转到 SSO Server
  2. SSO Server 认证通过 → 生成全局 Ticket → 重定向回系统 A
  3. 系统 A 用 Ticket 向 SSO Server 验证 → 创建本地 Session
  4. 用户再访问系统 B → 跳转到 SSO Server → SSO Server 已有全局 Session → 直接发 Ticket → 系统 B 验证通过
- **实现方案**：CAS 协议、OAuth2 / OIDC、JWT + Redis 共享 Session
