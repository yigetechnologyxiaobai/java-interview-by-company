## 一、Java 并发编程与虚拟线程（2026年社招重点）

### 1. 虚拟线程深度考察（蚂蚁国际一面真题）

**问题**：虚拟线程和传统线程的区别是什么？为什么虚拟线程可以降低上下文切换开销？

**参考答案**：
- **传统线程**：内核级线程，每个线程映射一个 OS 线程，内存开销约 1MB，上下文切换需要进入内核态
- **虚拟线程**：用户态轻量级线程，由 JVM 调度器管理，内存开销仅几百字节，可以创建数百万个
- **性能优势**：适合 I/O 密集型任务（数据库查询、HTTP 请求），减少系统调用和内核切换开销
- **劣势**：不适合 CPU 密集型计算；在 synchronized 代码块中会导致 pinning（线程 pinned 到载体线程），应改用 ReentrantLock
- **排查方法**：通过 `jcmd <pid> Thread.print` 查看虚拟线程状态，使用 `jdk.jfr` 记录虚拟线程调度事件

**追问**：Java 中有哪些方式可以避免多线程争抢同一资源？

**参考答案**：
1. `synchronized` 关键字（内置锁，JVM 优化完善）
2. `ReentrantLock`（显式锁，支持公平/非公平、可中断）
3. `volatile` 关键字（保证可见性，不保证原子性）
4. `Atomic` 原子类（CAS 操作，无锁并发）
5. `ConcurrentHashMap` 等并发容器
6. 信号量 `Semaphore`、读写锁 `ReadWriteLock`
7. 虚拟线程 + 结构化并发（Java 21+）

### 2. 锁升级机制（蚂蚁国际真题）

**问题**：Synchronized 锁升级流程是什么？轻量级/重量级具体指什么？

**参考答案**：
- **无锁 → 偏向锁**：首个线程访问时，将 Mark Word 指向该线程 ID
- **偏向锁 → 轻量级锁**：发生竞争时，通过 CAS 将 Mark Word 替换为锁记录指针，使用自旋等待
- **轻量级锁 → 重量级锁**：自旋超过阈值或竞争激烈时，膨胀为重量级锁，线程进入 OS 互斥量等待队列
- **轻量级锁**：通过 CAS 竞争，线程不阻塞，适合锁持有时间短的场景
- **重量级锁**：通过 OS 互斥量（Mutex）实现，线程阻塞挂起，适合锁持有时间长的场景

### 3. volatile 与内存可见性（蚂蚁国际真题）

**问题**：volatile 的原理是什么？不用 volatile 如何实现内存可见性？

**参考答案**：
- **volatile 原理**：通过内存屏障（Memory Barrier）禁止指令重排序，强制写入主存，保证可见性
- **替代方案**：
  1. 使用 `synchronized` 或 `Lock` 加锁
  2. 使用 `AtomicInteger`、`AtomicReference` 等原子类
  3. 使用 `final` 变量（构造完成后不可变，天然可见）
  4. 使用 `ConcurrentHashMap` 等并发容器（内部使用 volatile 和 CAS）

### 4. 并发请求统计失败率场景题（阿里虎鲸文娱真题）

**问题**：并发请求场景下，如何统计下游接口的失败率？如何在不影响原接口性能的前提下实现并发 counter 统计？

**参考答案**：
- 使用 `LongAdder`（JDK 8+）替代 `AtomicLong`，减少 CAS 竞争
- `LongAdder` 内部采用分段累加（Cell 数组），多线程各自累加自己的 Cell，最后汇总
- 失败率计算：`失败率 = LongAdder(失败数).sum() / LongAdder(总请求数).sum()`
- 异步统计：通过 `CompletableFuture` 异步记录统计结果，不影响主流程

## 二、AI Agent 与 LLM 工程化（2026年面试新增热点）

### 1. AI Agent 架构模式（阿里一面真题）

**问题**：常见的 Agent 架构模式有哪些？

**参考答案**：
1. **ReAct 模式**：Reasoning + Acting，模型交替执行思考和动作
2. **Planner-Executor 模式**：规划器生成任务计划，执行器逐一执行
3. **Supervisor 模式**：一个主 Agent 协调多个子 Agent 完成任务
4. **Multi-Agent 协作**：多个 Agent 通过消息队列或 A2A 协议协作
5. **Tool Use / Function Calling**：模型根据描述生成结构化数据调用工具

### 2. Function Calling 原理与问题处理（阿里一面真题）

**问题**：Function Calling 的原理是什么？如何处理大模型 Function Call 参数格式错误？

**参考答案**：
- **原理**：模型根据函数描述（名称、参数、返回类型）生成符合 JSON Schema 的结构化数据，框架解析后调用对应函数
- **参数格式错误处理**：
  1. **重试机制**：捕获解析异常，将错误信息反馈给模型重新生成
  2. **JSON Schema 强约束**：使用严格的 Schema 定义，限制参数类型和格式
  3. **预填充（Prefilling）**：在 Prompt 中预填充部分 JSON 结构引导模型
  4. **后处理校验**：对模型输出进行格式校验和修正

### 3. MCP 协议理解（阿里一面真题）

**问题**：MCP（Model Context Protocol）是什么？与传统 Tool 注册有什么区别？

**参考答案**：
- **MCP**：模型上下文协议，被称为"AI 领域的 USB-C 接口"，标准化了模型与工具之间的交互
- **与传统 Tool 注册的区别**：
  1. **标准化接口**：MCP 定义统一协议，不同厂商的工具可以即插即用
  2. **动态发现**：支持运行时发现可用工具，无需硬编码
  3. **多语言支持**：协议无关，支持 Python、Java、Node.js 等多种语言
  4. **安全隔离**：工具运行在独立沙箱中，降低安全风险

## 三、Spring 框架与架构设计

### 1. Spring Boot 自动装配原理（阿里一面真题）

**参考答案**：
1. `@SpringBootApplication` 组合了 `@EnableAutoConfiguration`
2. 通过 `spring.factories` 或 `org.springframework.boot.autoconfigure.AutoConfiguration.imports` 加载自动配置类
3. 自动配置类通过 `@Conditional` 注解判断条件是否满足（如类是否存在、属性是否配置）
4. 满足条件则注册 Bean 到 IoC 容器

### 2. Bean 生命周期（多公司高频）

**参考答案**：
1. **实例化**：通过构造函数或工厂方法创建 Bean 实例
2. **属性注入**：通过 `@Autowired`、`@Value` 等注入依赖
3. **Aware 回调**：调用 `BeanNameAware`、`BeanFactoryAware` 等
4. **初始化前**：`BeanPostProcessor.postProcessBeforeInitialization`
5. **初始化**：`@PostConstruct`、`afterPropertiesSet`、`init-method`
6. **初始化后**：`BeanPostProcessor.postProcessAfterInitialization`（AOP 代理在此阶段创建）
7. **就绪使用**：Bean 放入容器，可被其他 Bean 依赖
8. **销毁**：`@PreDestroy`、`destroy-method`

### 3. SpringBoot 分层与 Request 流程（亚信安全真题）

**分层架构**：
- **Controller 层**：处理 HTTP 请求，参数校验
- **Service 层**：核心业务逻辑
- **DAO/Repository 层**：数据库操作
- **DTO/VO 层**：数据传输对象

**Request 流程**：
1. 请求到达 Tomcat Servlet 容器
2. `DispatcherServlet` 接收请求
3. 通过 `HandlerMapping` 查找对应的 Controller 方法
4. 执行拦截器 `preHandle`
5. 参数解析与绑定（`HandlerAdapter`）
6. 执行 Controller 方法
7. 执行拦截器 `postHandle`
8. 视图解析或 JSON 序列化返回响应

## 四、分布式与系统架构

### 1. 分布式一致性场景（蚂蚁国际真题）

**问题**：用户支付获得积分，支付服务和积分服务是两套异构系统，支付服务通过消息告知积分服务。如何设计才能让支付和积分行为保持一致？

**参考答案**：
1. **可靠消息最终一致性**：
   - 支付服务本地事务 + 本地消息表
   - 定时任务扫描未发送消息并补偿
   - 积分服务消费后 ACK，失败重试
2. **Seata TCC/AT 模式**：
   - TCC：Try（预留）、Confirm（确认）、Cancel（取消）三阶段
   - AT：基于 SQL 解析的自动补偿，对业务侵入小
3. **消息队列事务消息**（如 RocketMQ）：
   - 发送半消息 → 执行本地事务 → 提交/回滚消息
   - MQ 反查生产者事务状态

### 2. 冷热数据分离（邦盛科技真题）

**问题**：一亿条数据如何实现冷热分离？

**参考答案**：
- **按时间维度**：近 3 个月为热数据（Redis / 高性能 MySQL），历史数据归档至低成本存储
- **按访问频率**：通过监控统计访问频次，将低频访问数据迁移至冷存储
- **技术方案**：
  1. 热数据：Redis 缓存 + 主库高性能 SSD
  2. 温数据：从库或归档表
  3. 冷数据：HDFS / OSS / 低成本对象存储，配合压缩算法（Snappy/Gzip）
- **查询路由**：通过中间件或应用层判断查询时间范围，路由到对应存储

### 3. 多设备登录排他（百特云享真题）

**问题**：如何实现"用户多设备登录时，最后一次登录的设备在线，其他设备 token 失效"？

**参考答案**：
- 在 Redis 中以用户 ID 为 Key，当前设备 Token 为 Value 存储
- 用户登录时更新该 Key 的值（覆盖旧 Token）
- 校验请求时对比请求头中的 Token 与 Redis 中的是否一致
- 不一致则拒绝访问（返回 401，提示重新登录）
- 可结合 `Redisson` 或自定义拦截器实现

## 五、数据库与缓存优化

### 1. 数据库与 ES 数据一致性（蚂蚁集团真题）

**问题**：MySQL 数据库与 Elasticsearch 数据一致性如何保证？

**参考答案**：
1. **同步双写**：同时写入 MySQL 和 ES，加分布式锁保证一致性（强一致，性能差）
2. **异步监听 Binlog**（推荐）：
   - 使用 Canal / Debezium 监听 MySQL Binlog
   - 将变更事件发送到 Kafka
   - ES 消费者消费 Kafka 消息，更新索引
   - 保证最终一致性，解耦且高性能
3. **定时全量同步**：周期性任务对比差异并修复

### 2. ThreadLocal 内存泄漏排查（蚂蚁集团真题）

**问题**：ThreadLocal 内存泄漏原理是什么？如何排查？

**参考答案**：
- **泄漏原理**：
  - `ThreadLocalMap` 的 Key 是 `ThreadLocal` 的弱引用，Value 是强引用
  - 当 `ThreadLocal` 变量被回收后，Key 变为 null，但 Value 仍被强引用持有
  - 线程池不销毁时，Value 无法被 GC 回收
- **排查方法**：
  1. 使用 `jmap -histo:live <pid>` 查看对象实例数
  2. 使用 MAT（Memory Analyzer Tool）分析堆转储文件
  3. 搜索 `ThreadLocal.ThreadLocalMap.Entry` 中 key 为 null 的 Entry
- **解决方案**：使用完毕后调用 `ThreadLocal.remove()` 手动清理

### 3. RocketMQ 事务消息（蚂蚁集团真题）

**问题**：RocketMQ 的事务消息是什么原理？

**参考答案**：
1. **半消息**：生产者先发送预处理消息（对消费者不可见）
2. **执行本地事务**：生产者执行本地业务逻辑
3. **提交/回滚**：根据本地事务结果发送 Commit 或 Rollback 指令
4. **反查机制**：若 MQ 超时未收到指令，主动反查生产者事务状态
5. **消费**：消息 Commit 后对消费者可见，正常消费流程

### 4. Redis 锁过期时间与业务执行关系（百特云享真题）

**问题**：使用 Redis 锁时，如何考虑锁的过期时间和业务执行逻辑的关系？

**参考答案**：
- 锁过期时间应略大于业务最大预估执行时间
- 防止业务执行过慢导致锁提前释放，其他线程获取锁引发并发问题
- **推荐方案**：使用「看门狗」机制（如 Redisson 的 WatchDog），在锁即将过期时自动延长有效期
- **底层实现**：后台定时任务检查锁剩余 TTL，若持有锁则续期

## 六、JVM 与性能优化

### 1. Full GC 排查与优化（邦盛科技真题）

**问题**：如何观察和解决项目中的 Full GC？

**参考答案**：
- **监控工具**：
  - `jstat -gcutil <pid> 1000` 实时查看 GC 情况
  - `jmap -heap <pid>` 查看堆内存分布
  - `jmap -dump:format=b,file=heap.hprof <pid>` 导出堆转储
- **常见原因**：
  - 老年代空间不足（大对象、长期存活对象过多）
  - 元空间溢出（类加载过多）
  - 内存泄漏（ThreadLocal、未关闭连接等）
- **优化手段**：
  - 调整堆内存参数（`-Xms`、`-Xmx`、`-XX:MaxMetaspaceSize`）
  - 使用 MAT 分析堆转储文件，定位泄漏对象
  - 优化代码（减少大对象创建、及时释放资源）

### 2. 秒杀系统线程池设计（邦盛科技真题）

**问题**：秒杀系统的线程池参数应如何考量？

**参考答案**：
- 不能照搬固定参数，需结合流量预估动态调整
- **核心参数**：
  - `corePoolSize`：根据 CPU 核心数和 I/O 等待时间计算
  - `maxPoolSize`：应对流量洪峰，但不宜过大（避免 OOM）
  - `workQueue`：建议使用有界队列（如 `ArrayBlockingQueue`），防止任务堆积
- **动态扩缩容**：使用支持动态调整的线程池，通过监控队列积压情况调整参数
- **降级策略**：超过容量时快速失败或排队，避免系统雪崩

## 七、网络与安全

### 1. HTTPS 工作流程与安全性（蚂蚁国际真题）

**问题**：描述 HTTPS 的工作流程。如果数据包在传输中被篡改，HTTPS 如何保证安全性？

**参考答案**：
- **工作流程**：
  1. 客户端发送 ClientHello（支持的 TLS 版本、密码套件）
  2. 服务端返回 ServerHello + 证书（含公钥）
  3. 客户端验证证书（CA 链校验）
  4. 客户端生成预主密钥，用服务端公钥加密后发送
  5. 双方根据预主密钥生成会话密钥
  6. 后续通信使用对称加密
- **防篡改机制**：
  - 对称加密保证数据机密性
  - MAC（消息认证码）或 AEAD 模式保证数据完整性
  - 证书校验防止中间人攻击
  - 若数据被篡改，MAC 校验失败，连接断开

### 2. JWT 登录完整流程（百特云享真题）

**参考答案**：
1. 用户输入用户名密码，发送到后端
2. 后端验证通过后，使用密钥对（HS256/RS256）生成 JWT Token
3. Token 返回客户端，存入 LocalStorage 或 Cookie
4. 后续请求在 Header 中携带 `Authorization: Bearer <token>`
5. 后端拦截器获取 Token，校验签名和有效期
6. 校验通过则放行，否则返回 401

## 八、算法与编程

### 1. 滑动窗口最大值（阿里一面真题）

**问题**：给定一个整数数组和滑动窗口大小 k，返回每个窗口中的最大值。

**参考答案**：
- **单调队列**：使用双端队列维护窗口内的候选最大值索引
- 队列中索引对应的值单调递减
- 每次滑动时：
  1. 移除超出窗口的索引（队首）
  2. 从队尾移除比当前值小的索引
  3. 当前索引入队
  4. 队首即为当前窗口最大值索引
- **时间复杂度**：O(n)，每个元素最多入队出队各一次

### 2. 股票买卖问题（字节跳动真题）

**一次买卖**：遍历数组，记录当前最小价格，计算当前价格与最小价格的差值，更新最大利润。

**两次买卖**：使用状态机定义四个状态（第一次买入、第一次卖出、第二次买入、第二次卖出），通过状态转移方程递推。
