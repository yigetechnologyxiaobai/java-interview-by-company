#### Q1: MVCC 详解？怎么解决的幻读？

**参考答案**：

**MVCC（多版本并发控制）核心组件**：
- **隐藏字段**：`trx_id`（事务ID）、`roll_pointer`（回滚指针）
- **Undo Log**：版本链
- **Read View**：可见性判断

**Read View 结构**：
```c
struct ReadView {
    trx_ids;      // 活跃事务ID列表
    min_trx_id;   // 最小活跃事务ID
    max_trx_id;   // 预分配事务ID（当前最大事务ID+1）
    creator_trx_id; // 创建该Read View的事务ID
}
```

**可见性判断**：
1. `trx_id < min_trx_id`：可见（事务已提交）
2. `trx_id >= max_trx_id`：不可见（后来开启的事务）
3. `trx_id in trx_ids`：不可见（事务未提交）
4. `trx_id == creator_trx_id`：可见（自己修改的）

**幻读解决**：
- **快照读**：MVCC 保证一致性读
- **当前读**：Next-Key Lock（记录锁 + 间隙锁）防止幻读

#### Q2: 间隙锁锁的是什么？索引还是数据？

**参考答案**：

**间隙锁锁的是索引间隙**，不是具体的数据行。

**示例**：
```sql
-- 表中有 id = 1, 5, 10
SELECT * FROM user WHERE id = 3 FOR UPDATE;
-- 锁的是间隙 (1, 5)，不是具体数据
```

**间隙锁作用**：
- 防止其他事务在该间隙插入新记录
- 解决幻读问题

#### Q3: 怎么排查死锁？怎么解决？

**参考答案**：

**排查步骤**：
1. **查看死锁日志**：
   ```sql
   SHOW ENGINE INNODB STATUS;
   ```
2. **分析锁等待**：
   ```sql
   SELECT * FROM performance_schema.data_locks;
   SELECT * FROM performance_schema.data_lock_waits;
   ```
3. **使用工具**：`pt-deadlock-logger`

**解决方案**：
1. **统一加锁顺序**：按主键顺序加锁
2. **降低隔离级别**：RC 无间隙锁
3. **减小事务范围**：减少锁持有时间
4. **使用乐观锁**：版本号代替行锁

#### Q4: Redis 主从的数据同步怎么同步的？

**参考答案**：

**主从同步流程**：

1. **全量同步**：
   - 从节点发送 SYNC 命令
   - 主节点执行 BGSAVE 生成 RDB
   - 主节点发送 RDB 给从节点
   - 从节点加载 RDB

2. **增量同步**：
   - 主节点将写命令写入 repl_backlog_buffer
   - 从节点请求同步偏移量
   - 主节点发送偏移量后的命令

**关键参数**：
- `repl_backlog_size`：环形缓冲区大小
- `repl_timeout`：同步超时时间

#### Q5: RDB 怎么写入的？

**参考答案**：

**RDB 写入流程**：
1. Redis 调用 `fork()` 创建子进程
2. 子进程将内存数据写入临时文件
3. 写入完成后，替换旧的 RDB 文件

**优点**：
- 子进程不影响主进程
- 适合大规模数据备份

**缺点**：
- fork 开销大
- 不是实时备份

#### Q6: Redis 有哪几种数据结构？ZSet 底层怎么实现的？

**参考答案**：

**Redis 数据结构**：

| 类型 | 底层实现 | 应用场景 |
|------|---------|---------|
| String | SDS | 缓存、计数器 |
| List | QuickList | 消息队列 |
| Hash | HashTable | 对象存储 |
| Set | HashTable + IntSet | 标签、去重 |
| ZSet | SkipList + HashTable | 排行榜 |

**ZSet 底层结构**：
- **跳表（SkipList）**：有序，支持范围查询，O(log n)
- **HashTable**：存储 member → score，O(1) 查询

```c
typedef struct zset {
    dict *dict;      // member -> score
    zskiplist *zsl;  // 按 score 排序
} zset;
```

#### Q7: 哨兵机制的作用？

**参考答案**：

**哨兵功能**：
1. **监控**：检测主从节点是否正常
2. **通知**：节点异常时通知管理员
3. **自动故障转移**：主节点故障时选举新主

**选举流程**：
1. 哨兵发现主节点下线（主观下线）
2. 询问其他哨兵（客观下线）
3. 哨兵间选举 Leader
4. Leader 从从节点中选新主

#### Q8: 集群新增一个节点的流程？

**参考答案**：

**集群扩容流程**：
1. 启动新节点
2. 使用 `CLUSTER MEET` 命令加入集群
3. 使用 `CLUSTER REBALANCE` 重新分配槽位
4. 数据迁移过程中服务不中断

**槽位迁移**：
```bash
# 迁移槽位
CLUSTER SETSLOT <slot> IMPORTING <source_node>
CLUSTER SETSLOT <slot> MIGRATING <target_node>
```

#### Q9: RabbitMQ 是推拉哪种模式？

**参考答案**：

**RabbitMQ 推模式**：
- Broker 主动推送消息给消费者
- 消费者被动接收

**拉模式**：
- 消费者主动调用 `basic_get` 获取消息

**默认使用推模式**，效率更高，实时性好。

#### Q10: 延迟消息怎么实现？

**参考答案**：

**RabbitMQ 延迟队列方案**：

1. **TTL + 死信队列**：
   ```java
   // 消息设置 TTL
   AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
       .expiration("60000")  // 60秒后过期
       .build();
   
   // 过期后进入死信队列
   ```

2. **延迟消息插件**：`rabbitmq_delayed_message_exchange`

3. **兼容动态延迟时间**：
   - 创建多个不同 TTL 的队列
   - 或使用消息级别的 TTL

#### Q11: 分布式锁可重入怎么实现的？

**参考答案**：

**Redisson 可重入锁实现**：

使用 Hash 结构存储：
```
lock_key = {
    "thread_id_1": 2,  // 线程1持有，重入2次
    "thread_id_2": 1   // 线程2持有，重入1次
}
```

**加锁 Lua 脚本**：
```lua
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
return redis.call('pttl', KEYS[1]);
```

#### Q14: 类加载器双亲委派机制？

**参考答案**：

**流程**：
```
ClassLoader → 父加载器 → 父加载器 → Bootstrap
             ↓ 委托      ↓ 委托      ↓ 尝试加载
             ↓           ↓           ↓ 失败返回
             ↓           ↓ 尔回
             ↓ 自己加载
```

**优点**：
1. 避免重复加载
2. 保护核心类不被篡改

**打破方式**：
- 自定义 ClassLoader，重写 loadClass 方法
- SPI 机制（线程上下文加载器）

#### Q15: volatile 是干嘛的？怎么实现的？

**参考答案**：

**作用**：
1. **可见性**：一个线程修改，其他线程立即可见
2. **有序性**：禁止指令重排序
3. **不保证原子性**

**实现原理**：
- **MESI 缓存一致性协议**：修改 volatile 变量后，使其他 CPU 缓存失效
- **内存屏障**：LoadLoad、StoreStore、LoadStore、StoreLoad

```java
// 内存屏障示例
volatile boolean ready;
// StoreStore 屏障
ready = true;
// StoreLoad 屏障
```

#### Q2: 线程池场景题

**题目**：有一个发压程序 100/s 调用 A 接口需要 1s，设计一个线程池提升 A 接口的吞吐量。

**参考答案**：

**分析**：
- 每秒 100 个请求
- 每个请求处理时间 1s
- 同时在处理的请求：100 * 1 = 100 个

**线程池配置**：
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    100,                    // 核心线程数
    200,                    // 最大线程数（考虑峰值）
    60, TimeUnit.SECONDS,   // 空闲时间
    new LinkedBlockingQueue<>(1000),  // 队列
    new ThreadPoolExecutor.CallerRunsPolicy()  // 拒绝策略
);
```

**优化建议**：
1. 使用异步非阻塞 IO
2. 减少单次请求耗时
3. 使用队列削峰
