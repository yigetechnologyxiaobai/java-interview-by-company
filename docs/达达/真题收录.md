### Q1: MVCC 详解？怎么解决的幻读？

**参考答案**：

**MVCC（Multi-Version Concurrency Control）**：多版本并发控制，解决读写冲突。

**核心概念**：
- **版本链**：每行数据有多个版本（undo log）
- **Read View**：事务开始时的可见性视图
- **隐藏字段**：`DB_TRX_ID`（事务ID）、`DB_ROLL_PTR`（回滚指针）

**幻读解决**：

| 隔离级别 | 幻读解决方式 |
|----------|------------|
| 读已提交 | MVCC（Read View）|
| 可重复读 | MVCC + Next-Key Lock |

**可重复读下的幻读解决**：
- 当前读：Next-Key Lock（行锁 + 间隙锁）
- 快照读：MVCC（Read View）

**示例**：
```sql
-- 可重复读隔离级别
-- 当前读：SELECT * FROM user WHERE id > 10 FOR UPDATE;
-- 锁住 id > 10 的所有行 + (10, ∞) 的间隙

-- 快照读：SELECT * FROM user WHERE id > 10;
-- 使用 Read View，看不到新插入的数据
```

### Q2: 间隙锁锁的是什么？索引还是数据？

**参考答案**：

**间隙锁锁的是索引间隙**，不是数据本身。

**间隙锁作用**：
- 防止幻读：阻止其他事务在间隙中插入数据
- 只在可重复读隔离级别下生效

**示例**：
```sql
-- 表中有 id = 1, 5, 10
SELECT * FROM user WHERE id > 5 FOR UPDATE;

-- 锁住的间隙：
-- (5, 10)：id 在 5 和 10 之间
-- (10, ∞)：id 大于 10

-- 其他事务无法插入 id = 6, 7, 8, 11, 12...
```

### Q3: 怎么排查死锁？怎么解决？

**参考答案**：

**排查死锁**：
```sql
-- 查看最近死锁日志
SHOW ENGINE INNODB STATUS;

-- 在 LATEST DETECTED DEADLOCK 部分
-- 可以看到：
-- 1. 涉死锁的事务
-- 2. 每个事务持有的锁
-- 3. 每个事务等待的锁
```

**解决方案**：
1. 统一加锁顺序
2. 降低隔离级别（读提交无间隙锁）
3. 减少事务持有时间
4. 使用乐观锁替代悲观锁

### Q4: Redis 主从的数据同步怎么同步的？

**参考答案**：

**主从同步流程**：

1. **全量同步**（首次连接）：
   - Master 执行 bgsave 生成 RDB
   - Master 将 RDB 发送给 Slave
   - Slave 加载 RDB
   - Master 同步期间的新数据通过缓冲区发送

2. **增量同步**（断线重连）：
   - Slave 发送 psync 命令
   - Master 检查 offset 和 runid
   - 如果 offset 在缓冲区范围内，发送增量数据
   - 否则执行全量同步

**关键参数**：
```bash
# 复制缓冲区大小
repl-backlog-size 1mb

# 主从延迟检测
repl-timeout 60
```

### Q5: RDB 怎么写入的？

**参考答案**：

**RDB 写入流程**：

```
BGSAVE 命令
    ↓
子进程 fork()
    ↓
子进程遍历内存，写入临时 RDB 文件
    ↓
写完后原子替换旧 RDB 文件
    ↓
通知主进程完成
```

**特点**：
- fork 子进程，不阻塞主进程
- 写时复制（Copy-On-Write）
- 每次全量备份

### Q7: ZSet 底层怎么实现的？

**参考答案**：

**ZSet 底层结构**：
- **ziplist**：元素数量少时使用（紧凑存储）
- **skiplist + dict**：元素数量多时使用

**跳表特点**：
- 多层索引，查找效率 O(log N)
- 支持范围查询
- 比 AVL 树实现简单

**dict 作用**：
- 存储 member → score 映射
- 快速查找元素是否存在

### Q8: 哨兵机制的作用？

**参考答案**：

**哨兵（Sentinel）功能**：
1. **监控**：检测 Master/Slave 是否正常
2. **通知**：故障时通知管理员或其他应用
3. **自动故障转移**：Master 故障时选举新 Master
4. **配置提供**：提供当前 Master 地址

**选举流程**：
```
Master 故障
    ↓
哨兵投票选举新 Master
    ↓
更新 Slave 配置
    ↓
通知客户端新 Master
```

### Q11: RabbitMQ 是推拉哪种模式？

**参考答案**：

**RabbitMQ 推模式**：
- Broker 主动推送消息到消费者
- 消费者订阅队列后自动接收消息

**推 vs 拉**：

| 模式 | 特点 | 适用场景 |
|------|------|---------|
| 推 | 实时性好，Broker 控制 | 低延迟场景 |
| 拉 | 消费者控制速率 | 流控、批处理 |

**RabbitMQ 默认推模式**，也可通过 `basic_get` 实现拉模式。

### Q12: UACK 导致什么问题？有哪些方案可以解决？

**参考答案**：

**UNACK 问题**：
- 消费者获取消息但未确认
- 消息保持在队列中，占用资源
- 其他消费者无法获取该消息

**解决方案**：
1. 设置消息 TTL
2. 使用死信队列
3. 手动 ACK（`channel.basicAck`）
4. 设置 prefetch count（限制未确认消息数）

```java
// 设置 prefetch count
channel.basicQos(10);  // 每次最多 10 条未确认消息
```

### Q13: 延迟消息怎么实现？如何实现兼容各种动态的延迟时间？

**参考答案**：

**RabbitMQ 延迟消息方案**：

1. **死信队列 + TTL**：
```java
// 创建 TTL 队列
Map<String, Object> args = new HashMap<>();
args.put("x-message-ttl", 5000);  // 5秒过期
args.put("x-dead-letter-exchange", "dlx.exchange");
channel.queueDeclare("delay.queue", true, false, false, args);
```

2. **延迟插件（推荐）**：
```bash
# 安装 rabbitmq_delayed_message_exchange 插件
rabbitmq-plugins enable rabbitmq_delayed_message_exchange

# 使用延迟交换机
Map<String, Object> args = new HashMap<>();
args.put("x-delayed-type", "direct");
channel.exchangeDeclare("delay.exchange", "x-delayed-message", true, false, args);

// 发送延迟消息（动态延迟时间）
AMQP.BasicProperties props = new AMQP.BasicProperties.Builder()
    .headers(Map.of("x-delay", delayTime))  // 动态延迟
    .build();
channel.basicPublish("delay.exchange", "routing.key", props, message.getBytes());
```

### Q14: 分布式锁可重入怎么实现的？

**参考答案**：

**Redisson 可重入锁实现**：

```java
// 使用 hash 结构存储线程ID和计数
// lock_key = {thread_id: count}

// 加锁时：
// 1. 如果 lock_key 不存在，创建并设置 count = 1
// 2. 如果 lock_key 存在且 thread_id 相同，count++

// 解锁时：
// count--
// 如果 count = 0，删除 lock_key

// Lua 脚本实现
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hset', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
return redis.call('pttl', KEYS[1]);
```

### Q16: 熔断了解吗？熔断后的恢复机制？

**参考答案**：

**熔断器状态**：

```
关闭状态 → 正常请求
    ↓ 错误率超过阈值
打开状态 → 直接拒绝请求
    ↓ 半开状态（一段时间后）
半开状态 → 放少量请求测试
    ↓ 成功 → 关闭状态
    ↓ 失败 → 打开状态
```

**Sentinel 实现**：
```java
@SentinelResource(value = "resource", blockHandler = "handleBlock")
public void method() {
    // 业务逻辑
}

public void handleBlock(BlockException ex) {
    // 熔断后的降级处理
}
```

**恢复机制**：
- 半开状态：尝试放少量请求
- 成功后恢复关闭状态
- 失败后重新打开

### Q19: 类加载器、双亲委派机制？

**参考答案**：

**类加载器层次**：
```
Bootstrap ClassLoader  ← 加载核心类
    ↑
Extension ClassLoader  ← 加载扩展类
    ↑
Application ClassLoader ← 加载应用类
    ↑
Custom ClassLoader      ← 自定义加载器
```

**双亲委派**：
- 委托父加载器加载
- 父加载器无法加载，自己加载
- 好处：安全、避免重复加载

### Q20: volatile 是干嘛的？怎么实现的（MESI 缓存一致性）？

**参考答案**：

**volatile 作用**：
1. 可见性：一个线程修改，其他线程立即可见
2. 有序性：禁止指令重排序
3. 不保证原子性

**MESI 协议**：

| 状态 | 说明 |
|------|------|
| M（Modified） | 已修改，独占 |
| E（Exclusive） | 独占，未修改 |
| S（Shared） | 共享，未修改 |
| I（Invalid） | 无效 |

**volatile 写操作**：
1. CPU 将缓存行状态改为 M
2. 其他 CPU 缓存行状态改为 I
3. 写回主内存

**volatile 读操作**：
1. CPU 从主内存读取最新值
2. 缓存行状态改为 S

### Q4: 线程池场景题：100/s 调用 A 接口需要 1s，设计线程池提升吞吐量？

**参考答案**：

**分析**：
- 每秒 100 个请求，每个请求耗时 1s
- 理论上需要 100 个线程并行处理

**设计**：
```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    100,    // 核心线程数 = 100（保证吞吐）
    200,    // 最大线程数 = 200（应对峰值）
    60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(500),  // 有界队列
    new CallerRunsPolicy()           // 降级策略
);
```

**优化点**：
- 核心线程数 = QPS × 平均响应时间
- 队列容量防止 OOM
- CallerRunsPolicy 降级处理

**追问**：
- 如果 A 接口响应时间不稳定怎么办？
  - 增加最大线程数，设置更长的队列
- 如何监控线程池状态？
  - 定时采集 activeCount、queueSize、completedTaskCount
