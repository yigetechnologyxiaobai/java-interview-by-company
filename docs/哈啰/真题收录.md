## 2026 社招一面真题

**来源**：[牛客网](https://www.nowcoder.com/discuss/733999137875595264)
**背景**：两年经验社招
**特点**：线程池 + JVM + OOM + 规则引擎

### Q1: 线程池生产遇到什么问题？怎么解决的？拒绝策略？

**参考答案**：

**常见问题**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 任务堆积 | 任务提交速度 > 处理速度 | 增加线程数、优化任务处理 |
| OOM | 队列无界 | 使用有界队列 |
| 线程泄漏 | 线程未正确释放 | 合理设置 keepAliveTime |
| 任务丢失 | 拒绝策略丢弃 | CallerRunsPolicy |

**拒绝策略选择**：
- `AbortPolicy`：抛异常，适合重要任务
- `CallerRunsPolicy`：调用者执行，降级处理
- `DiscardPolicy`：静默丢弃，适合非核心任务
- `DiscardOldestPolicy`：丢弃最老任务，适合实时任务

### Q2: 生产问题怎么排查？

**参考答案**：

**排查步骤**：
1. 查日志：`grep ERROR application.log`
2. 查线程栈：`jstack pid > thread.txt`
3. 查内存：`jmap -histo pid` 或 `jmap -dump:format=b,file=heap.hprof pid`
4. 查 GC：`jstat -gc pid 1000`
5. 查 CPU：`top -H -p pid`

**常见问题定位**：
```bash
# CPU 高：找最耗 CPU 的线程
top -H -p pid
# 找到线程 id，转换为 16 进制
printf "%x" thread_id
# 在 jstack 输出中搜索该线程

# 内存高：分析堆内存
jmap -histo pid | head 20  # 查对象数量
# 用 MAT 或 VisualVM 分析 dump 文件
```

### Q4: OOM 解决过吗？当时占用多少内存？内存怎么计算？字符占用多少字节？

**参考答案**：

**OOM 类型**：
- `Java heap space`：堆内存不足
- `Metaspace`：类加载过多
- `GC overhead limit exceeded`：GC 效率低

**字符占用字节**：
```java
// Java 字符串内部使用 char[]（JDK 9 后用 byte[]）
// char 类型占 2 字节（UTF-16）

// JDK 8：String = char[] + header
// 一个字符 = 2 字节

// JDK 9+：String = byte[] + coder
// Latin-1（ASCII）：1 字节/字符
// UTF-16：2 字节/字符

String s = "hello";  // JDK 8: 5 * 2 + 12 = 22 字节（不含对象头）
```

**内存计算**：
```java
// 对象内存 = 对象头 + 实例数据 + 对齐填充
// 对象头：12 字节（64位 JVM，开启指针压缩）
// 对齐填充：补齐到 8 字节倍数

// String 对象内存估算
// JDK 8：12(header) + 4(hash) + 4(char[] ref) + char[](12 + length*2 + padding)
```

### Q5: CMS 作用在什么代？方法区溢出会出现什么问题？

**参考答案**：

**CMS 收集区域**：老年代

**方法区溢出（Metaspace OOM）**：
- 原因：加载大量类（动态代理、反射）
- 表现：`java.lang.OutOfMemoryError: Metaspace`

**垃圾回收是否会涉及方法区**：
- JDK 8+：Metaspace 不是堆的一部分，有独立 GC
- 类卸载条件：该类所有实例被回收、加载该类的 ClassLoader 被回收、Class 对象无引用

### Q6: 算法题 - 手撕 LRU

**参考答案**：

```java
public class LRUCache {
    private int capacity;
    private Map<Integer, Node> map;
    private Node head, tail;  // 双向链表
    
    public LRUCache(int capacity) {
        this.capacity = capacity;
        this.map = new HashMap<>();
        head = new Node(0, 0);
        tail = new Node(0, 0);
        head.next = tail;
        tail.prev = head;
    }
    
    public int get(int key) {
        if (!map.containsKey(key)) return -1;
        Node node = map.get(key);
        moveToHead(node);  // 移到头部表示最近使用
        return node.value;
    }
    
    public void put(int key, int value) {
        if (map.containsKey(key)) {
            Node node = map.get(key);
            node.value = value;
            moveToHead(node);
        } else {
            Node node = new Node(key, value);
            map.put(key, node);
            addToHead(node);
            if (map.size() > capacity) {
                Node removed = removeTail();
                map.remove(removed.key);
            }
        }
    }
    
    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }
    
    private void addToHead(Node node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }
    
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }
    
    private Node removeTail() {
        Node node = tail.prev;
        removeNode(node);
        return node;
    }
    
    class Node {
        int key, value;
        Node prev, next;
        Node(int key, int value) {
            this.key = key;
            this.value = value;
        }
    }
}
```

## 2026 社招二面真题

### Q1: 线程池底层：核心线程什么时候创建？什么时候销毁？

**参考答案**：

**核心线程创建时机**：
- **默认**：任务提交时创建（懒加载）
- **预创建**：`executor.prestartAllCoreThreads()` 提前创建所有核心线程

**核心线程销毁时机**：
- **默认**：核心线程永不销毁（`allowCoreThreadTimeOut = false`）
- **开启销毁**：`executor.allowCoreThreadTimeOut(true)`，超过 keepAliveTime 后销毁

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    5, 10, 60, TimeUnit.SECONDS,
    new LinkedBlockingQueue<>(100)
);

// 预创建核心线程
executor.prestartAllCoreThreads();  // 创建 5 个核心线程

// 允许核心线程超时销毁
executor.allowCoreThreadTimeOut(true);  // 核心线程也会超时销毁
```

### Q2: CMS 工作流程？什么时候 STW？

**参考答案**：

**CMS 四个阶段**：

| 阶段 | STW | 说明 |
|------|-----|------|
| 初始标记 | Yes | 标记 GC Roots 直接关联的对象 |
| 并发标记 | No | 遍历整个对象图 |
| 重新标记 | Yes | 修正并发标记期间的变动 |
| 并发清除 | No | 清除未标记的对象 |

**STW 次数**：2 次（初始标记、重新标记）

**追问**：
- 为什么并发标记不 STW？
  - 与用户线程并发执行，效率高
- 重新标记为什么要 STW？
  - 保证标记结果的准确性

### Q5: 策略模式如何对抽象层代码做防腐？为什么不用编排形式？

**参考答案**：

**策略模式防腐**：
```java
// 定义策略接口
public interface PriceStrategy {
    BigDecimal calculate(Order order);
}

// 具体策略实现
public class NormalStrategy implements PriceStrategy { }
public class VIPStrategy implements PriceStrategy { }

// 上下文
public class PriceCalculator {
    private Map<String, PriceStrategy> strategies;
    
    public BigDecimal calculate(String type, Order order) {
        return strategies.get(type).calculate(order);
    }
}
```

**为什么不用编排**：
- 编排复杂度高，维护成本大
- 策略模式更直观，易于测试
- 简单场景用策略模式足够

### Q7: 线程数量标高怎么排查？

**参考答案**：

**排查步骤**：
```bash
# 1. 查看线程数量
jstack pid | grep "java.lang.Thread.State" | wc -l

# 2. 找到大量线程的状态
jstack pid | grep -A 5 "BLOCKED"

# 3. 分析线程堆栈
jstack pid > thread.txt
# 找到 BLOCKED 或 WAITING 状态的线程
# 定位等待的锁对象
```

**常见原因**：
- 线程池配置问题
- 锁竞争激烈
- 阻塞 IO 过多
- 无限创建线程
