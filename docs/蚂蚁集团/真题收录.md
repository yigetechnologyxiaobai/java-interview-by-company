### Q1: Java 和 Python 的区别？

**参考答案**：

| 特性 | Java | Python |
|------|------|--------|
| 类型系统 | 静态类型 | 动态类型 |
| 运行方式 | 编译成字节码，JVM 运行 | 解释执行 |
| 性能 | 较快（JIT 优化） | 较慢 |
| 内存管理 | JVM 自动 GC | 引用计数 + GC |
| 适用场景 | 企业级后端、大数据 | AI/ML、脚本、快速原型 |

**个人理解**：
```java
// Java：强类型，编译期检查，适合大型项目
public class User {
    private String name;
    private int age;
    // IDE 自动补全、重构友好
}
```

```python
# Python：灵活，开发效率高，适合快速验证想法
user = {"name": "test", "age": 20}  # 无需定义类
```

**追问**：
- 为什么 Java 适合企业级开发？
  - 类型安全、生态成熟、稳定性高
- Python 的 GIL 是什么？
  - 全局解释器锁，限制多线程并发

### Q2: 说说线程池，核心线程、最大线程数、阻塞队列等概念

**参考答案**：

**线程池核心参数**：
```java
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数（常驻）
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 非核心线程空闲存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 阻塞队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

**工作流程**：
1. 任务提交，核心线程数未满 → 创建核心线程执行
2. 核心线程已满，队列未满 → 任务入队等待
3. 队列已满，线程数 < 最大线程数 → 创建非核心线程
4. 线程数已达最大 → 执行拒绝策略

**拒绝策略**：
```java
// AbortPolicy（默认）：抛异常
// CallerRunsPolicy：调用者线程执行
// DiscardPolicy：静默丢弃
// DiscardOldestPolicy：丢弃队列最老任务
```

**追问**：
- 如何设置核心线程数？
  - CPU 密集型：核心数 + 1
  - IO 密集型：核心数 * 2
- 队列类型有哪些？
  - ArrayBlockingQueue：有界数组
  - LinkedBlockingQueue：可选有界链表
  - SynchronousQueue：不存储，直接交接

### Q3: JVM 编译加载类文件的流程？

**参考答案**：

**类加载过程**：
```
.java 源文件
    ↓ javac 编译
.class 字节码文件
    ↓ 类加载器
方法区（类信息）
    ↓
堆（Class 对象）
```

**类加载器层次**：
```
Bootstrap ClassLoader  ← 加载核心类（rt.jar）
    ↓
Extension ClassLoader  ← 加载扩展类（ext目录）
    ↓
Application ClassLoader ← 加载应用类（classpath）
    ↓
Custom ClassLoader      ← 自定义加载器
```

**双亲委派模型**：
```java
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 检查是否已加载
    // 2. 委托父加载器加载
    // 3. 父加载器无法加载，自己加载
}
```

**追问**：
- 为什么要双亲委派？
  - 避免核心类被篡改
  - 避免类重复加载
- 如何打破双亲委派？
  - 自定义 ClassLoader，重写 loadClass 方法

### Q1: 断点续传需要计算哪些东西？

**参考答案**：

**核心概念**：
1. **文件分片**：将大文件分成多个小块
2. **分片标识**：每个分片的唯一标识（序号 + MD5）
3. **上传进度**：记录已上传的分片
4. **断点恢复**：从上次中断位置继续

**实现要点**：
```java
public class FileUpload {
    // 1. 计算文件 MD5 作为唯一标识
    public String calculateMD5(File file) {
        MessageDigest md = MessageDigest.getInstance("MD5");
        // 读取文件计算哈希
        return Hex.encodeHexString(md.digest());
    }
    
    // 2. 分片上传
    public void uploadChunk(String fileId, int chunkIndex, byte[] data) {
        // 存储分片到临时目录
        // 记录进度到 Redis
        redis.hset("upload:" + fileId, "chunk:" + chunkIndex, "1");
    }
    
    // 3. 断点续传
    public Set<Integer> getUploadedChunks(String fileId) {
        return redis.hkeys("upload:" + fileId);
    }
    
    // 4. 合并分片
    public void mergeChunks(String fileId, int totalChunks) {
        // 按序号合并所有分片
    }
}
```

**追问**：
- 大文件上传如何优化？
  - 并发上传多个分片
  - 秒传（MD5 匹配已存在文件）
- 分片大小如何选择？
  - 一般 1-5MB，根据网络情况调整

### Q2: 秒传的具体应用场景？

**参考答案**：

**秒传原理**：
- 计算文件的唯一标识（MD5/SHA256）
- 服务器检查是否已存在相同文件
- 如果存在，直接关联，无需上传

**应用场景**：
1. **网盘系统**：百度网盘、阿里云盘
2. **企业文档管理**：重复文档直接关联
3. **图片存储**：社交媒体图片去重
4. **视频平台**：UGC 内容去重

**代码示例**：
```java
public class InstantUpload {
    public UploadResult upload(File file) {
        // 1. 计算文件 MD5
        String md5 = calculateMD5(file);
        
        // 2. 查询是否已存在
        FileMeta existing = fileMapper.selectByMD5(md5);
        if (existing != null) {
            // 3. 秒传：直接关联
            userFileMapper.insert(new UserFile(userId, existing.getId()));
            return UploadResult.instantSuccess();
        }
        
        // 4. 正常上传
        // ...
    }
}
```

### Q3: MD5 能否作为唯一的文件标识符？

**参考答案**：

**MD5 特点**：
- 输出固定 128 位（32 位十六进制）
- 计算速度快
- **存在碰撞风险**（不同文件产生相同 MD5）

**问题**：
1. **碰撞攻击**：可人为构造 MD5 相同的不同文件
2. **安全性不足**：MD5 已被破解

**改进方案**：
```java
// 方案 1：使用更安全的哈希算法
String sha256 = DigestUtils.sha256Hex(fileContent);

// 方案 2：组合多个特征
String identifier = MD5(file) + "_" + file.length();

// 方案 3：使用文件内容采样
String identifier = MD5(header) + MD5(middle) + MD5(footer);

// 方案 4：使用专业算法
String identifier = SimHash(fileContent);  // 相似文件检测
```

**追问**：
- SHA256 和 MD5 的区别？
  - SHA256 更安全，碰撞概率极低
  - MD5 更快，但不安全
- 如何检测相似文件？
  - SimHash、MinHash 等局部敏感哈希

### Q4: MQ 在业务中的使用？

**参考答案**：

**常见场景**：
1. **异步处理**：注册后发送邮件/短信
2. **应用解耦**：订单系统与库存系统
3. **流量削峰**：秒杀场景排队
4. **日志收集**：ELK 栈

**代码示例**：
```java
// 订单服务
public void createOrder(Order order) {
    // 1. 保存订单
    orderMapper.insert(order);
    
    // 2. 发送消息到 MQ
    rabbitTemplate.convertAndSend("order.exchange", "order.created", order);
}

// 库存服务
@RabbitListener(queues = "order.queue")
public void handleOrderCreated(Order order) {
    // 扣减库存
    stockMapper.decrease(order.getProductId(), order.getQuantity());
}
```

**MQ 选型**：

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 吞吐量 | 中 | 高 | 高 |
| 延迟 | 低 | 中 | 低 |
| 顺序消费 | 不保证 | 分区内保证 | 保证 |
| 事务消息 | 支持 | 支持 | 支持 |
| 适用场景 | 传统业务 | 日志/大数据 | 金融/电商 |

### Q5: 弹幕推送如何设计？

**参考答案**：

**场景分析**：
- 视频播放时实时推送弹幕
- 历史弹幕和实时弹幕混合展示
- 保证用户体验（不能全被一个人的弹幕覆盖）

**设计方案**：

```java
public class DanmakuService {
    // 1. 历史弹幕加载
    public List<Danmaku> getHistoryDanmaku(Long videoId, int currentTime) {
        // 查询该时间点前后的弹幕
        return danmakuMapper.selectByVideoAndTime(videoId, currentTime - 10, currentTime + 10);
    }
    
    // 2. 实时弹幕推送（WebSocket）
    @OnMessage
    public void onDanmaku(String message) {
        Danmaku danmaku = parseMessage(message);
        
        // 广播给所有观看该视频的用户
        broadcast(danmaku.getVideoId(), danmaku);
    }
    
    // 3. 弹幕优先级
    public int calculatePriority(Danmaku danmaku) {
        int score = 0;
        // 用户等级权重
        score += danmaku.getUserLevel() * 10;
        // 时间衰减（越新的弹幕优先级越高）
        score -= (System.currentTimeMillis() - danmaku.getCreateTime()) / 1000;
        // 发送频率限制（防止刷屏）
        if (getUserDanmakuCount(danmaku.getUserId(), 60) > 10) {
            score -= 100;
        }
        return score;
    }
}
```

**推送顺序策略**：
1. **时间权重**：新弹幕优先，但历史弹幕也要展示
2. **用户优先级**：VIP 用户、高等级用户优先
3. **频率限制**：同一用户短时间内限制发送数量
4. **去重**：相同内容弹幕去重

### Q6: CMS 和 G1 的区别？

**参考答案**：

| 特性 | CMS | G1 |
|------|-----|-----|
| 分代 | 分代收集 | Region 分区 |
| 算法 | 标记-清除 | 标记-整理 |
| 停顿时间 | 不可预测 | 可预测 |
| 碎片问题 | 有 | 无 |
| 适用场景 | 老年代收集 | 大堆内存 |

**CMS 工作流程**：
1. 初始标记（STW）
2. 并发标记
3. 重新标记（STW）
4. 并发清除

**G1 工作流程**：
1. 初始标记（STW）
2. 并发标记
3. 最终标记（STW）
4. 筛选回收（STW）
