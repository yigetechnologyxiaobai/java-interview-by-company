### Q2: 说说线程池，核心线程、最大线程数、阻塞队列等概念

**参考答案**：

**线程池核心参数**：
```java
public ThreadPoolExecutor(
    int corePoolSize,      // 核心线程数（常驻）
    int maximumPoolSize,   // 最大线程数
    long keepAliveTime,    // 非核心线程空闲存活时间
    TimeUnit unit,         // 时间单位
    BlockingQueue<Runnable> workQueue,  // 阻塞队列
    ThreadFactory threadFactory,         // 线程工厂
    RejectedExecutionHandler handler     // 拒绝策略
)
```

**工作流程**：
1. 任务提交，核心线程数未满 → 创建核心线程执行
2. 核心线程已满，队列未满 → 任务入队等待
3. 队列已满，线程数 < 最大线程数 → 创建非核心线程
4. 线程数已达最大 → 执行拒绝策略

**拒绝策略**：
```java
// AbortPolicy（默认）：抛异常
// CallerRunsPolicy：调用者线程执行
// DiscardPolicy：静默丢弃
// DiscardOldestPolicy：丢弃队列最老任务
```

**追问**：
- 如何设置核心线程数？
  - CPU 密集型：核心数 + 1
  - IO 密集型：核心数 * 2
- 队列类型有哪些？
  - ArrayBlockingQueue：有界数组
  - LinkedBlockingQueue：可选有界链表
  - SynchronousQueue：不存储，直接交接

### Q3: JVM 编译加载类文件的流程？

**参考答案**：

**类加载过程**：
```
.java 源文件
    ↓ javac 编译
.class 字节码文件
    ↓ 类加载器
方法区（类信息）
    ↓
堆（Class 对象）
```

**类加载器层次**：
```
Bootstrap ClassLoader  ← 加载核心类（rt.jar）
    ↓
Extension ClassLoader  ← 加载扩展类（ext目录）
    ↓
Application ClassLoader ← 加载应用类（classpath）
    ↓
Custom ClassLoader      ← 自定义加载器
```

**双亲委派模型**：
```java
protected Class<?> loadClass(String name, boolean resolve) {
    // 1. 检查是否已加载
    // 2. 委托父加载器加载
    // 3. 父加载器无法加载，自己加载
}
```

**追问**：
- 为什么要双亲委派？
  - 避免核心类被篡改
  - 避免类重复加载
- 如何打破双亲委派？
  - 自定义 ClassLoader，重写 loadClass 方法

```java
        return redis.hkeys("upload:" + fileId);
    }
    
    // 4. 合并分片

        // 按序号合并所有分片
    }
}
```

### Q2: 秒传的具体应用场景？

**参考答案**：

**秒传原理**：
- 计算文件的唯一标识（MD5/SHA256）
- 服务器检查是否已存在相同文件
- 如果存在，直接关联，无需上传

**应用场景**：
1. **网盘系统**：百度网盘、阿里云盘
2. **企业文档管理**：重复文档直接关联
3. **图片存储**：社交媒体图片去重
4. **视频平台**：UGC 内容去重

**代码示例**：
```java
public class InstantUpload {
    public UploadResult upload(File file) {
        // 1. 计算文件 MD5
        String md5 = calculateMD5(file);
        
        // 2. 查询是否已存在
        FileMeta existing = fileMapper.selectByMD5(md5);
        if (existing != null) {
            // 3. 秒传：直接关联
            userFileMapper.insert(new UserFile(userId, existing.getId()));
            return UploadResult.instantSuccess();
        }
        
        // 4. 正常上传
        // ...
    }
}
```

### Q3: MD5 能否作为唯一的文件标识符？

**参考答案**：

**MD5 特点**：
- 输出固定 128 位（32 位十六进制）
- 计算速度快
- **存在碰撞风险**（不同文件产生相同 MD5）

**问题**：
1. **碰撞攻击**：可人为构造 MD5 相同的不同文件
2. **安全性不足**：MD5 已被破解

**改进方案**：
```java
// 方案 1：使用更安全的哈希算法
String sha256 = DigestUtils.sha256Hex(fileContent);

// 方案 2：组合多个特征
String identifier = MD5(file) + "_" + file.length();

// 方案 3：使用文件内容采样
String identifier = MD5(header) + MD5(middle) + MD5(footer);

// 方案 4：使用专业算法
String identifier = SimHash(fileContent);  // 相似文件检测
```

**追问**：
- SHA256 和 MD5 的区别？
  - SHA256 更安全，碰撞概率极低
  - MD5 更快，但不安全
- 如何检测相似文件？
  - SimHash、MinHash 等局部敏感哈希

### Q5: 弹幕推送如何设计？

**参考答案**：

**场景分析**：
- 视频播放时实时推送弹幕
- 历史弹幕和实时弹幕混合展示
- 保证用户体验（不能全被一个人的弹幕覆盖）

**设计方案**：

```java
public class DanmakuService {
    // 1. 历史弹幕加载
    public List<Danmaku> getHistoryDanmaku(Long videoId, int currentTime) {
        // 查询该时间点前后的弹幕
        return danmakuMapper.selectByVideoAndTime(videoId, currentTime - 10, currentTime + 10);
    }
    
    // 2. 实时弹幕推送（WebSocket）
    @OnMessage
    public void onDanmaku(String message) {
        Danmaku danmaku = parseMessage(message);
        
        // 广播给所有观看该视频的用户
        broadcast(danmaku.getVideoId(), danmaku);
    }
    
    // 3. 弹幕优先级
    public int calculatePriority(Danmaku danmaku) {
        int score = 0;
        // 用户等级权重
        score += danmaku.getUserLevel() * 10;
        // 时间衰减（越新的弹幕优先级越高）
        score -= (System.currentTimeMillis() - danmaku.getCreateTime()) / 1000;
        // 发送频率限制（防止刷屏）
        if (getUserDanmakuCount(danmaku.getUserId(), 60) > 10) {
            score -= 100;
        }
        return score;
    }
}
```

**推送顺序策略**：
1. **时间权重**：新弹幕优先，但历史弹幕也要展示
2. **用户优先级**：VIP 用户、高等级用户优先
3. **频率限制**：同一用户短时间内限制发送数量
4. **去重**：相同内容弹幕去重

#### Q1: WebSocket 发一条阻塞了，后面的消息会怎么样？

**参考答案**：

**WebSocket 阻塞分析**：
- WebSocket 是全双工通信，基于 TCP
- 一条消息阻塞不会影响其他消息的发送
- 但会影响该连接的消息顺序

**原因分析**：
1. TCP 层阻塞：发送缓冲区满
2. 应用层阻塞：处理消息的业务逻辑慢

**解决方案**：
```java
// 1. 异步处理消息
@OnMessage
public void onMessage(String message) {
    CompletableFuture.runAsync(() -> {
        processMessage(message);
    });
}

// 2. 消息队列解耦
@OnMessage
public void onMessage(String message) {
    messageQueue.offer(message);
}

// 3. 设置发送超时
session.getAsyncRemote().setSendTimeout(5000);
```

#### Q3: Kafka 副本了解吗？聊聊 ISR？

**参考答案**：

**Kafka 副本机制**：
- 每个 Partition 有多个副本（Leader + Follower）
- Leader 负责读写，Follower 负责同步

**ISR（In-Sync Replicas）**：
- 与 Leader 保持同步的副本集合
- ISR 中的副本可以成为新 Leader
- 默认配置：`min.insync.replicas = 1`

**ISR 机制**：
```
Leader 副本
    ↓ 数据同步
ISR 副本（同步中）
    ↓ 超时（replica.lag.time.max.ms）
OSR 副本（落后副本）
```

**生产者配置**：
```java
// acks=all：ISR 所有副本确认
props.put("acks", "all");
props.put("min.insync.replicas", "2");
```

#### Q4: 线程池场景题：5核心、10最大、20队列

**题目**：线程池配置为 5核心、10最大、20队列容量。问：
1. 第 6 个任务来了是什么状态？
2. 第 26 个任务来了是什么状态？
3. 队列满了执行队列的任务是从队列头还是队尾取？
4. 核心线程和非核心线程执行结束后，谁先执行队列里的任务？

**参考答案**：

**状态分析**：

| 任务数 | 状态 |
|--------|------|
| 1-5 | 创建核心线程执行 |
| 6-25 | 入队等待（队列容量 20，核心线程 5，共 25 个位置） |
| 26-30 | 创建非核心线程执行 |
| > 30 | 执行拒绝策略 |

**详细答案**：
1. **第 6 个任务**：入队等待（核心线程已满，队列未满）
2. **第 26 个任务**：创建非核心线程执行（队列已满，线程数 5 < 最大 10）
3. **队列取任务**：从队列头取（FIFO，先进先出）
4. **谁先执行队列任务**：都有可能，取决于谁先完成任务

#### Q5: 联合索引 ABC，执行语句 A = XXX and C < XXX，索引怎么走？

**参考答案**：

**最左前缀原则**：
- 联合索引 (A, B, C)
- 查询条件必须从左连续匹配

**分析**：
```sql
-- A = XXX and C < XXX
-- 索引：只走 A（部分匹配）
-- B 缺失，无法继续匹配到 C
```

**执行过程**：
1. 用 A 定位到范围
2. 在该范围内扫描所有记录
3. 过滤满足 C < XXX 的记录

**索引使用情况**：
```
key_len = A 的长度（只用到了 A）
Extra = Using index condition（索引下推）
```

#### Q3: 堆内存结构？

**参考答案**：

```
┌─────────────────────────────────────────────────────────┐
│                      JVM 堆内存                          │
├─────────────────────────────────────────────────────────┤
│  新生代（Young Generation）                              │
│  ├── Eden（8/10）                                       │
│  ├── Survivor0（1/10）                                  │
│  ├── Survivor1（1/10）                                  │
│  ├── 特点：GC频繁，对象生命周期短                        │
│  └── 收集器：Serial、ParNew、Parallel Scavenge          │
├─────────────────────────────────────────────────────────┤
│  老年代（Old Generation）                               │
```

```java
public List<List<Integer>> levelOrder(TreeNode root) {
    List<List<Integer>> result = new ArrayList<>();
    if (root == null) return result;
    
    Queue<TreeNode> queue = new LinkedList<>();
    queue.offer(root);
    
    while (!queue.isEmpty()) {
        int size = queue.size();
        List<Integer> level = new ArrayList<>();
        for (int i = 0; i < size; i++) {
            TreeNode node = queue.poll();
            level.add(node.val);
            if (node.left != null) queue.offer(node.left);
            if (node.right != null) queue.offer(node.right);
        }
        result.add(level);
    }
    
    return result;
}
```

```sql
UPDATE user SET age = 25 WHERE id = 1;  -- 阻塞！
```

**分析**：
1. 事务A获取排他锁（X锁）
2. 事务B尝试获取X锁，被阻塞
3. 事务A提交后，事务B继续执行

**锁冲突**：
| 事务A锁 | 事务B请求 | 结果 |
|---------|----------|------|
| X锁 | X锁 | 阻塞 |
| X锁 | S锁 | 阻塞 |
| S锁 | X锁 | 阻塞 |
| S锁 | S锁 | 允许 |

```java
        processMessage(message);
    });

// 2. 消息队列解耦
@OnMessage

    messageQueue.offer(message);
}

// 3. 设置发送超时
session.getAsyncRemote().setSendTimeout(5000);
```

### Q3: Kafka 副本了解吗？聊聊 ISR？

**参考答案**：

**Kafka 副本机制**：
- 每个 Partition 有多个副本（Leader + Follower）
- Leader 负责读写，Follower 负责同步

**ISR（In-Sync Replicas）**：
- 与 Leader 保持同步的副本集合
- ISR 中的副本可以成为新 Leader
- 默认配置：`min.insync.replicas = 1`

**ISR 机制**：
```
Leader 副本
    ↓ 数据同步
ISR 副本（同步中）
    ↓ 超时（replica.lag.time.max.ms）
OSR 副本（落后副本）
```

**生产者配置**：
```java
// acks=all：ISR 所有副本确认
props.put("acks", "all");
props.put("min.insync.replicas", "2");
```

#### Q7: 联合索引原理和注意事项？

**最左前缀原则**：

|----------|---------|

| B = ? | 不用 |
| A = ? AND C = ? | 只用 A（B缺失） |
| A = ? AND B > ? AND C = ? | 用 A, B（C不用） |
