### Q1: JVM 三色标记算法是什么？

**参考答案**：

**三色标记算法**是 CMS 和 G1 垃圾收集器使用的并发标记算法。

**三种颜色**：
- **白色**：未被访问的对象（可能被回收）
- **灰色**：被访问但子对象未全部访问
- **黑色**：被访问且子对象全部访问（不会被回收）

**标记过程**：
```
初始：所有对象白色
  ↓
GC Roots → 灰色
  ↓
遍历灰色对象的子对象 → 子对象灰色，当前对象黑色
  ↓
灰色对象消失 → 标记完成
```

**多标问题（浮动垃圾）**：
- 并发标记时，用户线程新增引用
- 黑色对象指向白色对象
- 白色对象不会被标记，但实际存活
- 解决：下次 GC 再回收（浮动垃圾）

**漏标问题（对象消失）**：
- 灰色对象指向白色对象的引用被删除
- 同时黑色对象指向白色对象
- 白色对象被错误回收！

**解决方案（SATB 和增量更新）**：
```java
// CMS：增量更新（Incremental Update）
// 记录新增引用，重新扫描灰色对象

// G1：SATB（Snapshot-At-The-Beginning）
// 在标记开始时拍摄快照，引用变更记录到 SATB Buffer
```

**追问**：
- 为什么 G1 用 SATB 而不是增量更新？
  - SATB 效率更高，不需要重新扫描
- 三色标记如何保证正确性？
  - 强/弱三色不变性原则

### Q2: ZK（ZooKeeper）的角色有哪些？如何理解强一致性？

**参考答案**：

**ZK 角色**：

| 角色 | 作用 |
|------|------|
| Leader | 处理写请求，协调 Follower |
| Follower | 处理读请求，参与选举投票 |
| Observer | 处理读请求，不参与投票（提升读性能） |

**强一致性理解**：

ZK 保证**最终一致性**而非严格的强一致性：

1. **写请求**：必须经过 Leader，多数节点确认后返回
2. **读请求**：可以从任意 Follower/Observer 读取（可能读到旧数据）
3. **同步机制**：Leader 将变更同步给所有 Follower

**ZAB 协议**：
- **崩溃恢复**：Leader 崩溃后重新选举
- **消息广播**：Leader 广播变更给 Follower

**代码示例**：
```java
// ZK 客户端使用
ZooKeeper zk = new ZooKeeper("localhost:2181", 3000, watcher);

// 创建节点（写请求）
zk.create("/path", data, ZooDefs.Ids.OPEN_ACL_UNSAFE, CreateMode.PERSISTENT);

// 读取数据（读请求，可能读到旧数据）
byte[] data = zk.getData("/path", false, null);

// 强一致性读：sync 后再读
zk.sync("/path", null, null);  // 同步到最新状态
byte[] freshData = zk.getData("/path", false, null);
```

**追问**：
- ZK 为什么不用强一致性？
  - 性能考虑，强一致需要所有节点确认
- ZK 如何保证写请求顺序？
  - 全局 ZXID（事务ID）单调递增

### Q3: 为什么不用原生 Redis 实现分布式锁？

**参考答案**：

**原生 Redis 分布式锁的问题**：

| 问题 | 说明 |
|------|------|
| **锁误删** | A 的锁过期，B 获取锁，A 释放了 B 的锁 |
| **不可重入** | 同一线程无法多次获取同一锁 |
| **无等待机制** | 获取失败直接返回，无阻塞等待 |
| **无看门狗** | 业务执行超过过期时间，锁被自动释放 |

**Redisson 解决方案**：

```java
// 1. 锁误删：Lua 脚本校验
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
end

// 2. 可重入：hash 结构存储线程ID和计数
// lock_key = {thread_id: count}

// 3. 等待机制：pub/sub 订锁释放通知

// 4. 看门狗：后台线程自动续期
```

**看门狗续签机制**：
```java
// Redisson 看门狗原理
// 1. 锁默认过期时间 30s
// 2. 后台线程每隔 10s（过期时间/3）检查
// 3. 如果锁还被当前线程持有，续期 30s
// 4. 业务执行完毕，主动释放锁，看门狗停止

RLock lock = redisson.getLock("myLock");
lock.lock();  // 启动看门狗
try {
    // 业务逻辑（可能超过 30s）
} finally {
    lock.unlock();  // 停止看门狗
}
```

**追问**：
- 看门狗续签失败怎么办？
  - 锁过期，其他线程可获取，需要业务层处理并发
- Redis 集群模式下分布式锁如何保证？
  - RedLock 算法：多数节点获取成功才算成功

### Q4: 间隙锁导致的死锁如何修复？

**参考答案**：

**间隙锁死锁场景**：

```sql
-- 表中有 id = 1, 5, 10

-- 事务 A
SELECT * FROM user WHERE id = 3 FOR UPDATE;  -- 加间隙锁 (1, 5)

-- 事务 B
SELECT * FROM user WHERE id = 7 FOR UPDATE;  -- 加间隙锁 (5, 10)
INSERT INTO user (id, name) VALUES (4, 'test');  -- 等待 A 的间隙锁

-- 事务 A
INSERT INTO user (id, name) VALUES (6, 'test');  -- 等待 B 的间隙锁
-- 死锁！
```

**解决方案**：

1. **避免间隙锁**：
```sql
-- 使用唯一索引查询存在的记录
SELECT * FROM user WHERE id = 5 FOR UPDATE;  -- 只加行锁
```

2. **降低隔离级别**：
```sql
-- 读提交隔离级别无间隙锁
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

3. **加锁顺序一致**：
```java
// 统一按 id 升序加锁
public void batchInsert(List<User> users) {
    users.sort(Comparator.comparing(User::getId));
    for (User user : users) {
        insert(user);
    }
}
```

4. **死锁检测**：
```sql
-- 查看 InnoDB 死锁检测设置
SHOW VARIABLES LIKE 'innodb_deadlock_detect';

-- 查看最近死锁日志
SHOW ENGINE INNODB STATUS;
```

**追问**：
- 如何排查死锁？
  - SHOW ENGINE INNODB STATUS 查看 LATEST DETECTED DEADLOCK
- 死锁发生后 MySQL 如何处理？
  - 自动检测，回滚代价最小的事务

#### Q1: select 和 epoll 的区别？

**参考答案**：

| 特性 | select | epoll |
|------|--------|-------|
| 文件描述符限制 | 1024（FD_SETSIZE） | 无限制 |
| 效率 | O(n) 每次遍历 | O(1) 事件通知 |
| 工作模式 | 水平触发 | 边缘触发 + 水平触发 |
| 内存拷贝 | 每次调用拷贝 | 共享内存（mmap） |

**select 原理**：
```c
// 每次调用需要把 fd_set 从用户态拷贝到内核态
int select(int nfds, fd_set *readfds, fd_set *writefds, fd_set *exceptfds, struct timeval *timeout);
// 问题：每次调用都要遍历所有 fd，效率低
```

**epoll 原理**：
```c
// 三个核心 API
int epoll_create(int size);          // 创建 epoll 实例
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);  // 注册事件
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);  // 等待事件

// 只返回活跃的 fd，不需要遍历所有
```

**追问**：
- epoll 的边缘触发和水平触发有什么区别？
  - LT：事件未处理完会持续通知
  - ET：只通知一次，必须处理完所有数据
- Redis 为什么用 epoll？
  - 高并发连接、单线程处理、效率高

#### Q2: ZK 在 Kafka 中的应用？

**参考答案**：

**Kafka 使用 ZK 的场景**：

1. **Broker 注册**：
   - 每个 Broker 启动时在 ZK 注册临时节点
   - `/brokers/ids/[broker_id]`
   - Broker 下线时节点自动删除

2. **Topic 注册**：
   - Topic 的分区信息存储在 ZK
   - `/brokers/topics/[topic]/partitions/[partition]/state`

#### Q4: 多线程赛马题：10 个线程模拟赛马，所有马就绪后才能开跑，所有马到达终点后裁判宣布赛马成绩

**参考答案**：

**问题分析**：
- 需要两个同步点：起跑门、终点统计
- 使用 `CyclicBarrier` 实现

**代码实现**：
```java
public class HorseRace {
    private static final int HORSE_COUNT = 10;
    
    // 起跑门：所有马就绪后开跑
    private CyclicBarrier startGate = new CyclicBarrier(HORSE_COUNT, () -> {
        System.out.println("所有马就绪，比赛开始！");
    });
    
    // 终点门：所有马到达后公布成绩
    private CyclicBarrier finishGate = new CyclicBarrier(HORSE_COUNT, () -> {
        System.out.println("所有马到达终点，公布成绩！");
        announceResults();
    });
    
    // 存储成绩
    private Map<String, Long> results = new ConcurrentHashMap<>();
    
    public void race(String horseName) {
        new Thread(() -> {
            try {
                // 1. 等待所有马就绪
                System.out.println(horseName + " 已就绪");
                startGate.await();
                
                // 2. 开始跑（模拟随机时间）
                long startTime = System.currentTimeMillis();
                int runTime = ThreadLocalRandom.current().nextInt(1000, 5000);
                Thread.sleep(runTime);
                
                // 3. 记录成绩
                results.put(horseName, System.currentTimeMillis() - startTime);
                System.out.println(horseName + " 到达终点");
                
                // 4. 等待所有马到达
                finishGate.await();
            } catch (Exception e) {
                e.printStackTrace();
            }
        }).start();
    }
    
    private void announceResults() {
        results.entrySet().stream()
            .sorted(Map.Entry.comparingByValue())
            .forEach(e -> System.out.println(e.getKey() + ": " + e.getValue() + "ms"));
    }
}
```

**追问**：
- CyclicBarrier 和 CountDownLatch 的区别？
  - CyclicBarrier：可重用，线程互相等待
  - CountDownLatch：一次性，一个/多个线程等待其他线程
- 如果马中途退出怎么办？
  - 使用 `await(timeout)` 超时机制

#### Q2: select 和 epoll 的区别？

**参考答案**：

#### Q5: 多线程场景题 - 10个线程模拟赛马？

**参考答案**：

**问题要求**：
- 10个线程模拟赛马
- 所有马就绪后才能开跑
- 所有马到达终点后裁判宣布成绩

**解法**：
```java
public class HorseRace {
    private static final int HORSE_COUNT = 10;
    private static CyclicBarrier readyBarrier = new CyclicBarrier(HORSE_COUNT + 1); // +1裁判
    private static CyclicBarrier finishBarrier = new CyclicBarrier(HORSE_COUNT + 1);
    
    public static void main(String[] args) {
        // 裁判线程
        new Thread(() -> {
            try {
                readyBarrier.await(); // 等待所有马就绪
                System.out.println("比赛开始！");
                finishBarrier.await(); // 等待所有马到达终点
                System.out.println("比赛结束，公布成绩！");
            } catch (Exception e) {}
        }).start();
        
        // 10匹马
        for (int i = 1; i <= HORSE_COUNT; i++) {
            new Thread(() -> {
                try {
                    System.out.println("马" + i + "就绪");
                    readyBarrier.await(); // 等待开跑
                    // 跑步...
                    finishBarrier.await(); // 到达终点
                } catch (Exception e) {}
            }).start();
        }
    }
}
```

**追问**：
- CyclicBarrier 和 CountDownLatch 区别？
  - CyclicBarrier 可循环使用， parties 达到后一起执行
  - CountDownLatch 一次性，计数减到0后等待线程继续

## 2026-04-11 阿里算法岗笔试真题

### 一、字符映射（轮转）

**题目描述**：给定长度为 n 的字符串 s，进行 q 次操作。每次操作给出字符 x 和 y，将字符串里所有字符 x 替换为字符 y。输出完成所有操作后的字符串。

**考察点**：数据结构优化、时间复杂度分析

**思路分析**：
- 数据规模 n, q 可达 10^5，暴力扫描 O(nq) 会超时
- 关键：维护 26 个字母的映射关系，每次操作仅需 O(26)
- 不逐次扫描字符串，而是维护映射表 mp[c] 表示字符 c 最终变成什么

**参考代码**：
```python
def solve(s, ops):
    mp = list(range(26))  # mp[c] 记录字符 c 最终会变成什么
    for x, y in ops:
        for c in range(26):
            if mp[c] == x:
                mp[c] = y
    return ''.join(chr(mp[ord(ch) - 97] + 97) for ch in s)
```

**复杂度**：时间 O(26q + n)，空间 O(n)

### 二、凑对（构造题）

**题目描述**：构造一个 1 到 n 的排列（n 为偶数），使得每对相邻位置 (a[2i-1], a[2i]) 的和与差的绝对值都不是质数。若不存在合法排列，输出 -1。

**考察点**：数学观察、构造思维

**思路分析**：
- **关键发现**：两个同奇偶的数，和与差都是偶数（大于 2 的偶数不是质数）
- **策略**：将 1 到 n 按奇偶分成两组，组内对半配对
- 边界情况：n ≤ 6 时无解；n % 4 == 2 时需特殊处理跨奇偶对 (4,5)

**复杂度**：时间 O(n)，空间 O(n)

### 三、模 k 最大子序列（DP 位集优化）

**题目描述**：给定整数数组 a 和正整数 k，选择一个非空子序列，使元素之和对 k 取模的结果最大。

**考察点**：动态规划、位运算优化

**思路分析**：
- 本质是模 k 子集和问题
- 用位集 f 表示可达余数集合，第 j 位为 1 表示余数 j 可达
- 加入元素余数 v 等价于循环左移 v 位再取并集
- 位集优化将复杂度从 O(nk) 降到 O(nk/w)

**参考代码**：
```python
def solve(n, k, a):
    f = 0  # 位集表示可达余数
    mask = (1 << k) - 1
    for x in a:
        v = x % k
        shifted = ((f << v) | (f >> (k - v))) & mask
        shifted |= (1 << v)
        f |= shifted
    for j in range(k - 1, -1, -1):
        if f & (1 << j):
            return j
    return 0
```

**复杂度**：时间 O(nk/w)，空间 O(k/w)
