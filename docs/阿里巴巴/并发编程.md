### Q5: synchronized 底层原理？

**核心概念**：

synchronized 是 Java 内置的互斥锁，基于 Monitor 实现。

**三种使用方式**：

```java
public class SynchronizedDemo {
    private final Object lock = new Object();
    
    // 1. 同步代码块（锁指定对象）
    public void block() {
        synchronized (lock) {
            // 临界区
        }
    }
    
    // 2. 同步实例方法（锁 this）
    public synchronized void instanceMethod() {
        // 临界区
    }
    
    // 3. 同步静态方法（锁 Class 对象）
    public static synchronized void staticMethod() {
        // 临界区
    }
}
```

**底层原理**：

**同步代码块**：`monitorenter` / `monitorexit` 字节码指令

```java
// 源码
public void block();
    synchronized (lock) {
        // ...
    }

// 字节码
Code:
   0: aload_0
   1: getfield      #3  // Field lock:Ljava/lang/Object;
   4: dup
   5: astore_1
   6: monitorenter      // 获取 monitor
       // ... 临界区代码
  10: monitorexit       // 释放 monitor
```

**同步方法**：`ACC_SYNCHRONIZED` 标志

```java
// 方法 flags 中有 ACC_SYNCHRONIZED
// JVM 自动在方法入口获取 monitor，出口释放 monitor
public synchronized void method();
```

**Monitor 原理**：

```

│  Entry List: 等待获取锁的线程队列                               │

```

**锁升级过程**：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

升级方向：不可逆（只能升级，不能降级）
```

**面试加分点**：
> synchronized 和 ReentrantLock 区别：

| 维度 | synchronized | ReentrantLock |
|------|--------------|---------------|
| 实现层次 | JVM 关键字 | Java API |
| 锁释放 | 自动 | 手动 unlock() |
| 锁类型 | 非公平 | 可选公平/非公平 |
| 中断 | 不支持 | 支持 lockInterruptibly() |
| 超时 | 不支持 | 支持 tryLock(timeout) |
| 条件变量 | 单一（wait/notify） | 多个 Condition |
| 自旋 | 支持（自适应） | 不支持 |
| 性能 | JDK 6 后差距小 | 高竞争时略优 |

**追问延伸**：
- 锁升级为什么不可逆？
  - 性能考虑，降级需要额外开销
  - 实际场景中升级后竞争往往更激烈

- 偏向锁为什么有延迟？
  - JVM 启动时竞争多，偏向锁意义不大
  - 默认延迟 4 秒后启用

### Q6: ThreadLocal 原理和内存泄漏？

**核心概念**：

ThreadLocal 提供线程局部变量，每个线程都有独立的变量副本。

**原理图**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      ThreadLocal 原理                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Thread                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ThreadLocalMap                                         │   │
│  │  ┌──────────────────────────────────────────────────┐  │   │
│  │  │  Entry (WeakReference<ThreadLocal>)              │  │   │
│  │  │  ┌─────────────────┐  ┌───────────────────────┐  │  │   │
│  │  │  │ key (弱引用)     │  │ value (强引用)        │  │  │   │
│  │  │  │ ThreadLocal     │  │ 实际存储的值           │  │  │   │
│  │  │  └─────────────────┘  └───────────────────────┘  │  │   │
│  │  └──────────────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
public class ThreadLocalDemo {
    // 每个线程独立的用户上下文
    private static final ThreadLocal<User> userContext = new ThreadLocal<>();
    
    public void process() {
        try {
            userContext.set(new User("张三"));
            // 业务逻辑
            doSomething();
        } finally {
            userContext.remove();  // 防止内存泄漏
        }
    }
    
    public void doSomething() {
        User user = userContext.get();  // 获取当前线程的用户
    }
}
```

**内存泄漏原因**：

```
1. key 是弱引用，GC 后变为 null
2. value 是强引用，仍被 Entry 引用
3. 线程不结束，value 无法回收

┌─────────────────────────────────────────────────────────────────┐
│  ThreadLocalMap                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Entry                                                    │  │
│  │  ┌─────────────┐  ┌───────────────────────────────────┐  │  │
│  │  │ key = null  │  │ value = Object (无法访问也无法回收) │  │  │
│  │  │ (被 GC)     │  │                                   │  │  │
│  │  └─────────────┘  └───────────────────────────────────┘  │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**解决方案**：
1. **及时 remove()**：在 finally 块中清理
2. **使用 try-finally**：确保一定清理

**面试加分点**：
> key 为什么用弱引用？
> - 如果用强引用，ThreadLocal 对象无法被回收
> - 用弱引用，ThreadLocal 可以被 GC，避免内存泄漏
> - 但 value 仍需手动 remove()

**应用场景**：
1. 数据库连接管理
2. 用户上下文传递
3. SimpleDateFormat 线程安全

```java
// SimpleDateFormat 线程安全
private static final ThreadLocal<SimpleDateFormat> dateFormat = 
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

public String format(Date date) {
    return dateFormat.get().format(date);
}
```

### Q7: 线程池参数和执行流程？

**核心概念**：

线程池通过复用线程，减少创建销毁开销，提高性能。

**核心参数**：

```java
public ThreadPoolExecutor(
    int corePoolSize,              // 核心线程数
    int maximumPoolSize,           // 最大线程数
    long keepAliveTime,            // 空闲线程存活时间
    TimeUnit unit,                 // 时间单位
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,   // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略
)
```

**参数详解**：

| 参数 | 说明 | 建议 |
|------|------|------|
| corePoolSize | 核心线程数，空闲不回收 | CPU 密集型：N+1；IO 密集型：2N |
| maximumPoolSize | 最大线程数 | 根据系统资源设置 |
| keepAliveTime | 非核心线程空闲存活时间 | 默认 60s |
| workQueue | 工作队列 | **必须有界**，防止 OOM |
| handler | 拒绝策略 | 根据业务选择 |

**执行流程**：

```
提交任务
    │
    ▼
┌─────────────────┐
│ 线程数 < core？  │
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
    ▼         ▼
   是         否
    │         │
    ▼         ▼
创建核心线程  ┌─────────────────┐
    │        │ 队列未满？       │
    │        └────────┬────────┘
    │                 │
    │            ┌────┴────┐
    │            │         │
    │            ▼         ▼
    │           是         否
    │            │         │
    │            ▼         ▼
    │        加入队列  ┌─────────────────┐
    │            │    │ 线程数 < max？  │
    │            │    └────────┬────────┘
    │            │             │
    │            │        ┌────┴────┐
    │            │        │         │
    │            │        ▼         ▼
    │            │       是         否
    │            │        │         │
    │            │        ▼         ▼
    │            │   创建非核心线程  执行拒绝策略
    │            │        │
    └────────────┴────────┘
                │
                ▼
            执行任务
```

**四种拒绝策略**：

| 策略 | 行为 | 适用场景 |
|------|------|---------|
| AbortPolicy | 抛 RejectedExecutionException | 默认，需要感知错误 |
| CallerRunsPolicy | 调用者线程执行 | 不丢弃任务 |
| DiscardPolicy | 直接丢弃 | 允许丢失 |
| DiscardOldestPolicy | 丢弃队列最老任务 | 允许丢失旧任务 |

**为什么不用 Executors？**：

```java
// ❌ 不推荐
Executors.newFixedThreadPool(10);   // 队列无界，可能 OOM
Executors.newCachedThreadPool();     // 线程无界，可能 OOM

// ✅ 推荐
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    10,                             // 核心线程数
    20,                             // 最大线程数
    60L, TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000), // 有界队列
    new ThreadFactoryBuilder().setNameFormat("worker-%d").build(),
    new ThreadPoolExecutor.CallerRunsPolicy()
);
```

**面试加分点**：
> 如何合理设置参数？
> 1. CPU 密集型：corePoolSize = N + 1
> 2. IO 密集型：corePoolSize = 2N
> 3. 混合型：corePoolSize = N × (1 + 等待时间/计算时间)
> 4. **必须压测验证**

**追问延伸**：
- 线程池的状态有哪些？
  - RUNNING：接受新任务，处理队列任务
  - SHUTDOWN：不接受新任务，处理队列任务
  - STOP：不接受新任务，不处理队列任务
  - TIDYING：所有任务终止
  - TERMINATED：terminated() 完成

### Q8: ConcurrentHashMap 实现？

**核心概念**：

ConcurrentHashMap 是线程安全的 HashMap，JDK 1.7 和 1.8 实现差异很大。

**JDK 1.7：分段锁**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    JDK 1.7 分段锁结构                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  Segment[] segments                                       │ │
│  │  ┌─────────┬─────────┬─────────┬─────────┐              │ │
│  │  │Segment 0│Segment 1│Segment 2│  ...    │              │ │
│  │  │(Reentrant│(Reentrant│(Reentrant│       │              │ │
│  │  │  Lock)  │  Lock)  │  Lock)  │       │              │ │
│  │  └────┬────┴────┬────┴────┬────┴─────────┘              │ │
│  │       │         │         │                             │ │
│  │       ▼         ▼         ▼                             │ │
│  │  ┌─────────┬─────────┬─────────┐                        │ │
│  │  │HashEntry│HashEntry│HashEntry│  每个 Segment          │ │
│  │  │   []    │   []    │   []    │  独立加锁              │ │
│  │  └─────────┴─────────┴─────────┘                        │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

特点：
- 16 个 Segment，每个 Segment 是一个小的 HashMap
- 每个 Segment 独立加锁，并发度 = Segment 数量
```

**JDK 1.8：CAS + synchronized**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    JDK 1.8 结构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Node[] table                                                   │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐            │
│  │Node │Node │Node │Node │Node │Node │Node │ ... │            │
│  │     │     │     │     │     │     │     │     │            │
│  └──┬──┴──┬──┴──┬──┴─────┴─────┴─────┴─────┴─────┘            │
│     │     │     │                                              │
│     ▼     ▼     ▼                                              │
│   链表   红黑树  链表                                           │
│                                                                 │
│  put 流程：                                                     │
│  1. 桶为空 → CAS 插入                                           │
│  2. 桶不为空 → synchronized 锁头节点                             │
│  3. 链表/红黑树插入                                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**put 流程代码**：

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. 计算 hash
    int hash = spread(key.hashCode());
    
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        
        // 2. 桶为空，CAS 插入
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        // 3. 桶不为空，synchronized 锁头节点
        else {
            synchronized (f) {
                // 遍历链表或红黑树，插入节点
            }
        }
    }
    return null;
}
```

**面试加分点**：
> 为什么 JDK 1.8 放弃分段锁？
> 1. 每个 Segment 都要占用内存，空间开销大
> 2. 并发度受 Segment 数量限制（默认 16）
> 3. CAS + synchronized 锁粒度更细，只锁冲突的桶

**追问延伸**：
- ConcurrentHashMap 能完全替代 Hashtable 吗？
  - 不能！ConcurrentHashMap 不允许 null 键/值
  - Hashtable 允许 null 键/值
  
- ConcurrentHashMap 的 size() 是怎么实现的？
  - JDK 1.8 使用 LongAdder 思路，分散计数
  - 汇总 baseCount + CounterCell[]
