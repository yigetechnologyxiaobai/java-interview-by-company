## 分布式高频题


### Q13: 分布式事务解决方案？

**核心概念**：

分布式事务保证跨服务操作的原子性。

**方案对比**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      分布式事务方案                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 2PC（两阶段提交）                                            │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  准备阶段：协调者询问所有参与者是否可以提交            │  │
│     │  提交阶段：所有参与者都同意后，协调者发送提交命令      │  │
│     └───────────────────────────────────────────────────────┘  │
│     问题：阻塞、单点故障、数据不一致                             │
│                                                                 │
│  2. TCC（Try-Confirm-Cancel）                                    │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  Try:    尝试预留资源                                  │  │
│     │  Confirm: 确认提交（真正执行）                         │  │
│     │  Cancel: 取消预留（回滚）                              │  │
│     └───────────────────────────────────────────────────────┘  │
│     问题：业务侵入大，需要实现三个接口                            │
│                                                                 │
│  3. 本地消息表                                                   │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  1. 本地事务写入消息表                                 │  │
│     │  2. 定时任务扫描消息表，发送消息                       │  │
│     │  3. 消费者处理消息，更新状态                           │  │
│     └───────────────────────────────────────────────────────┘  │
│     优点：实现简单，最终一致性                                   │
│                                                                 │
│  4. 事务消息（RocketMQ）                                         │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  1. 发送半消息（不可消费）                             │  │
│     │  2. 执行本地事务                                       │  │
│     │  3. 提交/回滚消息                                      │  │
│     │  4. 消费者消费消息                                     │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  5. Seata（AT/TCC/SAGA）                                        │
│     AT 模式：无侵入，自动生成回滚 SQL                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**面试加分点**：
> TCC vs 2PC：
> - 2PC：资源层面，阻塞，依赖数据库
> - TCC：业务层面，非阻塞，需要业务代码实现

### Q14: 分布式锁实现？

**核心概念**：

分布式锁保证跨进程的互斥访问。

**Redis 实现**：

```java
// 加锁
SET lock:key value NX PX 30000

// 解锁（Lua 脚本保证原子性）
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

**Redisson 实现**：

```java
RLock lock = redissonClient.getLock("lock:key");
try {
    // 尝试获取锁，等待 10 秒，锁自动过期时间 30 秒
    if (lock.tryLock(10, 30, TimeUnit.SECONDS)) {
        // 业务逻辑
    }
} finally {
    lock.unlock();
}
```

**看门狗机制**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Redisson 看门狗                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：业务执行时间超过锁过期时间，锁自动释放                    │
│                                                                 │
│  解决：看门狗自动续期                                            │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  1. 获取锁成功后，启动看门狗线程                          │ │
│  │  2. 每 10 秒检查一次（过期时间/3）                        │ │
│  │  3. 如果锁还被持有，自动续期到 30 秒                      │ │
│  │  4. 业务执行完毕，停止看门狗                              │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Zookeeper 实现**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Zookeeper 分布式锁                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  实现方式：临时顺序节点                                          │
│                                                                 │
│  /lock                                                         │
│    ├── lock-000001    # 节点 1（获取锁）                        │
│    ├── lock-000002    # 节点 2（等待）                          │
│    └── lock-000003    # 节点 3（等待）                          │
│                                                                 │
│  流程：                                                         │
│  1. 创建临时顺序节点                                            │
│  2. 判断自己是否是最小节点                                       │
│  3. 是 → 获取锁                                                 │
│  4. 否 → 监听前一个节点的删除事件                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**方案对比**：

| 维度 | Redis | Zookeeper |
|------|-------|-----------|
| 性能 | 高 | 中 |
| 可靠性 | 中（主从切换可能丢锁） | 高（CP 系统） |
| 实现 | 简单 | 复杂 |
| 适用 | 性能优先 | 可靠性优先 |

### Q15: 如何设计一个秒杀系统？

**核心概念**：

秒杀系统特点是瞬时高并发、库存有限、防超卖。

**架构设计**：

```
┌─────────────────────────────────────────────────────────────────┐
│                       秒杀系统架构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  用户 → CDN → 网关 → 应用服务 → Redis → MQ → 数据库            │
│                                                                 │
│  1. 前端层                                                       │
│     └─ 按钮置灰、验证码、限流                                   │
│                                                                 │
│  2. CDN 层                                                       │
│     └─ 静态资源缓存、请求拦截                                   │
│                                                                 │
│  3. 网关层                                                       │
│     └─ 限流、黑名单、请求过滤                                   │
│                                                                 │
│  4. 应用层                                                       │
│     └─ 本地缓存、Redis 预扣库存                                 │
│                                                                 │
│  5. Redis 层                                                     │
│     └─ 库存预热、原子扣减                                       │
│                                                                 │
│  6. MQ 层                                                        │
│     └─ 异步下单、削峰填谷                                       │
│                                                                 │
│  7. 数据库层                                                     │
│     └─ 最终数据落地                                             │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**防超卖实现**：

```java
// Redis Lua 脚本原子扣减
String luaScript = """
    if redis.call('get', KEYS[1]) <= 0 then
        return 0
    end
    return redis.call('decr', KEYS[1])
    """;

Long result = redisTemplate.execute(luaScript, Collections.singletonList("stock:key"));
if (result < 0) {
    // 库存不足
}
```

**防重复下单**：

```java
// 使用 Redis SETNX
Boolean success = redisTemplate.opsForValue()
    .setIfAbsent("order:user:" + userId, "1", 1, TimeUnit.HOURS);
if (!success) {
    // 重复下单
}
```

**面试加分点**：
> 秒杀系统核心指标：
> 1. QPS：衡量系统并发能力
> 2. 成功率：下单成功比例
> 3. 响应时间：用户等待时间
> 
> 优化技巧：
> - 库存预热到 Redis
> - 本地缓存减少 Redis 访问
> - MQ 异步削峰
> - 限流保护系统

### Q16: Spring Bean 生命周期？

**核心概念**：

Bean 生命周期包括实例化、属性赋值、初始化、销毁四个阶段。

**详细流程**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Bean 生命周期                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 实例化                                                       │
│     └─ InstantiationAwareBeanPostProcessor.postProcessBefore   │
│     └─ 构造函数创建对象                                         │
│     └─ InstantiationAwareBeanPostProcessor.postProcessAfter    │
│                                                                 │
│  2. 属性赋值                                                     │
│     └─ InstantiationAwareBeanPostProcessor.postProcessProperties│
│     └─ @Autowired、@Value 依赖注入                              │
│                                                                 │
│  3. 初始化                                                       │
│     └─ Aware 接口回调                                           │
│        (BeanNameAware、BeanFactoryAware、ApplicationContextAware)│
│     └─ BeanPostProcessor.postProcessBeforeInitialization       │
│     └─ @PostConstruct                                          │
│     └─ InitializingBean.afterPropertiesSet                      │
│     └─ init-method                                             │
│     └─ BeanPostProcessor.postProcessAfterInitialization        │
│        (AOP 代理在此创建)                                       │
│                                                                 │
│  4. 使用                                                         │
│     └─ Bean 正常工作                                            │
│                                                                 │
│  5. 销毁                                                         │
│     └─ @PreDestroy                                             │
│     └─ DisposableBean.destroy                                   │
│     └─ destroy-method                                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
@Component
public class MyBean implements BeanNameAware, InitializingBean, DisposableBean {
    
    private String beanName;
    
    // 1. Aware 接口回调
    @Override
    public void setBeanName(String name) {
        this.beanName = name;
    }
    
    // 2. @PostConstruct
    @PostConstruct
    public void init() {
        System.out.println("@PostConstruct");
    }
    
    // 3. InitializingBean
    @Override
    public void afterPropertiesSet() {
        System.out.println("afterPropertiesSet");
    }
    
    // 4. @PreDestroy
    @PreDestroy
    public void preDestroy() {
        System.out.println("@PreDestroy");
    }
    
    // 5. DisposableBean
    @Override
    public void destroy() {
        System.out.println("destroy");
    }
}
```

**面试加分点**：
> 循环依赖如何解决？
> - Spring 使用三级缓存解决 setter 注入的循环依赖
> 
> ```java
> // 三级缓存
> // 一级：完整 Bean
> Map<String, Object> singletonObjects;
> // 二级：早期 Bean（未完成初始化）
> Map<String, Object> earlySingletonObjects;
> // 三级：ObjectFactory（用于生成代理）
> Map<String, ObjectFactory<?>> singletonFactories;
> ```

**追问延伸**：
- 构造器注入的循环依赖能解决吗？
  - 不能！只能用 @Lazy 延迟加载解决

### Q17: Spring AOP 原理？

**核心概念**：

AOP（面向切面编程）通过动态代理实现横切关注点的分离。

**动态代理**：

```java
// JDK 动态代理（接口）
public class JdkProxy implements InvocationHandler {
    private Object target;
    
    public Object proxy() {
        return Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            this
        );
    }
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 前置通知
        Object result = method.invoke(target, args);
        // 后置通知
        return result;
    }
}

// CGLIB（类）
public class CglibProxy implements MethodInterceptor {
    public Object proxy(Class<?> clazz) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(clazz);
        enhancer.setCallback(this);
        return enhancer.create();
    }
    
    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        // 前置通知
        Object result = proxy.invokeSuper(obj, args);
        // 后置通知
        return result;
    }
}
```

**选择策略**：

| 情况 | 代理方式 |
|------|---------|
| 有接口 | JDK 动态代理 |
| 无接口 | CGLIB |
| 配置 proxy-target-class=true | 强制 CGLIB |

**面试加分点**：
> AOP 如何实现事务？
> - TransactionInterceptor 作为环绕通知
> - 方法执行前开启事务
> - 方法执行成功提交事务
> - 方法执行异常回滚事务

### Q18: Spring 事务传播机制？

**核心概念**：

事务传播机制定义了事务方法之间如何传播事务上下文。

**传播行为**：

| 传播行为 | 说明 | 场景 |
|---------|------|------|
| REQUIRED | 有则加入，无则新建（默认） | 大多数场景 |
| SUPPORTS | 有则加入，无则非事务 | 查询 |
| MANDATORY | 必须有事务，否则抛异常 | 强制事务 |
| REQUIRES_NEW | 新建事务，挂起当前 | 独立日志 |
| NOT_SUPPORTED | 非事务执行，挂起当前 | 非事务操作 |
| NEVER | 非事务执行，有事务则抛异常 | 禁止事务 |
| NESTED | 嵌套事务 | 部分回滚 |

**REQUIRED vs NESTED**：

```java
// REQUIRED：外层回滚，内层也回滚
@Transactional(propagation = Propagation.REQUIRED)
public void outer() {
    inner();  // 内层异常，外层也回滚
}

@Transactional(propagation = Propagation.REQUIRED)
public void inner() {
    // 异常
}

// NESTED：内层回滚，外层可以不回滚
@Transactional(propagation = Propagation.REQUIRED)
public void outer() {
    try {
        inner();  // 内层异常，只回滚内层
    } catch (Exception e) {
        // 处理异常，外层继续
    }
}

@Transactional(propagation = Propagation.NESTED)
public void inner() {
    // 异常
}
```

**事务失效场景**：

```java
@Service
public class UserService {
    
    // ❌ 事务失效：方法不是 public
    @Transactional
    private void method1() {}
    
    // ❌ 事务失效：同类调用
    public void method2() {
        this.method3();  // 不走代理
    }
    
    @Transactional
    public void method3() {}
    
    // ❌ 事务失效：异常被捕获
    @Transactional
    public void method4() {
        try {
            // 异常
        } catch (Exception e) {
            // 吞掉异常
        }
    }
    
    // ❌ 事务失效：抛出非 RuntimeException
    @Transactional
    public void method5() throws Exception {
        throw new Exception();  // 非 RuntimeException
    }
}
```
