## MySQL 高频题


### Q9: MySQL 为什么用 B+ 树？

**核心概念**：

B+ 树是 B 树的变体，更适合磁盘存储。

**B+ 树结构**：

```
                    B+ 树结构
                    
                    ┌───────┐
                    │  20   │
                    └───┬───┘
                ┌────────┴────────┐
                ▼                 ▼
            ┌───────┐         ┌───────┐
            │10  20 │         │30  40 │
            └───┬───┘         └───┬───┘
                │                 │
        ┌───────┼───────┐    ┌───────┼───────┐
        ▼       ▼       ▼    ▼       ▼       ▼
    ┌───────┬───────┬───────┬───────┬───────┬───────┐
    │data1  │data2  │data3  │data4  │data5  │data6  │
    └───┬───┴───┬───┴───────┴───────┴───────┴───────┘
        │       │
        ▼       ▼
    所有叶子节点用指针串联
```

**为什么用 B+ 树**：

| 特点 | 好处 |
|------|------|
| 树高度低（3-4层） | 减少 IO 次数 |
| 叶子节点链表 | 范围查询高效 |
| 非叶子节点只存键值 | 单页存更多索引项 |

**B 树 vs B+ 树**：

| 维度 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点 | 仅叶子节点 |
| 叶子节点 | 无指针 | 有链表指针 |
| 范围查询 | 不友好 | 友好 |
| 树高度 | 较高 | 较低 |
| 单页存储 | 较少索引项 | 更多索引项 |

**面试加分点**：
> 为什么不用 Hash 索引？
> - Hash 索引不支持范围查询
> - Hash 索引不支持排序
> - Hash 索引不支持最左前缀匹配
> - Memory 引擎默认使用 Hash 索引

**追问延伸**：
- InnoDB 一页多大？能存多少数据？
  - 默认 16KB
  - 假设主键 8 字节，指针 6 字节
  - 非叶子节点可存：16KB / 14B ≈ 1170 个索引项
  - 3 层 B+ 树可存：1170 × 1170 × 16 ≈ 2000 万行

### Q10: 索引失效场景？

**核心概念**：

索引失效会导致全表扫描，严重影响性能。

**失效场景**：

```sql
-- 1. LIKE 以 % 开头
WHERE name LIKE '%三';   -- ❌ 索引失效
WHERE name LIKE '张%';   -- ✅ 索引生效

-- 2. 对索引列做运算
WHERE YEAR(create_time) = 2024;  -- ❌ 索引失效
WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';  -- ✅

-- 3. 类型隐式转换
WHERE phone = 13800138000;        -- ❌ phone 是 varchar
WHERE phone = '13800138000';      -- ✅

-- 4. OR 条件
WHERE name = '张三' OR age = 20;  -- ❌ 如果 age 没索引

-- 5. 使用函数
WHERE LOWER(name) = 'zhangsan';   -- ❌ 索引失效

-- 6. 不等于
WHERE name != '张三';             -- ❌ 可能失效

-- 7. IS NOT NULL
WHERE name IS NOT NULL;           -- ❌ 可能失效
```

**最左前缀原则**：

```sql
-- 联合索引 (a, b, c)

-- ✅ 能用到索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- ❌ 用不到索引
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3

-- ⚠️ 部分用到
WHERE a = 1 AND c = 3  -- 只用到 a
```

**覆盖索引**：

```sql
-- 联合索引 (name, age)

-- 覆盖索引：查询列都在索引中，无需回表
SELECT name, age FROM user WHERE name = '张三';  -- ✅ 无需回表

-- 需要回表
SELECT * FROM user WHERE name = '张三';          -- ❌ 需要回表
```

**EXPLAIN 分析**：

```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**type 从好到坏**：

```
system > const > eq_ref > ref > range > index > ALL

const: 主键或唯一索引常量查询
eq_ref: 主键或唯一索引关联查询
ref: 非唯一索引查询
range: 范围查询
index: 全索引扫描
ALL: 全表扫描（最差）
```

**面试加分点**：
> 如何优化深分页？
> 
> ```sql
> -- 问题：深分页效率低
> SELECT * FROM user LIMIT 1000000, 10;
> 
> -- 优化1：覆盖索引 + JOIN
> SELECT a.* FROM user a
> JOIN (SELECT id FROM user LIMIT 1000000, 10) b
> ON a.id = b.id;
> 
> -- 优化2：记录上次最大 ID
> SELECT * FROM user WHERE id > 1000000 LIMIT 10;
> ```

### Q11: MVCC 原理？

**核心概念**：

MVCC（Multi-Version Concurrency Control）是多版本并发控制，用于实现读已提交和可重复读隔离级别。

**核心组件**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        MVCC 核心组件                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 隐藏字段                                                     │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ DB_TRX_ID  │ 事务 ID，最后一次修改该行的事务            │  │
│     │ DB_ROLL_PTR│ 回滚指针，指向 Undo Log 中的上一个版本    │  │
│     │ DB_ROW_ID  │ 隐藏主键（如果没有显式主键）              │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  2. Undo Log（版本链）                                           │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  当前数据 → Undo Log 1 → Undo Log 2 → Undo Log 3      │  │
│     │   (trx=100)   (trx=99)    (trx=98)    (trx=97)        │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  3. Read View（可见性判断）                                      │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ m_ids:  生成 Read View 时的活跃事务 ID 列表             │  │
│     │ min_trx_id: 活跃事务中最小的事务 ID                     │  │
│     │ max_trx_id: 下一个将分配的事务 ID                       │  │
│     │ creator_trx_id: 创建该 Read View 的事务 ID              │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**版本链示例**：

```
┌─────────────────────────────────────────────────────────────────┐
│                       Undo Log 版本链                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  当前数据：name='王五', DB_TRX_ID=103, DB_ROLL_PTR→             │
│     │                                                           │
│     ▼                                                           │
│  Undo Log 2: name='李四', DB_TRX_ID=102, DB_ROLL_PTR→          │
│     │                                                           │
│     ▼                                                           │
│  Undo Log 1: name='张三', DB_TRX_ID=101                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘

事务 103 读取时：
- 如果 Read View 中 m_ids=[101, 102]，则看不到 101, 102 的修改
- 需要继续找更早的版本
```

**隔离级别实现**：

| 隔离级别 | Read View 生成时机 |
|---------|-------------------|
| RC（读已提交） | 每次 SELECT 都生成新的 Read View |
| RR（可重复读） | 事务第一次 SELECT 生成 Read View，后续复用 |

**面试加分点**：
> MVCC 能解决幻读吗？
> - 快照读：MVCC 可以解决
> - 当前读（SELECT ... FOR UPDATE）：需要间隙锁配合

**追问延伸**：
- 当前读和快照读的区别？
  - 快照读：普通 SELECT，读取历史版本
  - 当前读：SELECT ... FOR UPDATE、INSERT、UPDATE、DELETE，读取最新版本

### Q12: 主从复制原理？

**核心概念**：

MySQL 主从复制基于 Binlog 实现，用于读写分离、数据备份。

**复制流程**：

```
┌─────────────────────────────────────────────────────────────────┐
│                       主从复制流程                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Master                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  1. 写入 Binlog                                          │   │
│  │     └─ ROW/STATEMENT/MIXED 格式                         │   │
│  └──────────────────────────────┬──────────────────────────┘   │
│                                 │                               │
│                    ┌────────────▼────────────┐                  │
│                    │  Binlog Dump Thread     │                  │
│                    │  (发送 Binlog)          │                  │
│                    └────────────┬────────────┘                  │
│                                 │                               │
│                                 │  网络                         │
│                                 │                               │
│  Slave                          │                               │
│  ┌──────────────────────────────▼──────────────────────────┐   │
│  │  2. IO Thread                                           │   │
│  │     └─ 读取 Binlog，写入 Relay Log                      │   │
│  └──────────────────────────────┬──────────────────────────┘   │
│                                 │                               │
│  ┌──────────────────────────────▼──────────────────────────┐   │
│  │  3. SQL Thread                                          │   │
│  │     └─ 重放 Relay Log                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**主从延迟问题**：

| 原因 | 解决方案 |
|------|---------|
| 大事务 | 拆分大事务 |
| 单线程重放 | 开启并行复制 |
| 网络延迟 | 优化网络 |
| 硬件瓶颈 | 升级硬件 |

**并行复制配置**：

```sql
-- MySQL 5.7 并行复制
SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
SET GLOBAL slave_parallel_workers = 4;
```

**面试加分点**：
> 如何保证主从一致性？
> 1. 半同步复制：至少一个 Slave 确认收到 Binlog
> 2. 强一致性方案：使用 Paxos/Raft 协议（如 MySQL Group Replication）
