### 一、Java 基础

### 1. HashMap 在并发环境下会有什么问题？JDK 8 之后如何改进的？

**参考答案**：

**JDK 7 的问题**：
- **死循环**：多线程扩容时，头插法导致链表成环（transfer 方法中的 rehash 操作）
- **数据丢失**：多线程 put 时，同一个 bucket 的节点被覆盖

**JDK 8 改进**：
- 改用**尾插法**，避免了链表成环
- 扩容时链表转为**红黑树**（桶内元素 ≥ 8 且数组长度 ≥ 64）
- 并发 put 仍然不安全（数据覆盖），需要使用 ConcurrentHashMap

**追问**：ConcurrentHashMap 在 JDK 8 是如何保证线程安全的？
- CAS + synchronized，锁粒度细化到桶（Node）级别
- size 操作使用 CounterCell 数组分散计数，避免锁竞争

### 2. String、StringBuilder、StringBuffer 的区别？

| 特性 | String | StringBuilder | StringBuffer |
|------|--------|--------------|-------------|
| 可变性 | 不可变（final char[]） | 可变 | 可变 |
| 线程安全 | 是 | 否 | 是（synchronized） |
| 性能 | - | 最快 | 较慢 |
| 适用场景 | 少量字符串操作 | 单线程大量拼接 | 多线程拼接 |

**补充**：String 不可变的好处——线程安全、可缓存 hashCode、字符串常量池复用。

### 二、并发编程

### 3. 线程池的核心参数有哪些？拒绝策略有几种？

**参考答案**：

**7 大核心参数**：
```java
ThreadPoolExecutor(int corePoolSize,     // 核心线程数
                   int maximumPoolSize,   // 最大线程数
                   long keepAliveTime,    // 空闲线程存活时间
                   TimeUnit unit,
                   BlockingQueue<Runnable> workQueue,  // 工作队列
                   ThreadFactory threadFactory,        // 线程工厂
                   RejectedExecutionHandler handler)   // 拒绝策略
```

**4 种拒绝策略**：
1. **AbortPolicy**（默认）：抛出 RejectedExecutionException
2. **CallerRunsPolicy**：调用者线程执行，起到降速作用
3. **DiscardPolicy**：静默丢弃
4. **DiscardOldestPolicy**：丢弃队列最老的任务

**携程实战问题**：你们的线程池参数怎么设置的？
- IO 密集型：corePoolSize = 2N + 1（N = CPU 核数）
- CPU 密集型：corePoolSize = N
- 工作队列建议用 LinkedBlockingQueue 并设置合理容量

### 4. volatile 关键字的作用是什么？能替代 synchronized 吗？

**参考答案**：

**volatile 三大作用**：
1. **可见性**：修改变量后立即刷入主内存，其他线程立即可见（通过 MESI 缓存一致性协议）
2. **禁止指令重排序**：插入内存屏障（LoadLoad、StoreStore 等）
3. **不保证原子性**：复合操作（如 i++）仍需 synchronized 或 AtomicInteger

**与 synchronized 的区别**：
- volatile 是**轻量级**的，只保证可见性和有序性
- synchronized 保证**原子性 + 可见性**，但重量级
- **不能替代**，场景不同

### 三、MySQL

### 5. 一条 SQL 执行慢，怎么排查和优化？

**参考答案**：

**排查步骤**：
1. `EXPLAIN` 分析执行计划 → 看 type、key、rows、Extra
2. 查看慢查询日志 → `mysqldumpslow` 分析
3. 检查锁等待 → `SHOW ENGINE INNODB STATUS`
4. 查看执行 profile → `SET profiling=1; SHOW PROFILES;`

**优化方向**：
- **索引优化**：联合索引最左前缀、避免索引失效（函数运算、类型转换、OR 条件）
- **SQL 改写**：避免 SELECT *、子查询改 JOIN、分页优化（延迟关联）
- **表结构**：字段类型合理、避免 NULL、适当冗余
- **架构层**：读写分离、分库分表、缓存前置

### 6. MVCC 是什么？MySQL 是如何实现的？

**参考答案**：

**MVCC（多版本并发控制）**：通过保存数据的历史版本，实现读写不阻塞。

**MySQL 实现**：
- **隐藏列**：每行有 `trx_id`（事务ID）和 `roll_pointer`（回滚指针）
- **Undo Log**：通过回滚指针串联历史版本
- **Read View**：快照读时创建，包含活跃事务列表

**RC vs RR 级别差异**：
- **READ COMMITTED**：每次 SELECT 都生成新 Read View
- **REPEATABLE READ**：第一次 SELECT 生成 Read View，后续复用（实现可重复读）

### 四、Redis

### 7. Redis 缓存和数据库一致性怎么保证？

**参考答案**：

**常用方案对比**：

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 先删缓存再更新 DB | 简单 | 实现简单 | 并发读可能读到旧数据 |
| 先更新 DB 再删缓存 | Cache Aside | 主流方案 | 极端情况短暂不一致 |
| 延迟双删 | 先删 → 更新DB → 延时再删 | 降低不一致窗口 | 延时时间难确定 |
| 订阅 binlog 异步删缓存 | Canal + MQ | 最终一致 | 架构复杂 |

**携程实践**：采用 Cache Aside 模式 + Canal 订阅 binlog 异步更新缓存，对一致性要求高的场景加分布式锁。

### 8. Redis 大 Key 有什么危害？怎么发现和处理？

**参考答案**：

**危害**：
- 阻塞 Redis 主线程（删除大 Key 时 DEL 操作阻塞）
- 网络拥塞（一次传输数据过大）
- 内存不均，影响集群均衡

**发现方式**：
- `redis-cli --bigkeys` 扫描
- 监控 key 大小分布
- 定期分析 RDB 文件

**处理方式**：
- **拆分**：Hash 拆成多个小 Hash，String 拆成多个小 String
- **异步删除**：Redis 4.0+ 使用 `UNLINK` 替代 `DEL`
- **过期策略**：设置合理的 TTL，避免无限增长

### 五、分布式与微服务

### 9. 分布式锁有哪些实现方案？各有什么优缺点？

**参考答案**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| Redis SETNX + EXPIRE | 简单、性能好 | 非公平锁、RedLock 争议 | 一般业务场景 |
| ZooKeeper 临时节点 | 强一致、可重入 | 性能低、依赖 ZK | 强一致性要求 |
| 数据库唯一索引 | 简单 | 性能差、无超时自动释放 | 低频场景 |
| Redisson | 功能全（可重入、公平锁、WatchDog） | 依赖 Redisson 库 | 推荐首选 |

**Redisson WatchDog 原理**：后台线程每隔 10s 续期，避免业务未执行完锁过期。

### 10. 微服务之间如何保证调用不雪崩？

**参考答案**：

**防护体系**：
1. **超时控制**：设置合理超时时间，避免长时间等待
2. **熔断降级**：Hystrix / Sentinel，错误率达到阈值自动熔断
3. **限流**：令牌桶/漏桶算法，保护下游服务
4. **隔离**：线程池隔离或信号量隔离，故障不扩散
5. **重试机制**：指数退避重试，避免重试风暴
6. **异步化**：消息队列解耦，非核心链路异步处理

**携程实践**：使用 Sentinel 做限流熔断，结合自研的 RPC 框架实现服务治理。

## 携程二面真题（主管面）

**背景**：社招 Java 后端开发（高级）

### 11. 设计一个机票搜索系统，支持多条件筛选、实时价格、高并发查询

**参考答案要点**：

**架构分层**：
```
客户端 → CDN → 网关层 → 搜索服务 → 缓存层 → 数据层
```

**关键设计**：
- **搜索索引**：Elasticsearch 做多条件筛选，避免扫表
- **价格缓存**：Redis 缓存热点航线价格，设置短 TTL（如 30s）
- **异步更新**：航司价格变动通过 MQ 异步通知，更新缓存和索引
- **分页优化**：ES scroll / search_after 替代 deep paging
- **降级策略**：缓存不可用时返回兜底数据

### 12. 订单超时自动取消，怎么设计？

**参考答案**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 定时任务轮询 DB | 简单 | 延迟高、DB 压力大 |
| Redis 过期事件监听 | 延迟低 | 可靠性差（事件可能丢失） |
| Redisson DelayedQueue | 可靠、延迟低 | 依赖 Redis |
| RocketMQ 延时消息 | 企业级、可靠 | 依赖 MQ |
| 时间轮算法 | 内存高效 | 实现复杂 |

**推荐方案**：RocketMQ 延时消息 + 补偿任务兜底。
