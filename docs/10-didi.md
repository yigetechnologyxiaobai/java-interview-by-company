# 滴滴 Java 面试题

## 面试风格

**特点**：注重业务场景，关注高并发和分布式，喜欢项目经验

**轮次**：通常 2-3 轮技术面 + 1 轮 HR

**重点**：分布式、MySQL、Redis、消息队列、业务场景设计

## 一、分布式高频题

### Q1: 分布式锁实现？

**参考答案**：

**Redis 实现**：
```redis
# 加锁
SET lock:key value NX PX 30000

# 解锁（Lua 脚本保证原子性）
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

**Zookeeper 实现**：
- 临时顺序节点
- EPHEMERAL_SEQUENTIAL
- Watch 机制

**对比**：
| 方案 | 性能 | 可靠性 | 适用场景 |
|------|------|--------|---------|
| Redis | 高 | 中 | 性能优先 |
| Zookeeper | 中 | 高 | 可靠性优先 |

**Redisson 增强**：
- 可重入锁
- 看门狗自动续期
- 红锁（Redlock）

**追问**：
- 锁过期了但业务没执行完怎么办？
- 主从切换导致锁丢失怎么办？

### Q2: 分布式事务？

**参考答案**：

| 方案 | 原理 | 场景 |
|------|------|------|
| 2PC | 准备 + 提交 | 强一致 |
| TCC | Try-Confirm-Cancel | 高并发 |
| 本地消息表 | 消息 + 定时任务 | 最终一致 |
| 事务消息 | RocketMQ 半消息 | 最终一致 |
| Seata AT | 全局锁 + Undo Log | 简单场景 |

**TCC 流程**：
1. Try：预留资源
2. Confirm：确认提交
3. Cancel：回滚释放

**TCC 问题**：
- **空回滚**：Try 未执行，Cancel 被调用
- **悬挂**：Cancel 先于 Try 执行
- **幂等**：Confirm/Cancel 需要幂等

**追问**：
- TCC 和 2PC 区别？
- Seata AT 模式原理？

### Q3: 分布式 ID 生成？

**参考答案**：

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 随机生成 | 简单 | 无序 |
| 数据库自增 | 自增 ID | 简单 | 单点 |
| Redis | INCR | 高性能 | 依赖 Redis |
| Snowflake | 时间戳 + 机器 ID + 序列号 | 分布式 | 依赖时钟 |
| Leaf | Snowflake + 数据库 | 综合 | 复杂 |

**Snowflake 结构**：
```
0 - 41bit 时间戳 - 10bit 机器 ID - 12bit 序列号
```

**时钟回拨问题**：
- 等待时间追上
- 使用备用机器 ID
- 抛异常拒绝

**追问**：
- Snowflake 如何保证唯一？
- 如何处理时钟回拨？

### Q4: 分布式 Session？

**参考答案**：

**方案对比**：

| 方案 | 描述 | 优缺点 |
|------|------|--------|
| 粘性 Session | 绑定用户到服务器 | 简单，但不均衡 |
| Session 复制 | 服务器间同步 Session | 复杂，性能差 |
| 集中式存储 | Redis 存储 Session | 推荐 |

**Redis 实现**：
```java
// 登录时
redis.set("session:" + token, userJson, 3600);

// 请求时
String userJson = redis.get("session:" + token);
```

**JWT 方案**：
- 无状态
- 不需要存储
- 但无法主动失效

**追问**：
- JWT 和 Session 区别？
- 如何实现单点登录？

## 二、MySQL 高频题

### Q5: 索引为什么用 B+ 树？

**参考答案**：

**B+ 树特点**：
1. **树高度低**：3-4 层可存千万级数据
2. **范围查询友好**：叶子节点链表
3. **磁盘利用率高**：非叶子节点只存键值

**与 B 树区别**：
| 特性 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点 | 只有叶子节点 |
| 范围查询 | 需要中序遍历 | 叶子节点链表 |
| 树高度 | 较高 | 较低 |

**追问**：
- 聚簇索引和非聚簇索引区别？
- 为什么用自增主键？

### Q6: MVCC 原理？

**参考答案**：

**核心组件**：
1. **隐藏字段**：
   - DB_TRX_ID：事务 ID
   - DB_ROLL_PTR：回滚指针

2. **Undo Log**：版本链

3. **Read View**：可见性判断
   - m_ids：活跃事务 ID 列表
   - min_trx_id：最小活跃事务 ID
   - max_trx_id：下一个事务 ID

**可见性判断**：
- trx_id < min_trx_id → 可见
- trx_id >= max_trx_id → 不可见
- trx_id 在 m_ids 中 → 不可见

**隔离级别**：
- RC：每次 select 生成新的 Read View
- RR：事务第一次 select 生成 Read View

**追问**：
- MVCC 能解决幻读吗？
- 当前读和快照读区别？

### Q7: 主从复制？

**参考答案**：

**复制流程**：
1. Master 写入 Binlog
2. Slave IO 线程读取 Binlog，写入 Relay Log
3. Slave SQL 线程重放 Relay Log

**复制模式**：
- 异步复制：性能好，可能丢数据
- 半同步复制：至少一个从库确认
- 全同步复制：所有从库确认

**主从延迟解决**：
1. 并行复制
2. 读写分离优化（关键读走主库）
3. 分库分表

**追问**：
- 如何监控主从延迟？
- GTID 复制是什么？

### Q8: 分库分表？

**参考答案**：

**垂直分库**：按业务拆分
- 用户库、订单库、商品库

**垂直分表**：按字段拆分
- 大字段拆分到扩展表

**水平分库**：按数据拆分
- 用户 ID 取模分库

**水平分表**：按数据拆分
- 订单 ID 取模分表

**分片策略**：
- 范围分片：按时间、ID 范围
- Hash 分片：按 Hash 值取模
- 一致性 Hash：减少数据迁移

**问题**：
- 跨库 JOIN
- 分布式事务
- 全局唯一 ID
- 数据迁移

**追问**：
- 如何解决跨库 JOIN？
- 如何实现数据迁移？

## 三、Redis 高频题

### Q9: Redis 为什么快？

**参考答案**：

1. **内存操作**：数据在内存中
2. **单线程**：无锁竞争
3. **IO 多路复用**：epoll
4. **高效数据结构**：SDS、跳表、压缩列表

**追问**：
- 为什么用单线程？
- Redis 6.0 为什么引入多线程？

### Q10: 缓存穿透、击穿、雪崩？

**参考答案**：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 穿透 | 查询不存在的数据 | 布隆过滤器、缓存空值 |
| 击穿 | 热点 key 过期 | 永不过期、互斥锁 |
| 雪崩 | 大量 key 同时过期 | 随机过期时间、多级缓存 |

**追问**：
- 布隆过滤器原理？
- 如何设计多级缓存？

### Q11: Redis 集群？

**参考答案**：

| 方案 | 描述 | 特点 |
|------|------|------|
| 主从复制 | 一主多从 | 读性能提升 |
| 哨兵 | 监控+故障转移 | 高可用 |
| Cluster | 分片集群 | 读写扩展 |

**Cluster 原理**：
- 16384 个槽位
- CRC16(key) % 16384 定位槽位
- Gossip 通信

**追问**：
- Cluster 如何做故障转移？
- 如何处理热点 Key？

## 六、Java 基础高频题

### Q17: HashMap 原理？

**参考答案**：

**JDK 1.8 结构**：数组 + 链表 + 红黑树

**put 流程**：
1. 计算 hash
2. 定位桶
3. 桶为空 → 直接插入
4. 桶不为空 → 遍历链表/红黑树
5. 相同 key → 覆盖 value
6. 链表长度 >= 8 → 红黑树

**追问**：
