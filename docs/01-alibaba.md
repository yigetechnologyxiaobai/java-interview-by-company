## 一、JVM 高频题

### Q1: 说一下 JVM 内存结构？

**核心概念**：

JVM 运行时数据区分为线程私有和线程共享两大部分：

```
┌─────────────────────────────────────────────────────────────────┐
│                        JVM 运行时数据区                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────── 线程私有 ───────────────────┐            │
│  │                                               │            │
│  │  ┌────────────┐ ┌────────────┐ ┌───────────┐ │            │
│  │  │程序计数器   │ │ 虚拟机栈   │ │本地方法栈 │ │            │
│  │  │PC Register │ │ VM Stack  │ │Native Stack│ │            │
│  │  └────────────┘ └────────────┘ └───────────┘ │            │
│  │                                               │            │
│  └───────────────────────────────────────────────┘            │
│                                                                 │
│  ┌─────────────────── 线程共享 ───────────────────┐            │
│  │                                               │            │
│  │  ┌─────────────────────────────────────────┐  │            │
│  │  │              堆 (Heap)                  │  │            │
│  │  │  ┌────────────┬────────────────────┐   │  │            │
│  │  │  │ 新生代      │      老年代        │   │  │            │
│  │  │  │ Young Gen  │    Old Gen        │   │  │            │
│  │  │  └────────────┴────────────────────┘   │  │            │
│  │  └─────────────────────────────────────────┘  │            │
│  │                                               │            │
│  │  ┌─────────────────────────────────────────┐  │            │
│  │  │            方法区 (Method Area)         │  │            │
│  │  │  类信息、常量、静态变量、JIT代码         │  │            │
│  │  └─────────────────────────────────────────┘  │            │
│  │                                               │            │
│  └───────────────────────────────────────────────┘            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**详细解答**：

**1. 程序计数器（Program Counter Register）**
- **作用**：记录当前线程执行到的字节码行号
- **特点**：唯一不会 OOM 的区域，线程私有
- **为什么需要**：CPU 多线程切换后，需要知道恢复到哪一行继续执行

**2. 虚拟机栈（VM Stack）**
- **作用**：每个方法执行时创建一个栈帧，存储局部变量表、操作数栈、动态链接、返回地址
- **异常**：StackOverflowError（递归过深）、OutOfMemoryError（栈空间不足）
- **配置**：`-Xss256k` 设置单个线程栈大小

**栈帧结构**：
```
┌─────────────────────────────────────────┐
│              栈帧 (Stack Frame)          │
├─────────────────────────────────────────┤
│  ┌─────────────────────────────────┐   │
│  │        局部变量表                │   │
│  │  [0] this                       │   │
│  │  [1] 参数1                       │   │
│  │  [2] 局部变量1                   │   │
│  └─────────────────────────────────┘   │
│                                         │
│  ┌─────────────────────────────────┐   │
│  │        操作数栈                  │   │
│  │   用于计算的中间结果存储          │   │
│  └─────────────────────────────────┘   │
│                                         │
│  动态链接、方法返回地址                   │
└─────────────────────────────────────────┘
```

**3. 本地方法栈**
- 为 Native 方法服务，与虚拟机栈类似

**4. 堆（Heap）**
- **作用**：存储对象实例，GC 的主要区域
- **结构**：新生代（Eden + S0 + S1）+ 老年代
- **配置**：`-Xms2g -Xmx2g` 设置初始和最大堆大小

**堆内存划分**：
```
┌────────────────────────────────────────────────────────────────┐
│                           堆 (Heap)                             │
├────────────────────────────────────────────────────────────────┤
│  ┌──────────────────────────新生代 (1/3)────────────────────┐  │
│  │  ┌──────────────────────────────────────────────────┐   │  │
│  │  │              Eden 区 (8)                          │   │  │
│  │  └──────────────────────────────────────────────────┘   │  │
│  │  ┌────────────────────┐  ┌────────────────────┐        │  │
│  │  │  Survivor S0 (1)   │  │  Survivor S1 (1)   │        │  │
│  │  └────────────────────┘  └────────────────────┘        │  │
│  └──────────────────────────────────────────────────────────┘  │
│  ┌──────────────────────────老年代 (2/3)───────────────────┐  │
│  │                                                          │  │
│  └──────────────────────────────────────────────────────────┘  │
└────────────────────────────────────────────────────────────────┘
默认比例：Eden:S0:S1 = 8:1:1，新生代:老年代 = 1:2
```

**5. 方法区（Method Area）**
- **JDK 1.6 及之前**：永久代（PermGen），在 JVM 堆中
- **JDK 1.8 及之后**：元空间（Metaspace），在本地内存中
- **存储内容**：类信息、常量、静态变量、JIT 编译后的代码

**面试加分点**：
> 永久代改为元空间的原因：
> 1. 永久代大小难确定，容易 OOM
> 2. 类卸载效率低，GC 复杂
> 3. 统一使用本地内存，与 HotSpot 内存管理模型一致

**追问延伸**：
- 堆和栈的区别是什么？
  - 堆存对象实例，线程共享；栈存栈帧和局部变量，线程私有
  - 堆空间大（GB级），栈空间小（MB级）
  - 堆是 GC 主要区域，栈不需要 GC

- 对象一定在堆上分配吗？
  - 不一定！如果对象不逃逸（Escape Analysis），可能栈上分配
  - 逃逸分析：判断对象是否会逃逸出方法
  
```java
// 对象逃逸，不能栈上分配
public User getUser() {
    User user = new User();
    return user;  // 逃逸出方法
}

// 对象不逃逸，可能栈上分配
public void process() {
    User user = new User();  // 可能分配在栈上
    user.setId(1);
}  // 方法结束，对象自动销毁
```

- 方法区在不同 JDK 版本的变化？
  - JDK 1.6：永久代在 JVM 堆
  - JDK 1.7：字符串常量池移到堆
  - JDK 1.8：永久代完全移除，改为元空间（本地内存）

### Q2: 垃圾收集算法有哪些？

**核心概念**：

垃圾收集需要解决两个问题：**哪些内存需要回收**、**什么时候回收**、**如何回收**

**1. 判断对象存活**

**引用计数法**：
```java
// 问题：循环引用无法回收
public class ReferenceCountingGC {
    public Object instance = null;
    
    public static void main(String[] args) {
        ReferenceCountingGC objA = new ReferenceCountingGC();
        ReferenceCountingGC objB = new ReferenceCountingGC();
        objA.instance = objB;
        objB.instance = objA;
        objA = null;
        objB = null;
        // 循环引用，引用计数不为 0，但实际已无引用
    }
}
```

**可达性分析（JVM 使用）**：
- 从 GC Roots 开始，遍历对象图
- 不可达的对象即为垃圾

**GC Roots 包括**：
1. 虚拟机栈中引用的对象
2. 方法区中类静态属性引用的对象
3. 方法区中常量引用的对象
4. 本地方法栈中 JNI 引用的对象
5. 同步锁持有的对象

**2. 垃圾收集算法**

**标记-清除（Mark-Sweep）**：
```
标记前：[对象A][垃圾][对象B][垃圾][垃圾][对象C]
标记后：[对象A][  垃圾 ][对象B][  垃圾  ][垃圾][对象C]
清除后：[对象A][      ][对象B][         ][   ][对象C]
                          ↑         ↑
                        碎片       碎片
```
- **优点**：简单
- **缺点**：效率低，有内存碎片

**标记-整理（Mark-Compact）**：
```
标记后：[对象A][  垃圾 ][对象B][  垃圾  ][垃圾][对象C]
整理后：[对象A][对象B][对象C][                   ]
                                           ↑
                                       连续空闲空间
```
- **优点**：无碎片
- **缺点**：移动对象有开销

**复制算法（Copying）**：
```
复制前：
┌─────────────────From─────────────────┐
│[对象A][垃圾][对象B][垃圾][垃圾][对象C]│
└──────────────────────────────────────┘

复制后：
┌─────────────────To──────────────────┐
│[对象A][对象B][对象C][               ]│
└──────────────────────────────────────┘
```
- **优点**：高效，无碎片
- **缺点**：空间利用率低（需要 50% 空间作为保留区）

**分代收集**：
- 新生代：存活率低，用复制算法
- 老年代：存活率高，用标记-整理或标记-清除

**面试加分点**：
> 为什么新生代用复制算法？
> - 新生代对象存活率低（约 90% 都是垃圾）
> - 复制算法只需复制少量存活对象，效率高
> - 实际使用 Eden:S0:S1 = 8:1:1 的比例，只用 10% 作为保留区

**追问延伸**：
- CMS 用什么算法？有什么问题？
  - 用标记-清除算法，有内存碎片
  - 解决方案：开启 `-XX:+UseCMSCompactAtFullCollection` 压缩

- G1 的 Region 设计？
  - 将堆划分为多个大小相等的 Region（1-32MB）
  - 每个 Region 可以是 Eden、Survivor 或 Old
  - 优先收集垃圾最多的 Region（Garbage First）

### Q3: 说一下 CMS 收集器？

**核心概念**：

CMS（Concurrent Mark Sweep）是第一款真正意义上的并发收集器，让垃圾收集线程与用户线程同时运行。

**四个阶段**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        CMS 收集过程                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 初始标记（STW）                                              │
│     └─ 标记 GC Roots 直接关联的对象                              │
│     └─ 速度很快                                                 │
│                                                                 │
│  2. 并发标记                                                     │
│     └─ 遍历对象图，与用户线程并发                                │
│     └─ 时间长，但不阻塞用户线程                                  │
│                                                                 │
│  3. 重新标记（STW）                                              │
│     └─ 修正并发标记期间的变动（增量更新）                         │
│     └─ 修复"浮动垃圾"问题                                        │
│                                                                 │
│  4. 并发清除                                                     │
│     └─ 清除标记为垃圾的对象                                      │
│     └─ 与用户线程并发                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```bash
# 启用 CMS 收集器（JDK 1.8 默认使用 Parallel，需手动开启）
-XX:+UseConcMarkSweepGC
-XX:CMSInitiatingOccupancyFraction=75  # 老年代 75% 时触发
-XX:+UseCMSCompactAtFullCollection     # Full GC 时压缩
-XX:CMSFullGCsBeforeCompaction=0       # 每次 Full GC 都压缩
```

**优缺点分析**：

| 优点 | 缺点 |
|------|------|
| 并发收集，低停顿 | CPU 敏感（占用线程资源） |
| 老年代收集首选 | 浮动垃圾（并发期间新产生的垃圾） |
| 成熟稳定 | 内存碎片（标记-清除算法） |

**面试加分点**：
> CMS 的"三色标记"算法：
> - 白色：未被访问
> - 灰色：已被访问，但引用的对象未全部访问
> - 黑色：已被访问，且引用的对象全部访问
> 
> 问题：并发标记期间，用户线程可能修改引用，导致"对象消失"
> 解决：增量更新（Incremental Update）或原始快照（SATB）

**追问延伸**：
- CMS 和 G1 的区别？

| 维度 | CMS | G1 |
|------|-----|-----|
| 算法 | 标记-清除 | 标记-整理 + 复制 |
| 内存布局 | 连续新生代/老年代 | Region 分区 |
| 碎片 | 有 | 无 |
| 可预测停顿 | 不支持 | 支持（`MaxGCPauseMillis`） |
| JDK 版本 | JDK 5 引入，JDK 9 标记废弃 | JDK 7 引入，JDK 9 默认 |

- 为什么 CMS 要用标记-清除不用标记-整理？
  - 并发清除时，如果移动对象，需要更新引用，与用户线程冲突
  - G1 通过 Region 隔离解决了这个问题

### Q4: 类加载过程？

**核心概念**：

类加载分为 5 个阶段：加载 → 验证 → 准备 → 解析 → 初始化

**详细过程**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        类加载过程                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 加载（Loading）                                              │
│     └─ 读取 .class 文件到内存                                    │
│     └─ 创建 Class 对象                                           │
│     └─ 可以从文件系统、网络、ZIP 包加载                          │
│                                                                 │
│  2. 验证（Verification）                                         │
│     └─ 文件格式验证（魔数 0xCAFEBABE）                            │
│     └─ 字节码验证                                                │
│     └─ 符号引用验证                                              │
│                                                                 │
│  3. 准备（Preparation）                                          │
│     └─ 为静态变量分配内存，设初始值                              │
│     └─ 注意：final 变量直接赋真实值                              │
│                                                                 │
│  4. 解析（Resolution）                                           │
│     └─ 符号引用 → 直接引用                                       │
│     └─ 类/接口、字段、方法解析                                   │
│                                                                 │
│  5. 初始化（Initialization）                                     │
│     └─ 执行 <clinit> 方法（静态变量赋值 + 静态代码块）            │
│     └─ 父类先于子类初始化                                        │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
public class ClassInitDemo {
    // 准备阶段：value = 0
    // 初始化阶段：value = 123
    private static int value = 123;
    
    // 准备阶段：CONSTANT = 456（final 直接赋值）
    private static final int CONSTANT = 456;
    
    static {
        System.out.println("静态代码块执行");
    }
    
    public static void main(String[] args) {
        System.out.println(ClassInitDemo.value);  // 触发类初始化
    }
}
```

**双亲委派模型**：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                 │
│                    ┌────────────────────┐                       │
│                    │  Bootstrap Class   │                       │
│                    │  Loader            │                       │
│                    │  (核心类库)         │                       │
│                    └─────────┬──────────┘                       │
│                              │                                  │
│                    ┌─────────▼──────────┐                       │
│                    │  Extension Class   │                       │
│                    │  Loader            │                       │
│                    │  (扩展类库)         │                       │
│                    └─────────┬──────────┘                       │
│                              │                                  │
│                    ┌─────────▼──────────┐                       │
│                    │  Application Class │                       │
│                    │  Loader            │                       │
│                    │  (应用类路径)       │                       │
│                    └─────────┬──────────┘                       │
│                              │                                  │
│                    ┌─────────▼──────────┐                       │
│                    │  Custom Class      │                       │
│                    │  Loader            │                       │
│                    └────────────────────┘                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**双亲委派流程**：
1. 收到类加载请求
2. 先委托父类加载器加载
3. 父类无法加载，才自己加载

**为什么需要双亲委派？**
1. **安全**：防止核心类被篡改（如自定义 `java.lang.String`）
2. **避免重复加载**：父类已加载，子类不再加载

**如何打破双亲委派？**
1. **自定义类加载器**：重写 `loadClass()` 方法
2. **SPI 机制**：JDBC Driver 使用线程上下文类加载器

```java
// 打破双亲委派示例
public class CustomClassLoader extends ClassLoader {
    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {
        // 先尝试自己加载
        try {
            byte[] data = loadClassData(name);
            return defineClass(name, data, 0, data.length);
        } catch (Exception e) {
            // 失败则委托父类
            return super.loadClass(name);
        }
    }
}
```

**追问延伸**：
- 什么时候触发类初始化？
  1. 遇到 `new`、`getstatic`、`putstatic`、`invokestatic`
  2. 反射调用 `Class.forName()`
  3. 初始化子类时，父类未初始化
  4. 主类（main 方法所在类）

## 二、并发编程高频题

### Q5: synchronized 底层原理？

**核心概念**：

synchronized 是 Java 内置的互斥锁，基于 Monitor 实现。

**三种使用方式**：

```java
public class SynchronizedDemo {
    private final Object lock = new Object();
    
    // 1. 同步代码块（锁指定对象）
    public void block() {
        synchronized (lock) {
            // 临界区
        }
    }
    
    // 2. 同步实例方法（锁 this）
    public synchronized void instanceMethod() {
        // 临界区
    }
    
    // 3. 同步静态方法（锁 Class 对象）
    public static synchronized void staticMethod() {
        // 临界区
    }
}
```

**底层原理**：

**同步代码块**：`monitorenter` / `monitorexit` 字节码指令

```java
// 源码
public void block();
    synchronized (lock) {
        // ...
    }

// 字节码
Code:
   0: aload_0
   1: getfield      #3  // Field lock:Ljava/lang/Object;
   4: dup
   5: astore_1
   6: monitorenter      // 获取 monitor
       // ... 临界区代码
  10: monitorexit       // 释放 monitor
```

**同步方法**：`ACC_SYNCHRONIZED` 标志

```java
// 方法 flags 中有 ACC_SYNCHRONIZED
// JVM 自动在方法入口获取 monitor，出口释放 monitor
public synchronized void method();
```

**Monitor 原理**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        Monitor 结构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Owner: 持有锁的线程                                            │
│                                                                 │
│  Entry List: 等待获取锁的线程队列                               │
│  ┌───┬───┬───┬───┐                                            │
│  │ T1│ T2│ T3│...│                                            │
│  └───┴───┴───┴───┘                                            │
│                                                                 │
│  Wait Set: 调用 wait() 的线程队列                               │
│  ┌───┬───┬───┬───┐                                            │
│  │ T4│ T5│ T6│...│                                            │
│  └───┴───┴───┴───┘                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**锁升级过程**：

```
无锁 → 偏向锁 → 轻量级锁 → 重量级锁

升级方向：不可逆（只能升级，不能降级）
```

**对象头（Mark Word）**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     64 位对象头结构                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  无锁：                                                         │
│  ┌────────────────────────────────────┬───────┬────┬──────────┐│
│  │            hashcode (31)           │ age(4)│ 0  │ lock(01) ││
│  └────────────────────────────────────┴───────┴────┴──────────┘│

### Q6: ThreadLocal 原理和内存泄漏？

**核心概念**：

ThreadLocal 提供线程局部变量，每个线程都有独立的变量副本。

**原理图**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      ThreadLocal 原理                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Thread                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  ThreadLocalMap                                         │   │
│  │  ┌──────────────────────────────────────────────────┐  │   │
│  │  │  Entry (WeakReference<ThreadLocal>)              │  │   │
│  │  │  ┌─────────────────┐  ┌───────────────────────┐  │  │   │
│  │  │  │ key (弱引用)     │  │ value (强引用)        │  │  │   │
│  │  │  │ ThreadLocal     │  │ 实际存储的值           │  │  │   │
│  │  │  └─────────────────┘  └───────────────────────┘  │  │   │
│  │  └──────────────────────────────────────────────────┘  │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**代码示例**：

```java
public class ThreadLocalDemo {
    // 每个线程独立的用户上下文
    private static final ThreadLocal<User> userContext = new ThreadLocal<>();
    
    public void process() {
        try {
            userContext.set(new User("张三"));
            // 业务逻辑
            doSomething();
        } finally {
            userContext.remove();  // 防止内存泄漏
        }
    }
    
    public void doSomething() {
        User user = userContext.get();  // 获取当前线程的用户
    }
}
```

**内存泄漏原因**：

```
1. key 是弱引用，GC 后变为 null
2. value 是强引用，仍被 Entry 引用
3. 线程不结束，value 无法回收

┌─────────────────────────────────────────────────────────────────┐
│  ThreadLocalMap                                                 │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │  Entry                                                    │  │
│  │  ┌─────────────┐  ┌───────────────────────────────────┐  │  │
│  │  │ key = null  │  │ value = Object (无法访问也无法回收) │  │  │
│  │  │ (被 GC)     │  │                                   │  │  │
│  │  └─────────────┘  └───────────────────────────────────┘  │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**解决方案**：
1. **及时 remove()**：在 finally 块中清理
2. **使用 try-finally**：确保一定清理

**面试加分点**：
> key 为什么用弱引用？
> - 如果用强引用，ThreadLocal 对象无法被回收
> - 用弱引用，ThreadLocal 可以被 GC，避免内存泄漏
> - 但 value 仍需手动 remove()

**应用场景**：
1. 数据库连接管理
2. 用户上下文传递
3. SimpleDateFormat 线程安全

```java
// SimpleDateFormat 线程安全
private static final ThreadLocal<SimpleDateFormat> dateFormat = 
    ThreadLocal.withInitial(() -> new SimpleDateFormat("yyyy-MM-dd"));

public String format(Date date) {
    return dateFormat.get().format(date);
}
```

### Q7: 线程池参数和执行流程？

**核心概念**：

线程池通过复用线程，减少创建销毁开销，提高性能。

**核心参数**：

```java
public ThreadPoolExecutor(
    int corePoolSize,              // 核心线程数
    int maximumPoolSize,           // 最大线程数
    long keepAliveTime,            // 空闲线程存活时间
    TimeUnit unit,                 // 时间单位
    BlockingQueue<Runnable> workQueue,  // 工作队列
    ThreadFactory threadFactory,   // 线程工厂
    RejectedExecutionHandler handler    // 拒绝策略

```

**参数详解**：

| 参数 | 说明 | 建议 |
|------|------|------|
| corePoolSize | 核心线程数，空闲不回收 | CPU 密集型：N+1；IO 密集型：2N |
| maximumPoolSize | 最大线程数 | 根据系统资源设置 |
| keepAliveTime | 非核心线程空闲存活时间 | 默认 60s |

**执行流程**：

```

    │        └────────┬────────┘
    │                 │
    │            ┌────┴────┐
    │            │         │
    │            ▼         ▼
    │           是         否
    │            │         │
    │            ▼         ▼

**四种拒绝策略**：

### Q8: ConcurrentHashMap 实现？

**JDK 1.7：分段锁**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    JDK 1.7 分段锁结构                            │

└─────────────────────────────────────────────────────────────────┘

**put 流程代码**：

```java
final V putVal(K key, V value, boolean onlyIfAbsent) {
    // 1. 计算 hash
    int hash = spread(key.hashCode());
    
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i, fh;
        
        // 2. 桶为空，CAS 插入
        if (tab == null || (n = tab.length) == 0)
            tab = initTable();
        else if ((f = tabAt(tab, i = (n - 1) & hash)) == null) {
            if (casTabAt(tab, i, null, new Node<K,V>(hash, key, value, null)))
                break;
        }
        // 3. 桶不为空，synchronized 锁头节点
        else {
            synchronized (f) {
                // 遍历链表或红黑树，插入节点
            }
        }
    }
    return null;
}
```

**面试加分点**：
> 为什么 JDK 1.8 放弃分段锁？
> 1. 每个 Segment 都要占用内存，空间开销大
> 2. 并发度受 Segment 数量限制（默认 16）
> 3. CAS + synchronized 锁粒度更细，只锁冲突的桶

  - JDK 1.8 使用 LongAdder 思路，分散计数
  - 汇总 baseCount + CounterCell[]

### Q9: MySQL 为什么用 B+ 树？

**核心概念**：

B+ 树是 B 树的变体，更适合磁盘存储。

**B+ 树结构**：

```
                    B+ 树结构
                    
                    ┌───────┐
                    │  20   │
                    └───┬───┘
                ┌────────┴────────┐
                ▼                 ▼
            ┌───────┐         ┌───────┐

        │       │
        ▼       ▼

**为什么用 B+ 树**：

| 特点 | 好处 |
|------|------|
| 树高度低（3-4层） | 减少 IO 次数 |
| 叶子节点链表 | 范围查询高效 |
| 非叶子节点只存键值 | 单页存更多索引项 |

| 维度 | B 树 | B+ 树 |
|------|------|-------|
| 数据存储 | 所有节点 | 仅叶子节点 |
| 叶子节点 | 无指针 | 有链表指针 |
| 范围查询 | 不友好 | 友好 |
| 树高度 | 较高 | 较低 |
| 单页存储 | 较少索引项 | 更多索引项 |

> - Hash 索引不支持最左前缀匹配

### Q10: 索引失效场景？

**核心概念**：

索引失效会导致全表扫描，严重影响性能。

**失效场景**：

```sql
-- 1. LIKE 以 % 开头
WHERE name LIKE '%三';   -- ❌ 索引失效
WHERE name LIKE '张%';   -- ✅ 索引生效

-- 2. 对索引列做运算
WHERE YEAR(create_time) = 2024;  -- ❌ 索引失效
WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';  -- ✅

-- 3. 类型隐式转换
WHERE phone = 13800138000;        -- ❌ phone 是 varchar
WHERE phone = '13800138000';      -- ✅

-- 4. OR 条件
WHERE name = '张三' OR age = 20;  -- ❌ 如果 age 没索引

-- 5. 使用函数
WHERE LOWER(name) = 'zhangsan';   -- ❌ 索引失效

-- 6. 不等于
WHERE name != '张三';             -- ❌ 可能失效

-- 7. IS NOT NULL
WHERE name IS NOT NULL;           -- ❌ 可能失效
```

**最左前缀原则**：

```sql
-- 联合索引 (a, b, c)

-- ✅ 能用到索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3

-- ❌ 用不到索引
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3

-- ⚠️ 部分用到
WHERE a = 1 AND c = 3  -- 只用到 a
```

**覆盖索引**：

```sql
-- 联合索引 (name, age)

-- 覆盖索引：查询列都在索引中，无需回表
SELECT name, age FROM user WHERE name = '张三';  -- ✅ 无需回表

-- 需要回表
SELECT * FROM user WHERE name = '张三';          -- ❌ 需要回表
```

**EXPLAIN 分析**：

```sql
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

**type 从好到坏**：

```
system > const > eq_ref > ref > range > index > ALL

const: 主键或唯一索引常量查询
eq_ref: 主键或唯一索引关联查询
ref: 非唯一索引查询
range: 范围查询
index: 全索引扫描
ALL: 全表扫描（最差）
```

**面试加分点**：
> 如何优化深分页？
> 
> ```sql
> -- 问题：深分页效率低
> SELECT * FROM user LIMIT 1000000, 10;
> 
> -- 优化1：覆盖索引 + JOIN
> SELECT a.* FROM user a
> JOIN (SELECT id FROM user LIMIT 1000000, 10) b
> ON a.id = b.id;
> 
> -- 优化2：记录上次最大 ID
> SELECT * FROM user WHERE id > 1000000 LIMIT 10;
> ```

### Q11: MVCC 原理？

**核心概念**：

MVCC（Multi-Version Concurrency Control）是多版本并发控制，用于实现读已提交和可重复读隔离级别。

**核心组件**：

```
┌─────────────────────────────────────────────────────────────────┐
│                        MVCC 核心组件                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 隐藏字段                                                     │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ DB_TRX_ID  │ 事务 ID，最后一次修改该行的事务            │  │
│     │ DB_ROLL_PTR│ 回滚指针，指向 Undo Log 中的上一个版本    │  │
│     │ DB_ROW_ID  │ 隐藏主键（如果没有显式主键）              │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  2. Undo Log（版本链）                                           │
│     ┌───────────────────────────────────────────────────────┐  │
│     │  当前数据 → Undo Log 1 → Undo Log 2 → Undo Log 3      │  │
│     │   (trx=100)   (trx=99)    (trx=98)    (trx=97)        │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
│  3. Read View（可见性判断）                                      │
│     ┌───────────────────────────────────────────────────────┐  │
│     │ m_ids:  生成 Read View 时的活跃事务 ID 列表             │  │
│     │ min_trx_id: 活跃事务中最小的事务 ID                     │  │
│     │ max_trx_id: 下一个将分配的事务 ID                       │  │
│     │ creator_trx_id: 创建该 Read View 的事务 ID              │  │
│     └───────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**版本链示例**：

```

│                       Undo Log 版本链                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │

### Q14: 分布式锁实现？

**核心概念**：

分布式锁保证跨进程的互斥访问。

**Redis 实现**：

```java
// 加锁
SET lock:key value NX PX 30000

// 解锁（Lua 脚本保证原子性）

    return redis.call("del", KEYS[1])

end

**Redisson 实现**：

```java
RLock lock = redissonClient.getLock("lock:key");
try {
    // 尝试获取锁，等待 10 秒，锁自动过期时间 30 秒
    if (lock.tryLock(10, 30, TimeUnit.SECONDS)) {
        // 业务逻辑
    }
} finally {
    lock.unlock();
}
```

**看门狗机制**：

```
┌─────────────────────────────────────────────────────────────────┐
│                     Redisson 看门狗                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  问题：业务执行时间超过锁过期时间，锁自动释放                    │
│                                                                 │
│  解决：看门狗自动续期                                            │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │  1. 获取锁成功后，启动看门狗线程                          │ │
│  │  2. 每 10 秒检查一次（过期时间/3）                        │ │
│  │  3. 如果锁还被持有，自动续期到 30 秒                      │ │
│  │  4. 业务执行完毕，停止看门狗                              │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**Zookeeper 实现**：

```

│                    Zookeeper 分布式锁                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  实现方式：临时顺序节点                                          │
│                                                                 │
│  /lock                                                         │
│    ├── lock-000001    # 节点 1（获取锁）                        │

### Q17: Spring AOP 原理？

**核心概念**：

AOP（面向切面编程）通过动态代理实现横切关注点的分离。

**动态代理**：

```java
// JDK 动态代理（接口）
public class JdkProxy implements InvocationHandler {
    private Object target;
    
    public Object proxy() {
        return Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            this
        );
    }
    
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        // 前置通知
        Object result = method.invoke(target, args);
        // 后置通知
        return result;
    }
}

// CGLIB（类）
public class CglibProxy implements MethodInterceptor {
    public Object proxy(Class<?> clazz) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(clazz);
        enhancer.setCallback(this);
        return enhancer.create();
    }
    
    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        // 前置通知
        Object result = proxy.invokeSuper(obj, args);
        // 后置通知
        return result;
    }
}
```

**选择策略**：

| 情况 | 代理方式 |
|------|---------|
| 有接口 | JDK 动态代理 |
| 无接口 | CGLIB |
| 配置 proxy-target-class=true | 强制 CGLIB |

**面试加分点**：

> - TransactionInterceptor 作为环绕通知
> - 方法执行前开启事务
> - 方法执行成功提交事务

### Q18: Spring 事务传播机制？

**核心概念**：

事务传播机制定义了事务方法之间如何传播事务上下文。

**传播行为**：

| 传播行为 | 说明 | 场景 |
|---------|------|------|
| REQUIRED | 有则加入，无则新建（默认） | 大多数场景 |
| SUPPORTS | 有则加入，无则非事务 | 查询 |
| MANDATORY | 必须有事务，否则抛异常 | 强制事务 |
| REQUIRES_NEW | 新建事务，挂起当前 | 独立日志 |
| NOT_SUPPORTED | 非事务执行，挂起当前 | 非事务操作 |
| NEVER | 非事务执行，有事务则抛异常 | 禁止事务 |
| NESTED | 嵌套事务 | 部分回滚 |

**REQUIRED vs NESTED**：

```java
// REQUIRED：外层回滚，内层也回滚
@Transactional(propagation = Propagation.REQUIRED)
public void outer() {
    inner();  // 内层异常，外层也回滚
}

@Transactional(propagation = Propagation.REQUIRED)
public void inner() {
    // 异常
}

// NESTED：内层回滚，外层可以不回滚
@Transactional(propagation = Propagation.REQUIRED)
public void outer() {
    try {
        inner();  // 内层异常，只回滚内层
    } catch (Exception e) {
        // 处理异常，外层继续
    }
}

@Transactional(propagation = Propagation.NESTED)
public void inner() {
    // 异常
}
```

**事务失效场景**：

```java
@Service
public class UserService {
    
    // ❌ 事务失效：方法不是 public
    @Transactional
    private void method1() {}
    
    // ❌ 事务失效：同类调用
    public void method2() {
        this.method3();  // 不走代理
    }
    
    @Transactional
    public void method3() {}
    
    // ❌ 事务失效：异常被捕获
    @Transactional
    public void method4() {
        try {
            // 异常
        } catch (Exception e) {
            // 吞掉异常
        }
    }
    
    // ❌ 事务失效：抛出非 RuntimeException
    @Transactional
    public void method5() throws Exception {
        throw new Exception();  // 非 RuntimeException
    }
}
```

**RIP 协议（距离向量）**：

| 收敛速度 | 慢（分钟级） | 快（秒级） |
| 网络规模 | 小型（<15跳） | 大型 |

### Q6: 手撕算法 - 反转链表 II

**题目**：反转链表从位置 m 到 n 的部分

**参考解答**：
```java
public ListNode reverseBetween(ListNode head, int m, int n) {
    ListNode dummy = new ListNode(0);
    dummy.next = head;
    ListNode pre = dummy;
    
    // 1. 找到 m 的前一个节点
    for (int i = 1; i < m; i++) {
        pre = pre.next;
    }
    
    // 2. 反转 m 到 n 的节点
    ListNode cur = pre.next;
    for (int i = m; i < n; i++) {
        ListNode next = cur.next;
        cur.next = next.next;
        next.next = pre.next;
        pre.next = next;
    }
    
    return dummy.next;
}
```

**时间复杂度**：O(n)
**空间复杂度**：O(1)

#### Q4: 网络层如何找到下一跳？RIP 和 OSPF 协议？

**参考答案**：

**路由选择原理**：
1. 路由器维护路由表
2. 根据目的 IP 查找路由表
3. 选择最优路径转发

**RIP 协议（距离向量）**：

```
特点：
- 基于跳数计算距离（最大 15 跳）
- 定期向邻居广播路由表
- 收敛慢，适合小型网络

工作原理：
1. 每个路由器维护到各目的地的距离向量
2. 向邻居路由器发送自己的距离向量
3. 收到邻居的距离向量后，更新自己的路由表
4. 选择跳数最小的路径

缺点：
- 收敛慢（坏消息传播慢）
- 最大跳数限制（15 跳不可达）
- 占用带宽（定期广播）
```

**OSPF 协议（链路状态）**：

```
特点：
- 基于链路状态计算最短路径
- 每个路由器维护完整的网络拓扑
- 收敛快，适合大型网络

工作原理：
1. 每个路由器了解周围链路状态
2. 向所有路由器泛洪链路状态信息
3. 每个路由器独立计算最短路径树（Dijkstra 算法）
4. 生成路由表

优点：
- 收敛快
- 支持大型网络
- 无跳数限制
```

**Dijkstra 算法示例**：
```java
public void dijkstra(int[][] graph, int src) {
    int n = graph.length;
    int[] dist = new int[n];      // 最短距离
    boolean[] visited = new boolean[n];
    
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[src] = 0;
    
    for (int i = 0; i < n - 1; i++) {
        int u = minDistance(dist, visited);
        visited[u] = true;
        
        for (int v = 0; v < n; v++) {
            if (!visited[v] && graph[u][v] != 0 
                && dist[u] != Integer.MAX_VALUE 
                && dist[u] + graph[u][v] < dist[v]) {
                dist[v] = dist[u] + graph[u][v];
            }
        }
    }
}
```

- RIP 和 OSPF 如何选择？
  - 小型网络：RIP 配置简单
  - 大型网络：OSPF 收敛快，更可靠
- BGP 协议了解吗？
  - 边界网关协议，用于 AS 之间的路由

#### Q4: 为什么选择 Flask？项目怎么部署到阿里云？

**参考答案**：

**Flask 选择理由**：
1. **轻量级**：微框架，按需扩展
2. **灵活性高**：不强制项目结构
3. **适合小型项目/原型**：快速验证想法

**阿里云部署流程**：
```bash
# 1. 购买 ECS 实例
# 2. 安装环境
sudo apt update
sudo apt install python3 python3-pip nginx

# 3. 上传代码
scp -r ./project root@ecs-ip:/var/www/

# 4. 配置 Gunicorn
pip install gunicorn
gunicorn -w 4 -b 0.0.0.0:8000 app:app

# 5. Nginx 反向代理
server {
    listen 80;
    location / {
        proxy_pass http://127.0.0.1:8000;
    }
}
```

**追问**：
- Flask vs Django？
  - Flask：灵活、轻量、适合小型项目
  - Django：功能全、适合大型项目
- 生产环境为什么用 Gunicorn？
  - 多进程、性能好、稳定
