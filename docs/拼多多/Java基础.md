## 一、Java 基础高频题


### Q1: HashMap 实现原理？

**核心概念**：

HashMap 是 Java 最常用的 Map 实现，基于哈希表。

**JDK 1.8 结构**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    HashMap 结构 (JDK 1.8)                        │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Node[] table                                                   │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐            │
│  │Node │Node │Node │Node │Node │Node │Node │ ... │            │
│  └──┬──┴─────┴──┬──┴─────┴─────┴─────┴──┬──┴─────┘            │
│     │           │                       │                      │
│     ▼           ▼                       ▼                      │
│   链表         红黑树                  链表                     │
│   (长度<8)    (长度≥8)               (长度<8)                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**哈希映射过程**：

```java
// 1. 计算 hash
int hash = (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);

// 2. 定位桶
int index = (n - 1) & hash;  // 等价于 hash % n，但位运算更快

// 为什么用 (n - 1) & hash？
// - n 是 2 的幂，n-1 的二进制全是 1
// - 如 n = 16，n-1 = 15 = 1111（二进制）
// - hash & 1111 = hash 的低 4 位
// - 位运算比取模快
```

**put 流程**：

```
1. 计算 hash
2. 定位桶
3. 桶为空 → 直接插入
4. 桶不为空：
   a. 相同 key → 覆盖 value
   b. 红黑树节点 → 插入树
   c. 链表节点 → 尾插法插入，链表长度 >= 8 转红黑树
5. 扩容检查：size > threshold
```

**面试加分点**：
> TreeMap 实现原理？
> - 基于红黑树
> - key 有序（根据 Comparable 或 Comparator）
> - 查找、插入、删除 O(log n)

### Q2: 为什么 HashMap 要使用红黑树？

**核心概念**：

红黑树解决链表过长导致的查询效率问题。

**对比分析**：

```
┌─────────────────────────────────────────────────────────────────┐
│                    链表 vs 红黑树                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  链表：                                                         │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  查找：O(n)                                             │   │
│  │  插入：O(1)（已知位置）                                  │   │
│  │  删除：O(1)（已知位置）                                  │   │
│  │  问题：长度过长时查找效率低                              │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  红黑树：                                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  查找：O(log n)                                         │   │
│  │  插入：O(log n)                                         │   │
│  │  删除：O(log n)                                         │   │
│  │  优点：平衡性好，最坏情况也有保证                        │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  转换条件：                                                      │
│  - 链表长度 >= 8 且数组长度 >= 64 → 红黑树                     │
│  - 红黑树节点数 <= 6 → 链表                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么不用 AVL 树？**
- AVL 树更严格平衡，查找略快
- 红黑树插入删除更快，旋转次数少
- HashMap 插入删除操作多，红黑树更适合

### Q3: Java 反射是什么？

**核心概念**：

反射允许程序在运行时动态获取类的信息，创建对象、调用方法、访问属性。

**核心 API**：

```java
// 1. 获取 Class 对象
Class<?> clazz = Class.forName("com.example.User");
Class<?> clazz = User.class;
Class<?> clazz = user.getClass();

// 2. 创建对象
Object obj = clazz.newInstance();  // 已废弃
Object obj = clazz.getDeclaredConstructor().newInstance();

// 3. 获取方法
Method method = clazz.getMethod("setName", String.class);
method.invoke(obj, "张三");

// 4. 获取字段
Field field = clazz.getDeclaredField("name");
field.setAccessible(true);  // 突破 private 限制
field.set(obj, "李四");

// 5. 获取注解
Annotation[] annotations = clazz.getAnnotations();
```

**应用场景**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      反射应用场景                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 框架开发                                                     │
│     └─ Spring IoC：动态创建 Bean                                │
│     └─ Spring AOP：动态代理                                     │
│     └─ MyBatis：结果集映射                                      │
│                                                                 │
│  2. 工具类                                                       │
│     └─ BeanUtils：属性复制                                      │
│     └─ JSON 序列化/反序列化                                     │
│                                                                 │
│  3. 测试框架                                                     │
│     └─ JUnit：运行测试方法                                      │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**性能开销**：

| 操作 | 直接调用 | 反射调用 |
|------|---------|---------|
| 方法调用 | 快 | 慢（2-100 倍） |
| 字段访问 | 快 | 慢 |

**性能优化**：
- 缓存 Method/Field 对象
- 使用 setAccessible(true) 关闭安全检查
- 使用 MethodHandle（JDK 7+）

## 三、操作系统高频题


### Q7: 用户态和核心态切换条件？

**核心概念**：

操作系统分为用户态和核心态，限制程序权限。

**切换条件**：

```
┌─────────────────────────────────────────────────────────────────┐
│                   用户态 → 核心态切换                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 系统调用                                                     │
│     └─ 文件操作：open、read、write                             │
│     └─ 进程控制：fork、exec                                    │
│     └─ 内存管理：mmap                                         │
│                                                                 │
│  2. 异常                                                         │
│     └─ 除零异常                                                │
│     └─ 缺页异常                                                │
│     └─ 内存访问越界                                            │
│                                                                 │
│  3. 外设中断                                                     │
│     └─ 网络中断                                                │
│     └─ 磁盘中断                                                │
│     └─ 时钟中断                                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**为什么要切换？**

| 原因 | 说明 |
|------|------|
| 安全性 | 防止用户程序直接操作硬件 |
| 稳定性 | 隔离用户程序和内核 |
| 权限管理 | 不同级别程序有不同权限 |

**切换开销**：
- 保存/恢复寄存器
- 切换内核栈
- 刷新 TLB
- 开销约几微秒
